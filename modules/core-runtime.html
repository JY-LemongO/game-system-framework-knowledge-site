Core Runtime / System Contract

Core Runtime은 Stat, Effect, Skill, Combat, Status가 같은 계약으로 대화하게 만드는 기반 계층이다. 현재 C# 실행 참조는 EntityId와 SourceRef 불변 값 타입을 제공하고, 시간·랜덤·이벤트 포트는 이 페이지의 normative conceptual contract로 설계한다.

예제 언어 · C# EntityId TagSet SourceRef EventBus GameContext · Concept RandomStream
Learning goals

학습 목표

공유 식별자는 C# 값 타입으로, 시간·랜덤·이벤트는 인터페이스로 표현하고 도메인 규칙과 기반 규약의 경계를 설명한다.

Prerequisites

선수 개념

객체 참조와 안정적인 ID의 차이, 이벤트 발행과 직접 호출의 차이를 알고 있으면 좋다.

왜 Core Runtime이 필요한가?

Stat과 Effect만 있을 때는 각 문서 안에서 공통 개념을 설명해도 큰 문제가 없다. 하지만 Skill, Combat, Status가 붙는 순간 동일한 Entity 참조, 태그 조건, 이벤트 발행, 시간 처리, 원천 추적 규칙이 여러 곳에 반복된다. Core Runtime은 이 반복을 줄이고, 시스템이 서로 직접 엮이지 않도록 공통 계약을 제공한다.

핵심. Core Runtime은 전투 공식이나 스킬 규칙을 구현하는 곳이 아니다. 모든 시스템이 공유하는 식별자, 컨텍스트, 시간, 랜덤, 이벤트, 디버그 추적의 최소 규약을 제공하는 곳이다.

책임 경계

담당한다 담당하지 않는다
EntityId, EntityRef, SourceRef 같은 공통 식별자 규칙 피해량 공식이나 명중/치명타 판정
TagSet, GameContext 개념, IEventBus 같은 시스템 간 계약 스킬 사용 가능 여부와 쿨다운 정책
IGameClock, Tick, Duration 기준 제공 상태 중첩이나 상태별 해제 규칙
IRandomStream과 Seed 분기 규칙 장비 옵션 생성 확률표 자체
Definition / Runtime 분리 기준과 IDataRegistry 조회 UI 버튼, 아이콘, 전투 텍스트 렌더링
DebugTrace, Replay, 원천 추적을 위한 공통 로그 형식 AI가 어떤 행동을 선택할지 판단
실행 참조의 event envelope. JavaScript Runtime Lab은 EventId·CorrelationId·CausationId·OccurredTick·SchemaVersion을 모두 실행 검증한다. 현재 compact C# DomainEvent는 EventId·CommandId·Source만 보존하므로, correlation·causation·version envelope 전체는 C# 실행 완료가 아닌 normative 확장 계약이다.

핵심 컴포넌트

EntityId / EntityRef

플레이어, 몬스터, 투사체, 소환수, 함정처럼 월드에 존재하는 대상을 안정적으로 참조한다. C# 객체 참조를 직접 저장하지 않고, 필요할 때 IEntityRegistry로 해석한다.

Source identity / SourceRef

Modifier, Effect, Status가 어디서 왔는지 나타내는 개념이 source identity다. 공개 C# 계약은 이를 별도 문자열 ID가 아니라 SourceRef로 표현하고 Kind, DefinitionId, InstanceId를 분리한다. 제거와 추적에는 등록할 때 보존한 동일한 SourceRef를 사용한다.

TagSet

fire , spell , boss , debuff , dot 처럼 규칙 분기에 쓰는 공통 분류다. 타입 상속보다 데이터 기반 조건식을 만들기 쉽다.

GameContext

ActorId, TargetId, Position, Source, Tags, Random, Time, CustomPayload를 묶어 생각하는 Conceptual umbrella다. 현재 C# 참조에는 GameContext나 SkillContext 타입이 없고 실제 타입은 StatContext, EffectContext, SkillRequest다. 제품에서는 공통 기반 타입으로 강제하기보다 각 소비자가 필요한 필드만 가진 문맥으로 특화한다.

IEventBus

SkillCommitted, DamageCommitted, StatusApplied 같은 commit 이후 이벤트를 발행한다. 시스템 간 직접 호출을 줄이고 장비 발동 효과, 업적, 로그, UI 알림을 느슨하게 연결한다.

IRandomStream

명중, 치명타, 장비 발동 확률처럼 랜덤이 필요한 판정을 Seed 기반 스트림으로 처리한다. 테스트, 리플레이, 서버 검증, 밸런스 시뮬레이션에 중요하다.

Definition과 Runtime 분리

Core Runtime이 강제해야 하는 가장 중요한 규칙은 정의 데이터와 런타임 상태의 분리다. 원본 정의를 실행 중에 수정하면 저장, 동기화, 리플레이, 밸런스 검증이 모두 어려워진다.

구분 예시 원칙
Definition Data StatDefinition, EffectDefinition, SkillDefinition, StatusDefinition 정적 데이터다. 런타임 중 원본을 직접 바꾸지 않는다.
Runtime State StatInstance, SkillRuntime, StatusInstance, ItemInstance 특정 Entity나 세션에 종속되는 현재 상태다. 저장/로드 대상이며 자체 InstanceId와 원천 SourceRef를 분리해 추적한다.
Runtime Override SkillMutation, temporary modifier, buff override 원본 Definition을 수정하지 않고 적용 시점에 겹쳐 읽는다.
Execution Context EffectContext, CombatContext, SkillRequest 한 번의 실행에 필요한 임시 정보다. 결과 로그를 제외하면 장기 저장하지 않는다.

시스템 계약 API

Core Runtime은 거대한 매니저 하나가 아니라, 작은 서비스와 값 객체의 집합으로 설계하는 편이 좋다. 각 시스템은 필요한 계약만 주입받는다.

계약 대표 API 사용 예
IEntityRegistry TryResolve(EntityId, out EntityRef), Exists(EntityId), QueryByTag(Tag) Effect의 TargetSelector가 대상 후보를 찾는다.
IDataRegistry GetDefinition<TDefinition>(DefinitionId), ValidateSchema(DefinitionId) SkillExecutor가 SkillDefinition과 EffectBundle을 조회한다.
IEventBus Publish<TEvent>(TEvent), Subscribe<TEvent>(Action<TEvent>) 장비의 TriggeredEffect가 DamageCommitted(outcome=Hit)에 반응한다.
IGameClock CurrentTick, Schedule(long tick, Action callback) StatusSystem이 지속시간과 tick을 관리한다.
IRandomStream Fork(RandomKey key), RollBps(int probabilityBps), Range(int min, int max) CombatResolver가 판정 key로 분기한 stream에서 명중과 치명타를 결정론적으로 판정한다.
IDebugTrace BeginScope(), AddStep(), AttachBreakdown() Stat 계산식, Skill 실행, Effect 결과를 한 로그로 묶는다.

C# 실행 값 타입과 개념적 포트

EntityId·SourceRef 부분은 현재 source/csharp/ 실행 참조와 같은 계약이다. 이어지는 IGameClock·IRandomStream·IEventBus는 소비자가 필요로 하는 최소 경계를 보여주는 Normative conceptual contract이며 현재 C# 프로젝트에는 구현·선언되어 있지 않다. Unity나 서버에 적용할 때 해당 adapter와 전달 보장을 별도로 구현한다.

public readonly record struct EntityId : IComparable<EntityId>
{
    private readonly string? _value;

    public EntityId(string value)
    {
        if (!TryValidate(value, out var error))
        {
            throw new ArgumentException(error, nameof(value));
        }

        _value = value;
    }

    public bool IsValid => _value is not null;

    public string Value => _value ?? throw new InvalidOperationException(
        "A default EntityId is invalid and has no value.");

    public static bool TryCreate(string? value, out EntityId entityId)
    {
        if (!TryValidate(value, out _))
        {
            entityId = default;
            return false;
        }

        entityId = new EntityId(value!);
        return true;
    }

    public int CompareTo(EntityId other) =>
        StringComparer.Ordinal.Compare(Value, other.Value);

    public override string ToString() => Value;

    internal static void ThrowIfInvalid(
        EntityId entityId,
        string parameterName)
    {
        if (!entityId.IsValid)
        {
            throw new ArgumentException(
                "An EntityId must be initialized with a canonical value.",
                parameterName);
        }
    }

    private static bool TryValidate(string? value, out string error)
    {
        if (string.IsNullOrWhiteSpace(value))
        {
            error = "An ID cannot be empty.";
            return false;
        }

        var segments = value.Split('.');
        if (segments.Length < 2)
        {
            error = "An ID must be namespaced, for example 'skill.fireball'.";
            return false;
        }

        for (var segmentIndex = 0; segmentIndex < segments.Length; segmentIndex++)
        {
            var segment = segments[segmentIndex];
            var validFirstCharacter = segment.Length > 0 &&
                (IsLowerAsciiLetter(segment[0]) ||
                 (segmentIndex > 0 && char.IsAsciiDigit(segment[0])));
            if (!validFirstCharacter)
            {
                error = segmentIndex == 0
                    ? "The namespace segment must start with a lowercase ASCII letter."
                    : "Local ID segments must start with a lowercase ASCII letter or digit.";
                return false;
            }

            for (var index = 1; index < segment.Length; index++)
            {
                var character = segment[index];
                if (!IsLowerAsciiLetter(character) &&
                    !char.IsAsciiDigit(character) &&
                    (character != '_' || segmentIndex == 0) &&
                    character != '-')
                {
                    error = "ID segments may contain lowercase ASCII letters, digits, underscores, and hyphens only.";
                    return false;
                }
            }
        }

        error = string.Empty;
        return true;
    }

    private static bool IsLowerAsciiLetter(char character) =>
        character is >= 'a' and <= 'z';
}

public enum SourceKind
{
    SkillExecution,
    Status,
    System
}

public readonly record struct SourceRef
{
    public SourceRef(
        SourceKind kind,
        EntityId definitionId,
        EntityId? instanceId = null)
    {
        if (!Enum.IsDefined(kind))
        {
            throw new ArgumentOutOfRangeException(nameof(kind));
        }

        EntityId.ThrowIfInvalid(definitionId, nameof(definitionId));
        if (instanceId.HasValue)
        {
            EntityId.ThrowIfInvalid(instanceId.Value, nameof(instanceId));
        }

        if (kind is SourceKind.SkillExecution or SourceKind.Status &&
            instanceId is null)
        {
            throw new ArgumentException(
                $"A {kind} source must retain its instance ID.",
                nameof(instanceId));
        }

        Kind = kind;
        DefinitionId = definitionId;
        InstanceId = instanceId;
    }

    public SourceKind Kind { get; }

    public EntityId DefinitionId { get; }

    public EntityId? InstanceId { get; }

    public bool IsValid =>
        Enum.IsDefined(Kind) &&
        DefinitionId.IsValid &&
        (Kind is not (SourceKind.SkillExecution or SourceKind.Status) ||
         InstanceId.HasValue) &&
        (!InstanceId.HasValue || InstanceId.Value.IsValid);

    public EntityId? StatusInstanceId =>
        Kind == SourceKind.Status ? InstanceId : null;

    internal static void ThrowIfInvalid(
        SourceRef source,
        string parameterName)
    {
        if (!source.IsValid)
        {
            throw new ArgumentException(
                "A SourceRef must contain valid definition and instance IDs.",
                parameterName);
        }
    }

    public static SourceRef SkillExecution(
        EntityId skillDefinitionId,
        EntityId commandId) =>
        new(SourceKind.SkillExecution, skillDefinitionId, commandId);

    public static SourceRef Status(
        EntityId statusDefinitionId,
        EntityId statusInstanceId) =>
        new(SourceKind.Status, statusDefinitionId, statusInstanceId);

    public static SourceRef System(EntityId systemId) =>
        new(SourceKind.System, systemId);
}

SourceRef와 추적성

SourceRef는 원천 종류와 정의, 실행 인스턴스를 함께 보존한다. Modifier 제거, 중첩 판정, 이벤트 원인 분석, 저장/로드 복원은 이 구조화된 값에 의존한다. 로그나 JavaScript 실습에 표시되는 sourceId는 검색 편의를 위한 보조 필드다. 값은 SourceRef.InstanceId가 있으면 그 ID, instance가 없는 System source라면 DefinitionId이며, 독립적인 원천 계약이 아니다.

System

시스템 Source

던전 규칙이나 난이도 보정처럼 실행 인스턴스가 필수가 아닌 원천은 SourceRef.System(ruleDefinitionId)으로 표현한다. 임시 규칙 인스턴스를 구분해야 한다면 InstanceId까지 보존한다.

Status

상태 Source

Burn 적용 원인은 SourceRef.SkillExecution(skillDefinitionId, commandId)로 남긴다. Burn tick이나 Rage/Freeze Modifier 자체의 원천은 SourceRef.Status(statusDefinitionId, statusInstanceId)로 구분한다.

Skill

스킬 Source

Fireball의 DamageEffect와 ApplyStatusEffect(effect.apply-burn)는 SourceRef.SkillExecution(skill.fireball, command.fireball.cast.0001)을 공유해 CombatLog와 EffectLog를 연결한다.

현재 SourceKind 범위. C#과 JavaScript 실행 계약이 허용하는 원천 종류는 SkillExecution/skill-execution, Status/status, System/system 세 가지뿐이다. 장비·아이템·패시브 전용 원천 종류는 현재 실행 참조 범위 밖이다. product 확장에서 이를 독립 종류로 추가할 때는 SourceKind, 직렬화 schema, 저장·event·replay migration을 함께 버전업해야 한다.

결정론과 테스트

전투 시스템은 디버그하기 어렵기 때문에 “같은 입력이면 같은 결과”를 최대한 보장해야 한다. 위 개념 포트의 root IRandomStream은 전역 랜덤을 바로 소비하지 않고 CastId + Branch + TargetId + EventIndex로 만든 RandomKey를 Fork하여 판정별 scoped stream을 만든다. 따라서 한 분기의 호출 추가가 다른 분기의 값을 밀어내지 않는다. JavaScript 실행 참조는 같은 의도를 stream 객체 대신 sampleBps([correlationId, branch, targetId])로 직접 검증하며, 이 compact key 모양은 rngKeySchemaVersion=correlation-branch-target-v1으로 별도 고정한다.

AttackerId처럼 결과에 영향을 주는 식별자가 필요하면 RandomKey의 명시적 필드나 안정된 key part로 추가한다. 어떤 필드를 쓰든 key schema와 RNG algorithm version을 replay envelope에 고정해야 한다.

테스트 대상 검증 포인트
동일 Seed 반복 실행 명중, 치명타, 확률 발동 결과가 동일해야 한다.
이벤트 순서 SkillCommitted → DamageCommitted → StatusApplied 순서가 보장되어야 한다. 계산 중간값은 DamageCalculated trace로만 남긴다.
Source 제거 상태나 시스템 규칙 제거 시 등록 때 보존한 SourceRef와 일치하는 Modifier만 제거되어야 한다.
리플레이 Command, 입력 스냅샷, Definition/Formula/Numeric 버전, RootSeed, 정렬 정책을 포함한 replay envelope로 결과를 재현해야 한다.
재현성 조건. Seed만 같다고 결과가 같아지는 것은 아니다. Definition 버전, 입력 스냅샷, 대상 정렬, ClockDomain, 부동소수점/반올림 정책, 이벤트 순서까지 고정해야 재현 가능한 시뮬레이션이 된다.

Event와 Reaction의 실행 순서

IEventBus는 관찰과 후속 반응을 분리해야 한다. UI·로그용 알림은 즉시 관찰해도 되지만, 흡혈·반사·장비 발동처럼 게임 상태를 바꾸는 반응은 우선순위와 재진입 방지 규칙이 있는 ReactionQueue 에 넣어 다음 단계에서 순서대로 실행한다.

관찰 callback은 동결된 trace snapshot만 받아야 하며, callback 실행 중 commit·clock advance·ReactionQueue enqueue/drain 같은 mutation entry를 다시 호출할 수 없어야 한다. 예외를 삼키는 것만으로는 충분하지 않다. 관찰자가 새 작업을 주입하거나 optimistic version 검사 사이에 재진입하면 실행 결과와 replay가 달라지기 때문이다.

종류 처리 필수 메타데이터
Observation UI, 로그, 분석처럼 결과를 바꾸지 않는 구독 EventId, CorrelationId
Reaction 흡혈, 반사, TriggeredEffect처럼 새 명령을 만드는 후속 처리 CausationId, Priority, Depth, IdempotencyKey
Rule step 명중·비용·상태 면역처럼 결과를 결정하는 핵심 규칙 명시적 파이프라인 안에서 실행

안티패턴

전역 Singleton 남용

모든 시스템이 GameManager.Instance 를 통해 서로 접근하면 테스트와 리플레이가 어려워진다. 필요한 계약만 인터페이스로 주입한다.

원본 Definition 수정

레벨업 보상으로 FireballDefinition 자체를 수정하면 다른 Entity와 저장 데이터까지 오염된다. SkillMutation 또는 RuntimeOverride로 분리한다.

문자열 Source 난립

별도 문자열 source ID를 원천 계약으로 사용하면 제거와 추적이 불안정하다. Kind, DefinitionId, InstanceId를 가진 구조화된 SourceRef를 사용한다.

이벤트에 게임 규칙 숨기기

핵심 규칙을 이벤트 리스너에만 숨기면 실행 순서가 불명확해진다. IEventBus는 후속 반응과 관찰용으로 쓰고 핵심 판정은 명시적 파이프라인에 둔다.

Comprehension check

이해도 확인

답을 펼치기 전에 Core가 제공해야 할 공통 계약과 각 도메인 시스템이 소유할 규칙을 먼저 구분해 본다.

  1. 01 · 책임 경계
    Fireball의 치명타 확률과 배율 공식을 Core Runtime의 IRandomStream 안에 넣어도 될까?
    정답과 해설

    넣으면 안 된다. Core는 Seed 분기와 결정론적 난수 API만 제공하고, 치명타 규칙과 공식은 Combat 정책이 소유해야 한다.

  2. 02 · 경계 상황
    같은 RootSeed지만 대상 정렬 순서와 Definition 버전이 다르면 결과가 같다고 보장할 수 있을까?
    정답과 해설

    보장할 수 없다. 입력 snapshot, Definition·공식·수치 버전, 대상 정렬, ClockDomain, 반올림과 이벤트 순서까지 고정해야 같은 실행을 재현할 수 있다.

  3. 03 · 시스템 연결
    DamageCommitted를 UI 피해 표시와 흡혈 효과가 함께 구독한다면 두 처리는 어떻게 달라야 할까?
    정답과 해설

    UI와 로그는 상태를 바꾸지 않는 Observation으로 처리한다. 흡혈은 CausationId, 우선순위, 깊이, 멱등 키를 가진 Reaction으로 큐에 넣어 명시적인 순서로 실행한다.

설계 점검

  • EntityId, SourceRef, TagSet, GameContext의 직렬화 형식이 정해져 있다.
  • Definition Data와 Runtime State가 분리되어 있고 원본 정의를 직접 수정하지 않는다.
  • IEventBus 이벤트 이름과 Payload 최소 규격이 문서화되어 있다.
  • IRandomStream이 Seed를 replay envelope의 버전·입력·정렬·시간·수치 정책과 함께 사용해 재현 가능하게 동작한다.
  • DebugTrace가 Skill, Effect, Stat, Combat 로그를 하나의 원인 체인으로 묶을 수 있다.
  • 상태나 시스템 규칙 제거 시 SourceRef 기준으로 관련 런타임 변경을 되돌릴 수 있다.
