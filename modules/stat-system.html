Core System

Stat System

스탯 시스템은 모든 전투·스킬·장비·상태·성장 시스템의 숫자 기반이다. 현재 C# 참조는 Stat·Modifier 계약뿐 아니라 Derived DAG 검증, 결정적 evaluator, 명시적 수치 lane과 versioned context cache를 실행한다. StatBlock·ResourceController·DebugBreakdown은 여전히 product 확장 설계다.

Prerequisites

선수 개념

스탯 시스템의 책임 경계

Stat System은 “최종 공격력이 얼마인가?”에 답하는 시스템이지, “그 공격력으로 피해량을 어떻게 계산하는가?”까지 담당하면 안 된다. 피해 공식은 Combat System의 책임이다.

핵심 객체 읽기

증거 범위. StatDefinition·StatInstance·StatBlock은 production 저장 모델을 설명하는 개념적 확장이다. 실행 기준은 Contracts/Stats.cs의 StatContext·StatModifier와 ReferenceDerivedStatEvaluator.cs의 DerivedStatDefinition·DerivedStatGraph·versioned cache다.
Modifier 원천 범위. 현재 C#과 JavaScript의 실행 SourceKind 계약은 SkillExecution/skill-execution, Status/status, System/system 세 종류만 지원한다. 아래 장비·아이템·패시브 Modifier는 product 확장 예시이며 전용 원천 종류는 실행 참조 범위 밖이다. 이를 독립 SourceKind로 도입할 때는 직렬화 schema와 저장·event·replay migration까지 함께 버전업해야 한다.

StatDefinition

스탯 종류의 설계도다. 예를 들어 attack_power, crit_chance, max_hp 같은 스탯의 표시명, 기본값, 최소/최대, 표시 형식, 반올림 정책을 정의한다.

StatInstance

특정 Entity가 가진 스탯 하나의 개념적 런타임 값이다. production 확장 시 BaseValue, CachedValue, IsDirty, Modifiers 같은 필드를 둘 수 있지만 현재 C# 실행 참조에 이 타입은 없다.

StatBlock

Entity 하나의 개념적 스탯 보관함이다. Player, Monster, Summon, Tower가 공유할 수 있는 목표 모델이며 현재 C# 실행 참조에 이 타입은 없다.

StatModifier

장비, 버프, 디버프, 패시브, 던전 룰처럼 스탯을 바꾸는 원인이다. 최종값을 직접 바꾸지 않고 Modifier를 등록한다.

C# 핵심 모델

Executable · abridged. 아래 코드는 공개 모양과 operation별 수치 guard를 보여 주는 축약이다. ID·SourceRef 검증을 포함한 전체 생성자는 Contracts/Stats.cs가 기준이다.

// Executable contract · abridged; ID/source guard bodies are omitted.
public enum ModifierOperation
{
    Add,
    PercentAdd,
    More,
    Less,
    Override
}

public sealed class StatModifier
{
    public EntityId ModifierId { get; }
    public EntityId StatId { get; }
    public ModifierOperation Operation { get; }
    public decimal Value { get; }
    public SourceRef Source { get; }
    public int Priority { get; }
    public EntityId StackRuleId { get; }
    public IModifierCondition? Condition { get; }

    public StatModifier(
        EntityId modifierId,
        EntityId statId,
        ModifierOperation operation,
        decimal value,
        SourceRef source,
        int priority,
        EntityId stackRuleId,
        IModifierCondition? condition = null)
    {
        if (!Enum.IsDefined(operation))
            throw new ArgumentOutOfRangeException(nameof(operation));
        if (operation == ModifierOperation.PercentAdd && value < -1m)
            throw new ArgumentOutOfRangeException(nameof(value));
        if (operation == ModifierOperation.More && value < 0m)
            throw new ArgumentOutOfRangeException(nameof(value));
        if (operation == ModifierOperation.Less && (value < 0m || value > 1m))
            throw new ArgumentOutOfRangeException(nameof(value));

        ModifierId = modifierId;
        StatId = statId;
        Operation = operation;
        Value = value;
        Source = source;
        Priority = priority;
        StackRuleId = stackRuleId;
        Condition = condition;
    }
}

public interface IModifierCondition
{
    bool Evaluate(StatContext context);
}

public interface IStatQuery
{
    decimal GetValue(EntityId ownerId, EntityId statId, StatContext context);
}
질의 불변식. ownerId는 반드시 context.OwnerId와 같아야 한다. 다르면 다른 소유자의 문맥으로 조건을 평가하게 되므로 유효하지 않은 질의로 거부한다.
StackRuleId는 현재 실행 정책이 아니다. 이 필드는 product loader가 “같은 규칙으로 묶을 modifier”를 식별할 수 있게 보존한 계약 값이다. 현재 ReferenceDerivedStatEvaluator는 StackRuleId별 중복 제거·최대 stack·source scope 병합을 하지 않고, 전달받은 모든 modifier를 결정 순서로 적용한다. 제품에서는 evaluator에 넣기 전에 stack policy가 accepted modifier snapshot을 만들고, 규칙 ID의 definition version도 cache 무효화 경계에 포함해야 한다.

Modifier는 왜 중요한가?

좋은 스탯 시스템은 숫자만 보관하지 않고 “왜 그 숫자가 되었는지”를 추적한다. 공격력 156이라는 결과보다 더 중요한 것은 기본값 100, 철검 +20, 분노 버프 +30%라는 원인이다. 

Attack Power Breakdown
Base: 100
Add:
  Iron Sword +20           source=item.iron-sword.001
PercentAdd:
  Rage Buff +30%           source=status.rage.42
Clamp:
  min=0 max=9999
Final: 156

계산 파이프라인

Modifier의 적용 순서는 결과를 바꾼다. 따라서 모든 프로젝트에서 같은 규칙을 쓰려면 파이프라인을 명시해야 한다.

단계 예시 설명
Base 기본 공격력 100 캐릭터 성장이나 기본 데이터에서 온 값
Add 검 +20, 반지 +10 고정값 보정의 합
PercentAdd 공격력 증가 +50% 증가율을 합산한 뒤 한 번 적용
More/Less 최종 공격력 ×1.2 독립 배율. 강력하므로 밸런스 주의
Override 이동속도 0 고정 빙결, 기절 등 강제 변경
Clamp 치명타 확률 0~100% 최소/최대 제한
Executable 수치 lane. StatScalar는 decimal 계산 lane, BasisPointRate는 정수 BPS lane이다. StatNumericLanes.ApplyRate는 두 lane의 변환 위치를 드러내고 ToCommitInteger만 MidpointRounding.AwayFromZero를 사용한다. 기존 StatModifier.Value는 compact decimal ratio(0.30m = 30%)이므로 저장 definition이 BPS라면 loader에서 명시적으로 변환해야 한다. 선택 근거와 Unity Burst 경계는 ADR-001에 기록했다.
Executable 계산 순서. Base + ΣAdd → × (1 + ΣPercentAdd) → × Π(1 + Moreᵢ) → × Π(1 - Lessᵢ) → Override → decimal Clamp까지 C# evaluator가 실행한다. 여러 PercentAdd의 합이 -100%보다 작으면 거부하고 decimal overflow도 숨기지 않는다. 정수 반올림은 evaluator 내부가 아니라 명시적인 commit 변환 경계에서 한 번만 호출한다.
동률 규칙. evaluator는 operation stage를 먼저 고정하고, 같은 stage에서는 (Priority, ModifierId Ordinal) 로 오름차순 정렬한다. ModifierId는 definition 안에서 유일하므로 입력 collection 순서가 결과를 바꾸지 않는다. Override는 Priority가 높은 항목이 이기고, Priority까지 같으면 Ordinal 정렬의 마지막 ID가 이긴다. SourceRef는 provenance이며 이 reference의 tie-break에는 사용하지 않는다.
실험. 아래 JavaScript 계산기는 한 개의 More·Less·Override로 단계 순서를 확인하는 근사 실험이다. binary Number와 Math.round를 사용하므로 C# decimal midpoint, 복수 보정자 정렬, Override 동률을 검증하는 replay 기준 구현이 아니다. Override를 비워 두면 앞 단계 계산값을 사용한다.

조건부 Modifier와 StatContext

모든 Modifier가 항상 적용되는 것은 아니다. “화상 상태의 적을 공격할 때 치명타 확률 +10%” 같은 효과는 현재 대상, 사용 스킬, 거리, 태그를 알아야 평가할 수 있다.

Executable · abridged. 아래는 Contracts/Stats.cs의 핵심 입력과 보관 형태를 옮긴 축약이다. ID·상태 목록 검증 본문은 생략했다. 비스킬 조회는 SkillId=null을 사용하고, 문자열 태그 입력은 생성 시 canonical TagSet으로 바뀐다.

// Executable contract · abridged; ID/status guard bodies are omitted.
public sealed class StatContext
{
    public StatContext(
        EntityId ownerId,
        EntityId? targetId,
        EntityId? skillId,
        IEnumerable<string>? skillTags = null,
        IEnumerable<string>? targetTags = null,
        IEnumerable<EntityId>? targetStatuses = null,
        decimal distance = 0m,
        string moment = "default")
    {
        if (distance < 0m)
            throw new ArgumentOutOfRangeException(nameof(distance));
        if (string.IsNullOrWhiteSpace(moment))
            throw new ArgumentException("A stat query moment is required.", nameof(moment));

        var skillTagSet = skillTags is null ? TagSet.Empty : new TagSet(skillTags);
        var targetTagSet = targetTags is null ? TagSet.Empty : new TagSet(targetTags);
        var targetStatusCopy = (targetStatuses ?? Enumerable.Empty<EntityId>()).ToArray();

        OwnerId = ownerId;
        TargetId = targetId;
        SkillId = skillId;
        SkillTags = skillTagSet;
        TargetTags = targetTagSet;
        TargetStatuses = Array.AsReadOnly(targetStatusCopy);
        Distance = distance;
        Moment = moment;
    }

    public EntityId OwnerId { get; }
    public EntityId? TargetId { get; }
    public EntityId? SkillId { get; }
    public TagSet SkillTags { get; }
    public TagSet TargetTags { get; }
    public IReadOnlyList<EntityId> TargetStatuses { get; }
    public decimal Distance { get; }
    public string Moment { get; }
}
Executable 범위. PercentAdd=-1은 단독 multiplier 0의 경계이고 그보다 작은 값은 생성 시 거부한다. More는 음수를, Less는 0~1 밖을 거부한다. evaluator는 적용 가능한 여러 PercentAdd를 합산해 최종 multiplier가 음수가 되는 조합도 실행 시 거부한다.

조건부 Modifier를 지원하지 않으면 장비 옵션, 패시브, 로그라이트 보상은 금방 하드코딩으로 무너진다.

조건부 계산과 캐시 의미론

target , 거리, 스킬 태그에 따라 값이 달라지는 Modifier가 있다면 StatInstance 하나에 “최종값 하나”만 캐시하면 잘못된 값을 재사용할 수 있다. 안전한 기본값은 무조건부 레이어까지만 캐시하고, 조건부 레이어는 정규화된 context key별로 평가하거나 질의 시 계산하는 것이다.

Safe default

Base cache

Base/Add/항상 활성인 Modifier만 owner 단위로 캐시한다.

Contextual

Query layer

대상·거리·moment가 필요한 Modifier는 context fingerprint를 키로 제한 캐시하거나 매번 계산한다.

Executable · correctness 우선 cache. ReferenceDerivedStatEvaluator는 caller가 명시한 StatEvaluationVersion(OwnerVersion, DefinitionVersion, NumericPolicyVersion), StatId, 전체 StatContext descriptor를 key로 사용한다. nullable ID는 presence bit를 가지며 hash가 같아도 전체 descriptor equality가 다르면 miss다. TagSet은 계약상 중복 제거·Ordinal 정렬하지만, formula가 관찰할 수 있는 TargetStatuses 목록 순서는 그대로 key에 남긴다.
현재 제한. formula에 context read-set 선언을 강제하지 않기 때문에 쓰지 않는 context 값도 key에 들어가 hit rate가 보수적이다. cache는 아직 크기 제한 없고 thread-safe하지 않은 in-memory Dictionary이며 ClearCache()만 제공한다. 같은 evaluator instance를 여러 thread에서 공유하지 말고, production adapter가 동기화·수명·상한을 소유해야 한다. LRU, owner별 제거, reverse-edge dirty invalidation은 production 확장이고 구현 증거가 아니다. 정상 canonical hit, 강제 hash collision, optional presence, 세 version 각각의 miss는 AdvancedFoundationVerification.cs가 검증하며 trade-off는 ADR-004에 정리했다.

Resource Stat 정책

max_hp, max_mana, shield_capacity 같은 최대치와 파생값은 Stat이 계산한다. 현재 HP, Mana, Shield는 자주 변하는 ResourceController의 상태이며 Stat 캐시 안에서 직접 증감하지 않는다. 아래 정책은 최대치가 바뀔 때 ResourceController가 현재값을 어떻게 동기화할지 정하는 경계 계약이다.

정책 동작 사용 예
KeepCurrent newMax ≥ current일 때만 현재값을 유지한다. 최대값 감소로 current가 초과하면 clamp하거나 변경을 거부한다. 증가 전용 마나 최대치 변경
KeepRatio 현재 비율을 유지 몬스터 레벨 스케일링
AddDelta 최대값 증가분만큼 현재값도 증가 장비로 최대 체력 증가
ClampOnly 현재값이 최대를 넘으면 자름 최대 스태미나 감소
Conceptual policy table. 현재 C# reference에는 ResourceController와 이 네 정책의 실행 구현이 없다. 제품 계약은 newMax < current에서 clamp할지 거부할지, KeepRatio의 oldMax = 0 의미, current × newMax / oldMax의 정수 lane·midpoint 반올림, AddDelta의 감소·overflow 처리까지 하나씩 고정해야 한다. 이 선택 없이 “비율 유지”만 구현하면 서버·클라이언트나 save migration 결과가 달라질 수 있다.

Derived Stat과 의존성

파생 스탯은 다른 스탯으로 계산되는 값이다. vitality가 바뀌면 max_hp도 dirty 처리되어야 하고, strength가 바뀌면 attack_power가 재계산되어야 한다.

Executable · Derived DAG. DerivedStatGraph는 중복 stat, missing dependency와 cycle을 구성 시 거부하고, 동시에 준비된 노드를 EntityId Ordinal로 정렬한다. evaluator는 요청 stat의 dependency closure만 실행한다. formula는 DeclaredStatValues를 통해 선언한 dependency만 읽을 수 있고, derived base 뒤에 자신의 local modifier와 optional min/max clamp가 적용된다. 변경 전파는 version key와 전체 ClearCache까지만 구현되어 있으며 reverse dependency별 invalidation은 아직 없다.
// Executable · abridged
EntityId vitalityId = new("stat.vitality");
EntityId maxHpId = new("stat.max-hp");

var vitality = new DerivedStatDefinition(
    vitalityId,
    new StatScalar(12m));
var maxHp = new DerivedStatDefinition(
    maxHpId,
    new StatScalar(100m),
    dependencies: new[] { vitalityId },
    formula: (baseValue, values, _) =>
        new StatScalar(
            baseValue.Value + values.Get(vitalityId).Value * 10m),
    minimumValue: new StatScalar(1m));

var evaluator = new ReferenceDerivedStatEvaluator(
    new DerivedStatGraph(new[] { maxHp, vitality }),
    new StatEvaluationVersion(
        ownerVersion: 8,
        definitionVersion: 3,
        numericPolicyVersion: 1));

decimal value = evaluator.GetValue(ownerId, maxHpId, context); // 220

저장/로드 원칙

최종값은 저장하지 않는 편이 좋다. 최종값은 공식과 Modifier의 결과이기 때문이다. 저장해야 하는 것은 원인이다.

저장하지 않는 것 저장하는 것
최종 공격력 156 기본 공격력, 장착 아이템, 활성 버프와 남은 시간
최종 이동속도 3.2 base move speed, 둔화 디버프 source, stack count
최종 치명타 확률 25% dexterity, 장비 옵션, 조건부 passive 상태

구현 안티패턴

  • 버프가 AttackPower += 30처럼 최종값을 직접 수정하는 구조
  • 장비 해제 시 어떤 보정자를 제거해야 하는지 source 추적이 없는 구조
  • Stat System 안에 damage formula를 넣어 Combat System과 결합하는 구조
  • 파생 스탯 의존성 그래프 없이 수동으로 재계산하는 구조
  • 디버그 breakdown 없이 최종 숫자만 표시하는 구조

Combat / Status가 Stat을 읽는 방식

Combat과 Status는 Stat System을 강하게 확장하지만, Stat의 책임을 침범해서는 안 된다. Combat은 공격자와 방어자의 최종 스탯 값을 읽어 피해를 계산하고, Status는 상태가 제공하는 StatModifier를 등록/제거한다.

연결 Stat의 역할 주의점
Combat → Stat attack_power, spell_power, fire_resistance, armor, shield_capacity 같은 최종값 제공 Combat은 Modifier를 직접 등록하지 않는다.
Status → Stat Rage, Freeze, Poison 같은 상태가 StatModifier를 등록한다. Status는 attackPower 값을 직접 더하지 않는다.
Status 제거 → Stat SourceRef.StatusInstanceId가 일치하는 Modifier를 제거한다. 현재 HP나 Shield는 ResourceController가 별도로 관리한다. 구조화된 SourceRef가 없으면 중첩과 제거 디버그가 어려워진다.
DebugTrace statReads와 modifierSources를 계산 로그에 제공한다. 피해 결과와 상태 결과를 추적 가능하게 한다.

Conceptual product adapter pseudocode. 아래 StatusInstance, _stats.AddModifier, RemoveModifiersBySource는 현재 C# reference에 없는 저장·조율 API다. 실행되는 부분의 값 계약은 StatModifier와 SourceRef.Status이며, 실제 제품 adapter가 등록 저장소와 제거 원자성을 구현해야 한다.

// Conceptual product adapter pseudocode
public void ApplyRage(StatusInstance rage)
{
    _stats.AddModifier(new StatModifier(
        modifierId: new EntityId("modifier.rage.attack-power"),
        statId: new EntityId("stat.attack-power"),
        operation: ModifierOperation.PercentAdd,
        value: 0.30m,
        source: SourceRef.Status(new EntityId("status.rage"), rage.InstanceId),
        priority: 0,
        stackRuleId: new EntityId("stack-rule.stack")));
}

public void RemoveRage(StatusInstance rage)
{
    _stats.RemoveModifiersBySource(rage.InstanceId);
}
Comprehension check

이해도 확인

최종 수치 계산, 현재 리소스 상태, 조건부 문맥이 각각 어느 경계에 속하는지 먼저 판단한 뒤 해설을 확인한다.

  1. 01 · 책임 경계
    장비로 max_hp가 100에서 120으로 증가할 때 Stat과 ResourceController는 각각 무엇을 결정해야 할까?
    정답과 해설

    Stat은 최종 최대치 120을 계산한다. 현재 HP 변경은 ResourceController가 KeepCurrent, KeepRatio, AddDelta 같은 동기화 정책에 따라 처리한다.

  2. 02 · 경계 상황
    Burn 대상 공격 시 치명타 확률 +10% 결과를 Burn이 없는 대상에게도 캐시 재사용하면 무엇이 잘못될까?
    정답과 해설

    대상과 상태가 다른데 조건부 값을 재사용하게 된다. 현재 C# reference는 전체 StatContext descriptor와 owner·definition·numeric policy version이 모두 같을 때만 hit로 판정한다. 선언 read-set cache를 도입한다면 누락 추적 검증이 먼저 필요하다.

  3. 03 · 시스템 연결
    Rage가 적용되고 제거되는 동안 Status, Stat, Combat은 각각 어떤 역할을 맡아야 할까?
    정답과 해설

    Status가 StatusInstanceId를 포함한 SourceRef의 Modifier를 등록·제거하고 Stat이 최종 공격력을 계산한다. Combat은 계산된 값을 읽을 뿐 Modifier나 원본 값을 직접 변경하지 않는다.

설계 점검

  • StatDefinition과 StatInstance가 분리되어 있다.
  • Modifier는 SourceRef, Operation, Priority, StackRuleId, Condition을 가진다.
  • Base/Add/PercentAdd/More/Less/Override/Clamp/Round 계산 순서, 비율 단위와 Override 동률 규칙이 명시되어 있다.
  • Resource max 변경 정책이 데이터로 결정된다.
  • 조건부 Modifier가 StatContext로 평가된다.
  • Derived DAG와 versioned context cache의 실행 범위, 아직 없는 per-owner invalidation·LRU·DebugBreakdown·StatChangedEvent의 확장 범위가 구분되어 있다.
