Core System

Stat System

스탯 시스템은 모든 전투·스킬·장비·상태·성장 시스템의 숫자 기반이다. 공통화할 것은 숫자 관리 엔진이고, 게임별로 바뀌어야 하는 것은 스탯 목록과 공식이다.

StatBlockModifierResourceDerived StatDebug

스탯 시스템의 책임 경계

Stat System은 “최종 공격력이 얼마인가?”에 답하는 시스템이지, “그 공격력으로 피해량을 어떻게 계산하는가?”까지 담당하면 안 된다. 피해 공식은 Combat System의 책임이다.

담당한다담당하지 않는다
스탯 정의와 기본값 관리게임별 피해 공식
Modifier 추가/제거/중첩명중, 회피, 치명타 판정
최종값 계산과 캐시스킬 입력과 애니메이션 타이밍
자원형 스탯 정책몬스터 AI 의사결정
변경 이벤트와 디버그 로그보상 선택 로직

핵심 객체 읽기

StatDefinition

스탯 종류의 설계도다. 예를 들어 attack_power, crit_chance, max_hp 같은 스탯의 표시명, 기본값, 최소/최대, 표시 형식, 반올림 정책을 정의한다.

StatInstance

특정 Entity가 가진 스탯 하나의 런타임 값이다. baseValue, cachedValue, dirty flag, active modifiers를 가진다.

StatBlock

Entity 하나의 스탯 보관함이다. Player, Monster, Summon, Tower가 모두 StatBlock을 가질 수 있다.

StatModifier

장비, 버프, 디버프, 패시브, 던전 룰처럼 스탯을 바꾸는 원인이다. 최종값을 직접 바꾸지 않고 Modifier를 등록한다.

Modifier는 왜 중요한가?

좋은 스탯 시스템은 숫자만 보관하지 않고 “왜 그 숫자가 되었는지”를 추적한다. 공격력 156이라는 결과보다 더 중요한 것은 기본값 100, 철검 +20, 분노 버프 +30%라는 원인이다.

Attack Power Breakdown
Base: 100
Add:
  Iron Sword +20           source=item:iron_sword_001
PercentMultiply:
  Rage Buff +30%           source=status:rage
Clamp:
  min=0 max=9999
Final: 156

계산 파이프라인

Modifier의 적용 순서는 결과를 바꾼다. 따라서 모든 프로젝트에서 같은 규칙을 쓰려면 파이프라인을 명시해야 한다.

단계예시설명
Base기본 공격력 100캐릭터 성장이나 기본 데이터에서 온 값
Add검 +20, 반지 +10고정값 보정의 합
PercentAdd공격력 증가 +50%증가율을 합산한 뒤 한 번 적용
More/Less최종 공격력 ×1.2독립 배율. 강력하므로 밸런스 주의
Override이동속도 0 고정빙결, 기절 등 강제 변경
Clamp치명타 확률 0~100%최소/최대 제한
실험. 아래 계산기는 Modifier 적용 순서가 결과에 어떤 영향을 주는지 빠르게 확인하기 위한 간단한 예시다.

조건부 Modifier와 StatContext

모든 Modifier가 항상 적용되는 것은 아니다. “화상 상태의 적을 공격할 때 치명타 확률 +10%” 같은 효과는 현재 대상, 사용 스킬, 거리, 태그를 알아야 평가할 수 있다.

StatContext
- owner: attacker
- target: defender
- skillId: fire_slash
- skillTags: [melee, fire, skill]
- targetTags: [enemy, boss]
- targetStatuses: [burn]
- distance: 2.4
- moment: OnAttackHit

조건부 Modifier를 지원하지 않으면 장비 옵션, 패시브, 로그라이트 보상은 금방 하드코딩으로 무너진다.

Resource Stat 정책

HP, Mana, Stamina처럼 현재값과 최대값이 나뉘는 스탯은 일반 스탯보다 정책이 더 필요하다. 최대값 변경 시 현재값을 어떻게 처리할지가 게임마다 다르기 때문이다.

정책동작사용 예
KeepCurrent현재값을 유지하고 최대값만 바꿈마나 최대치 증가
KeepRatio현재 비율을 유지몬스터 레벨 스케일링
AddDelta최대값 증가분만큼 현재값도 증가장비로 최대 체력 증가
ClampOnly현재값이 최대를 넘으면 자름최대 스태미나 감소

Derived Stat과 의존성

파생 스탯은 다른 스탯으로 계산되는 값이다. vitality가 바뀌면 max_hp도 dirty 처리되어야 하고, strength가 바뀌면 attack_power가 재계산되어야 한다.

max_hp = 100 + vitality * 10
attack_power = weapon_attack + strength * 2
crit_chance = clamp(0, 1, dexterity * 0.002 + item_bonus)

저장/로드 원칙

최종값은 저장하지 않는 편이 좋다. 최종값은 공식과 Modifier의 결과이기 때문이다. 저장해야 하는 것은 원인이다.

저장하지 않는 것저장하는 것
최종 공격력 156기본 공격력, 장착 아이템, 활성 버프와 남은 시간
최종 이동속도 3.2base move speed, 둔화 디버프 source, stack count
최종 치명타 확률 25%dexterity, 장비 옵션, 조건부 passive 상태

구현 안티패턴

  • 버프가 attackPower += 30처럼 최종값을 직접 수정하는 구조
  • 장비 해제 시 어떤 보정자를 제거해야 하는지 source 추적이 없는 구조
  • Stat System 안에 damage formula를 넣어 Combat System과 결합하는 구조
  • 파생 스탯 의존성 그래프 없이 수동으로 재계산하는 구조
  • 디버그 breakdown 없이 최종 숫자만 표시하는 구조

완료 체크리스트

  • StatDefinition과 StatInstance가 분리되어 있다.
  • Modifier는 sourceId, operation, priority, stackRule, condition을 가진다.
  • Base/Add/Percent/More/Override/Clamp 계산 순서가 명시되어 있다.
  • Resource max 변경 정책이 데이터로 결정된다.
  • 조건부 Modifier가 StatContext로 평가된다.
  • StatChangedEvent와 DebugBreakdown이 제공된다.

Combat / Status가 Stat을 읽는 방식

Combat과 Status는 Stat System을 강하게 확장하지만, Stat의 책임을 침범해서는 안 된다. Combat은 공격자와 방어자의 최종 스탯 값을 읽어 피해를 계산하고, Status는 상태가 제공하는 StatModifier를 등록/제거한다.

연결Stat의 역할주의점
Combat → Statattack_power, spell_power, fire_resistance, armor, shield 같은 최종값 제공Combat은 Modifier를 직접 등록하지 않는다.
Status → StatRage, Freeze, Poison 같은 상태가 StatModifier를 등록한다.Status는 attackPower 값을 직접 더하지 않는다.
Status 제거 → StatsourceId=statusInstanceId인 Modifier를 제거한다.sourceId가 없으면 중첩과 제거 디버그가 어려워진다.
DebugTracestatReads와 modifierSources를 계산 로그에 제공한다.피해 결과와 상태 결과를 추적 가능하게 한다.
Rage Status Applied
  sourceId = statusInstance:rage:player_001:42
  add StatModifier(attack_power, percent_add, +0.30, sourceId)

Rage Status Removed
  removeModifiersBySource(sourceId)
  recalculate attack_power if dirty
