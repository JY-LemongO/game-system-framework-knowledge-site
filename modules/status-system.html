Status System

Status System은 ApplyStatusEffect로 부여된 버프, 디버프, DoT, HoT, CC를 지속시간, 중첩, tick, 해제, 면역, StatModifier 연결까지 관리하는 런타임 상태 계층이다. 상태는 값을 직접 바꾸지 않고 Modifier를 제공하거나 주기 Effect를 실행한다.

StatusDefinitionStackDurationTickDispelModifier

Status의 역할

Effect는 “Burn을 부여해라”라고 요청할 수 있다. 하지만 Burn이 이미 있을 때 중첩할지, 지속시간을 갱신할지, 1초마다 피해를 줄지, 제거될 때 어떤 Modifier를 정리할지는 Status System이 담당한다.

핵심 계약. Status는 스탯 값을 직접 수정하지 않는다. 상태를 누가 부여했는지는 appliedBySourceId로 남기고, 상태가 제공하는 StatModifier와 tick 예약은 sourceId = statusInstanceId로 등록한다. 제거할 때는 같은 sourceId로 안전하게 정리한다. 주기 피해가 필요하면 Status tick에서 Effect를 실행한다.
담당한다담당하지 않는다
상태 부여, 갱신, 중첩, 제거피해 공식 자체
지속시간과 tick 스케줄 관리최종 스탯 계산 파이프라인
상태가 제공하는 StatModifier 등록/해제스킬 입력과 쿨다운 관리
주기 Effect 실행과 이벤트 반응 Hook장비 옵션 생성
면역, 정화, 태그 기반 제거 규칙UI 아이콘 렌더링
StatusResult와 DebugTrace 기록AI 의사결정

핵심 객체

객체책임
StatusDefinition상태의 정적 데이터. 태그, 지속시간, 중첩, tick, modifier, periodic effect를 정의한다.
StatusInstance특정 Entity에게 적용된 상태의 런타임 데이터. instanceId, target, appliedBySourceId, remainingTime, stackCount, nextTickTime을 가진다.
StatusControllerEntity 하나의 상태 목록을 관리하고 apply, refresh, remove, tick을 처리한다.
StackPolicyrefresh, add stack, replace, independent stack 같은 재적용 규칙을 결정한다.
DurationPolicy지속시간 갱신, 연장, 고정, 감소, 만료 규칙을 결정한다.
TickPolicyinterval, 턴 시작, 턴 종료, fixed tick 등 주기 실행 방식을 결정한다.
StatusModifierProvider상태가 제공하는 StatModifier를 생성하고 modifier sourceId를 statusInstanceId로 추적한다.
StatusEffectHooktick 또는 이벤트 반응으로 Effect를 실행한다.
DispelPolicy정화, 해제, 면역, 태그 기반 제거를 처리한다.
StatusResult적용, 갱신, 중첩 변경, 면역, 제거, tick 예약, DebugTrace를 담는 상태 처리 결과다.

상태 생명주기

  1. ApplyStatusEffect가 StatusSystem에 적용 요청을 보낸다.
  2. 면역, 태그, 대상 생존 여부를 검사한다.
  3. 기존 StatusInstance가 있으면 StackPolicy와 DurationPolicy를 적용한다.
  4. 새 상태라면 StatusInstance를 만들고 Modifier를 등록한다.
  5. TickPolicy가 있으면 다음 tick을 예약한다.
  6. 만료, 정화, 사망, 태그 제거 조건이 발생하면 상태를 제거한다.
  7. 제거 시 StatusInstance, StatModifier, 예약 tick, 이벤트 구독을 정리한다.

ApplyStatusEffect와의 연결

{
  "effect": "effect_apply_burn",
  "request": {
    "statusId": "status_burn",
    "target": "enemy_001",
    "appliedBySourceId": "skill:fireball:cast_771",
    "durationScale": 1.0,
    "stackDelta": 1,
    "tags": ["fire", "debuff", "dot"]
  }
}

Tick 흐름

Burn 같은 DoT는 tick 시점마다 DamageEffect를 실행한다. 이때 피해량은 적용 시점의 스탯을 고정하는 snapshot 방식과 매 tick마다 다시 계산하는 dynamic 방식 중 하나를 선택해야 한다.

방식설명장점주의점
Snapshot상태 적용 시점의 공격자 스탯과 계수를 저장한다.재현성이 좋고 계산이 단순하다.버프가 사라져도 DoT 피해가 유지될 수 있다.
Dynamic Recalculationtick마다 현재 스탯과 상태를 다시 읽는다.현재 전투 상황을 잘 반영한다.계산량과 디버그 복잡도가 증가한다.
Hybrid기본 계수는 snapshot, 대상 저항은 dynamic으로 처리한다.밸런스 의도를 세밀하게 반영한다.정책 문서화가 필수다.

Burn 예시

Burn은 Status System을 설명하기 좋은 대표 DoT다. 화염 태그와 디버프 태그를 가지며, 재적용 시 중첩을 올리고 지속시간을 갱신한다.

{
  "id": "status_burn",
  "displayName": "Burn",
  "tags": ["status", "debuff", "fire", "dot"],
  "duration": { "base": 5.0, "refreshPolicy": "refresh_to_max" },
  "stacking": {
    "type": "intensity_stack",
    "maxStacks": 3,
    "onReapply": "add_stack_and_refresh"
  },
  "tick": { "interval": 1.0, "start": "after_first_interval" },
  "periodicEffects": [
    { "effect": "effect_burn_tick_damage", "timing": "on_tick" }
  ],
  "removeConditions": ["target_dies", "dispel_fire_debuff"]
}

Rage / Freeze / Poison 예시

Buff

Rage

공격력 +30% Modifier를 제공한다. 재적용 시 지속시간만 refresh하고 제거 시 sourceId로 Modifier를 정리한다.

CC

Freeze

move_speed override 0, attack_speed 감소 Modifier를 제공한다. 피격 시 해제 또는 ice 태그 면역을 정책으로 둘 수 있다.

DoT

Poison

주기 피해와 회복량 감소 Modifier를 동시에 제공할 수 있다. stackCount가 tick 피해량에 반영된다.

{
  "id": "status_rage",
  "displayName": "Rage",
  "tags": ["status", "buff", "offense"],
  "duration": { "base": 8.0, "refreshPolicy": "refresh_to_max" },
  "stacking": { "type": "refresh_only", "maxStacks": 1 },
  "modifiers": [
    {
      "stat": "attack_power",
      "operation": "percent_add",
      "value": 0.3,
      "source": "status_instance"
    }
  ]
}

면역, 정화, 해제

개념설명예시
Immunity상태 적용 자체를 거부한다.fire_immune 대상에게 Burn 적용 실패
Status Resistance상태 적용 확률이나 지속시간을 줄인다. Combat의 피해 저항과 이름을 구분해 관리한다.poison_status_resistance 50% → 지속시간 절반
Cleanse특정 태그의 상태를 제거한다.dispel_fire_debuff → Burn 제거
Dispel버프 또는 디버프를 정책에 따라 제거한다.remove_tag: buff 또는 debuff
Protection상태를 막는 별도 보호막을 먼저 소모한다.1회 CC 면역 보호막

안티패턴

안티패턴문제대안
Status가 attackPower 값을 직접 더함중첩, 제거, 저장, 디버그가 어려워진다.StatModifier를 등록하고 sourceId로 제거한다.
Tick이 HP를 직접 깎음Combat, Effect, Log, Event 흐름이 우회된다.PeriodicEffect에서 DamageEffect를 실행한다.
재적용 규칙이 코드에 흩어짐상태별 중첩 버그가 반복된다.StackPolicy와 DurationPolicy를 데이터로 명시한다.
면역과 정화를 문자열 if문으로 처리태그가 늘어날수록 분기가 폭발한다.TagSet과 DispelPolicy로 처리한다.

완료 체크리스트

  • StatusDefinition과 StatusInstance가 분리되어 있다.
  • StackPolicy, DurationPolicy, TickPolicy가 상태별로 명시되어 있다.
  • Status가 제공하는 Modifier와 tick 예약은 statusInstanceId를 sourceId로 가진다.
  • 상태 제거 시 Modifier, tick 예약, 이벤트 구독이 함께 정리된다.
  • DoT/HoT/aura는 직접 값을 수정하지 않고 periodic Effect로 실행된다.
  • 면역, 저항, 정화, 해제 정책이 TagSet 기반으로 설명된다.
  • Burn과 Rage 예제가 Stat, Effect, Combat과 연결된다.
