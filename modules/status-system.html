Status System

Status System은 목표형 StatusDefinition·StatusInstance 저장 모델, 순수 C# 재적용·지속시간·면역·정화 정책, JavaScript Burn tick 실행 참조를 구분해 버프, 디버프, DoT, HoT, CC를 설명한다. 상태는 값을 직접 바꾸지 않고 Modifier를 제공하거나 주기 Effect를 요청한다.

Status의 역할

Effect는 “Burn을 부여해라”라고 요청할 수 있다. 하지만 Burn이 이미 있을 때 중첩할지, 지속시간을 갱신할지, 주기 피해를 언제 계산할지, 제거될 때 어떤 Modifier를 정리할지는 Status System이 담당한다. 적용 조건도 hit, any_damage, hp_damage 중 하나로 명시해야 한다.

핵심 계약. Status는 스탯 값을 직접 수정하지 않는다. 상태를 누가 부여했는지는 SourceRef 로 남기고, 상태가 제공하는 StatModifier와 tick 예약은 SourceRef.Status(StatusDefinitionId, StatusInstanceId) 로 등록한다. 제거할 때는 SourceRef.StatusInstanceId로 안전하게 정리한다. 주기 피해가 필요하면 Status tick에서 Effect를 실행한다.

개념적 정의와 C# compact 서비스 계약

StatusDefinition 부분은 production 데이터 모델을 설명하는 Conceptual extension이다. ApplyStatusRequest부터는 실행 C# 계약의 abridged presentation이며, ID·enum·성공/실패 조합 guard를 포함한다. 대상별 StatusInstance 저장소와 원자적 commit을 수행하는 전체 service는 포함하지 않는다.

public sealed class StatusDefinition
{
    public EntityId StatusId { get; }
    public int DurationTicks { get; }
    public int TickInterval { get; }
    public int MaxCatchUpTicks { get; }
    public int MaxStacks { get; }
    public IReadOnlyList<EntityId> PeriodicEffectIds { get; }

    public StatusDefinition(
        EntityId statusId,
        int durationTicks,
        int tickInterval,
        int maxCatchUpTicks,
        int maxStacks,
        EntityId[] periodicEffectIds)
    {
        if (durationTicks < 0)
            throw new ArgumentOutOfRangeException(nameof(durationTicks));
        if (maxStacks < 1)
            throw new ArgumentOutOfRangeException(nameof(maxStacks));
        if (maxCatchUpTicks < 1)
            throw new ArgumentOutOfRangeException(nameof(maxCatchUpTicks));
        if (periodicEffectIds == null)
            throw new ArgumentNullException(nameof(periodicEffectIds));
        if (periodicEffectIds.Length > 0 && tickInterval <= 0)
            throw new ArgumentOutOfRangeException(nameof(tickInterval));

        StatusId = statusId;
        DurationTicks = durationTicks;
        TickInterval = tickInterval;
        MaxCatchUpTicks = maxCatchUpTicks;
        MaxStacks = maxStacks;
        PeriodicEffectIds = Array.AsReadOnly((EntityId[])periodicEffectIds.Clone());
    }
}

public sealed class ApplyStatusRequest
{
    public EntityId StatusId { get; }
    public EntityId TargetId { get; }
    public SourceRef Source { get; }
    public int StackDelta { get; }

    public ApplyStatusRequest(
        EntityId statusId,
        EntityId targetId,
        SourceRef source,
        int stackDelta)
    {
        EntityId.ThrowIfInvalid(statusId, nameof(statusId));
        EntityId.ThrowIfInvalid(targetId, nameof(targetId));
        SourceRef.ThrowIfInvalid(source, nameof(source));

        if (stackDelta <= 0)
            throw new ArgumentOutOfRangeException(
                nameof(stackDelta),
                "Application adds stacks; removal uses a separate command.");

        StatusId = statusId;
        TargetId = targetId;
        Source = source;
        StackDelta = stackDelta;
    }
}

public sealed class StatusResult
{
    private StatusResult(
        bool succeeded,
        EntityId? statusInstanceId,
        StatusRemoveReason? removeReason,
        StatusFailureReason? failureReason)
    {
        Succeeded = succeeded;
        StatusInstanceId = statusInstanceId;
        RemoveReason = removeReason;
        FailureReason = failureReason;
    }

    public bool Succeeded { get; }
    public EntityId? StatusInstanceId { get; }
    public StatusRemoveReason? RemoveReason { get; }
    public StatusFailureReason? FailureReason { get; }

    public static StatusResult Applied(EntityId instanceId) =>
        new(true, ValidInstanceId(instanceId), null, null);

    public static StatusResult Removed(
        EntityId instanceId,
        StatusRemoveReason reason)
    {
        if (!Enum.IsDefined(reason))
            throw new ArgumentOutOfRangeException(nameof(reason));
        return new(true, ValidInstanceId(instanceId), reason, null);
    }

    public static StatusResult Failed(StatusFailureReason reason)
    {
        if (!Enum.IsDefined(reason))
            throw new ArgumentOutOfRangeException(nameof(reason));
        return new(false, null, null, reason);
    }

    private static EntityId ValidInstanceId(EntityId instanceId)
    {
        EntityId.ThrowIfInvalid(instanceId, nameof(instanceId));
        return instanceId;
    }
}

public enum StatusFailureReason
{
    NotFound,
    Immune,
    StackPolicyRejected
}

public interface IStatusService
{
    StatusResult Apply(ApplyStatusRequest request);
    StatusResult Remove(EntityId instanceId, StatusRemoveReason reason);
    void AdvanceTo(long simulationTick);
}

StatusResult는 공개 생성자 대신 Applied·Removed·Failed factory만 제공한다. 성공 적용은 instance와 no removal/failure reason, 성공 제거는 instance와 removal reason, 실패는 failure reason과 no instance라는 유효 조합만 생성한다.

실행 범위 경계. C# 참조는 요청·결과 계약, catch-up, 아래 재적용·지속시간·면역·정화 결정을 실행 검증한다. 그러나 StatusInstance 저장소, Modifier·tick 예약 조율, version precondition, 실제 제거와 outbox를 확정하는 commit service는 여전히 Conceptual extension이다. JavaScript fixture는 단일 Burn instance의 최초 적용·tick·expire 경로만 실행한다.

실행 C# 재적용 정책

ReferenceStatusReapplicationPolicy는 엔진과 저장소에 의존하지 않는 순수 결정 함수다. 입력은 새 적용 요청과 한 시점의 active instance snapshot이고, 출력은 만들거나 갱신할 불변 instance와 제거할 기존 ID다. 이 함수는 저장소를 수정하지 않으며, 호출자는 반환값을 version precondition이 있는 commit plan으로 바꿔야 한다. 정상·경계·실패 진리표는 StatusPolicyVerification.cs에서 실행한다.

지속시간 진리표와 overflow 경계

새 생성·교체·independent는 언제나 checked(CurrentTick + DurationTicks)의 fresh expiry를 사용한다. 아래 세 정책은 기존 instance를 병합할 때만 적용한다. DurationTicks는 1 이상이며 이미 만료된 instance를 active로 넘기면 실패한다.

C# 정책과 JavaScript fixture의 duration 경계는 다르다.

C# ReferenceStatusReapplicationPolicy는 active instance 정책이므로 DurationTicks >= 1만 허용하고 저항 적용 뒤에도 최소 1 tick을 보장한다. 반면 JavaScript Burn fixture는 스케줄러 경계 증거를 위해 durationTicks=0과 tick interval보다 짧은 1도 허용하며, 이 경우 periodic damage 없이 적용 시각 또는 +1에 정확히 한 번 만료한다. 한 구현의 입력 범위를 다른 구현의 보편 규칙으로 일반화하면 안 된다.

DurationBehavior공식효과와 주의점
RefreshFromCurrentTicknow + duration남은 시간이 길어도 새 기준으로 바꾸므로 만료 시각이 앞당겨질 수 있다.

BPS 지속시간 저항

ReferenceStatusDurationPolicy.ScaleByResistance는 정의의 base duration을 먼저 유효 duration으로 바꾼다. 공식은 baseTicks × (10,000 - resistanceBps) / 10,000이고, decimal 중간값을 유지한 뒤 MidpointRounding.AwayFromZero로 단 한 번 반올림하며 최소 1 tick을 보장한다. 그 결과를 새 생성 또는 위 재적용 expiry 공식의 duration 입력으로 사용한다.

저항 범위는 0..9,000 BPS로 고정한다. 100%를 넣어 duration 0으로 만드는 방식은 “적용됐지만 즉시 만료”와 “적용 자체가 거부됨”을 섞으므로 사용하지 않는다. 완전 면역은 별도 tag gate의 Immune 결과다. 반올림 정책이나 저항 상한을 바꾸면 replay와 balance가 달라지므로 numeric policy version도 함께 올려야 한다.

decimal 결과의 long 변환과 모든 expiry 덧셈은 checked로 계산한다. 저항 배율은 0..1이어서 유효 duration은 base보다 커지지 않으며 base가 long.MaxValue이고 저항이 0%인 경계도 검증한다. 반면 now + duration이나 oldExpiry + duration이 범위를 넘으면 포화시키거나 되감지 않고 OverflowException으로 실패해 저장 데이터·설정 오류를 드러낸다.

면역도 값 결과다. 실행 정책은 incoming StatusTags와 대상 정책이 제공한 BlockedStatusTags의 교집합을 canonical 순서로 찾는다. 하나라도 일치하면 Immune와 일치 태그를 반환하고 생성·갱신·제거를 모두 계획하지 않는다. status.immunity.cc 같은 게임 데이터 표기를 실제 blocked category cc로 해석하는 일은 설정/adapter 경계의 책임이다.

상태 생명주기

  1. ApplyStatusEffect가 StatusSystem에 적용 요청을 보낸다.
  2. 면역, 태그, 대상 생존 여부를 현재 snapshot에서 검사한다. committed damage event의 과거 생존 값은 적용 후보를 만드는 근거일 뿐이며, reaction dispatch 시 live target을 다시 읽는다.
  3. 기존 StatusInstance가 있으면 StackPolicy와 DurationPolicy를 적용한다.
  4. 새 상태라면 StatusInstance를 만들고 Modifier를 등록한다.
  5. TickPolicy가 있으면 다음 tick을 예약한다.
  6. 만료, 정화, 사망, 태그 제거 조건이 발생하면 상태를 제거한다.
  7. 제거 시 StatusInstance, StatModifier, 예약 tick, 이벤트 구독을 정리한다.
Reaction-time 재검증.

주 피해 commit과 Status 적용은 서로 다른 transaction이다. 그 사이 대상이 사망할 수 있으므로 적용 handler는 현재 HP와 version을 다시 확인한다. JavaScript 실행 참조는 이미 죽은 대상이면 NotApplicable / TARGET_NOT_ALIVE를 반환하고 Status instance와 StatusApplied를 만들지 않는다. 살아 있는 snapshot으로 plan을 만든 뒤 version이 바뀌면 commit precondition 실패로 전체를 거부해야 한다.

인스턴스 키와 동일 시각 순서

재적용 시 같은 상태로 볼 기준을 먼저 정한다. 기본 키는 (TargetId, StatusDefinitionId, StackingGroup, SourceScope) 이며 source의 실제 비교 범위는 위 실행 정책의 AnySource·SourceDefinition·SourceInstance 중 하나로 고정한다. independent stack은 identity 검색 결과를 병합하지 않고 각자 고유한 StatusInstanceId를 가진다. expire와 tick이 같은 시각이면 어느 쪽을 먼저 처리할지도 clock domain마다 고정한다. 이 사이트의 fixed simulation tick 계약은 tickAt ≤ expireAt 인 마지막 tick 계산을 먼저 plan에 넣고, 같은 원자적 commit에서 피해·다음 상태·outbox와 함께 expire를 확정한다.

triggerFact·outcomePredicate·durationScale·tag는 EffectDefinition과 Status 정책에서 먼저 해석한다. IStatusService에 전달되는 ApplyStatusRequest는 StatusId, TargetId, SourceRef, StackDelta만 가진다.

Tick 흐름

일반 설계에서는 Burn 같은 DoT가 tick마다 DamageEffect/DamageRequest를 만들 수 있다. 이 저장소의 JavaScript fixture는 별도 EffectContext 객체를 만들지 않고 resolveDamageAgainstTarget에 snapshot raw tick과 live target을 전달하지만, 같은 저항·보호막 계산과 원자적 commit 계약을 재사용한다. StateStore가 HP·Shield, 다음 tick 또는 만료 상태, outbox와 사망 사실을 하나의 plan으로 확정한다. 이때 공격 계수는 적용 시점에 고정하는 snapshot 방식과 매 tick 다시 계산하는 dynamic 방식 중 하나를 선택해야 한다.

방식 설명 장점 주의점
Snapshot 상태 적용 시점의 공격자 스탯과 계수를 저장한다. 재현성이 좋고 계산이 단순하다. 버프가 사라져도 DoT 피해가 유지될 수 있다.
Dynamic Recalculation tick마다 현재 스탯과 상태를 다시 읽는다. 현재 전투 상황을 잘 반영한다. 계산량과 디버그 복잡도가 증가한다.
Hybrid 기본 계수는 snapshot, 대상 저항은 dynamic으로 처리한다. 밸런스 의도를 세밀하게 반영한다. 정책 문서화가 필수다.
Snapshot 주의. Caster 객체 참조를 보관하지 말고 필요한 계수·태그·DefinitionVersion을 불변 스냅샷으로 저장한다. 대상 저항을 dynamic으로 읽는다면 그 정책도 trace에 남긴다.
실행 참조의 시간 단조성.

복원한 active Status의 appliedTick은 state clock보다 미래일 수 없다. 새 status.add는 commit 시각에 적용되고 첫 nextTickAt은 그보다 뒤여야 한다. status.patch는 기존 nextTickAt보다 큰 값으로만 진행한다. 과거 시각으로 되감아 overdue tick을 중복 실행하는 입력은 commit 전에 거부된다. 순수 C# 재적용 정책은 stack과 expiry 결정을 제공하지만 JavaScript commit fixture 및 NextTickAt phase와 연결되지 않았으므로, 같은 instance ID를 add해 덮어쓰는 방식으로 둘을 임의 결합하면 안 된다.

개념적 C# Tick 조율(비실행 의사코드)

아래 코드는 production tick 조율 순서를 설명하는 Conceptual extension이다. C#은 StatusCatchUpPolicy.Evaluate와 별도의 재적용·지속시간 정책을 검증하지만 StatusSnapshot·EffectPlan·StatusTransition·BuildTickPlan·StatusService는 구현하지 않는다. JavaScript fixture는 마지막 tick과 만료가 같을 때 tick을 먼저 계산하고 같은 원자적 commit에서 만료를 확정하며, 한 advance의 MaxCatchUpTicks와 CatchUpLimited 정책을 실행 검증한다.

Catch-up의 호출 주기 의존성. 이 정책은 한 번의 AdvanceTo마다 상한을 적용하고, 이미 만료된 backlog가 남으면 그 tick을 버리고 제거한다. tick 2·4·6, 만료 6, 상한 2인 상태는 AdvanceTo(6) 한 번이면 tick 6을 잃지만 AdvanceTo(4) 뒤 AdvanceTo(6)을 호출하면 세 번 모두 처리된다. 따라서 frame 지연과 무관한 tick 횟수 불변식은 없다. replay가 같은 결과를 요구하면 advance 호출 cadence를 보존하거나 별도의 backlog 정책을 정의해야 한다.
private void AdvanceInstance(StatusSnapshot status, long now)
{
    int processedTicks = 0;

    while (IsTickDue(status, now) &&
           processedTicks < status.MaxCatchUpTicks)
    {
        long tickAt = status.NextTickAt;
        long nextTickAt = checked(tickAt + status.TickInterval);
        bool expireWithThisTick =
            status.ExpiresAt <= now &&
            nextTickAt > status.ExpiresAt;

        EffectPlan[] effects = status.PeriodicEffectIds
            .Select(effectId => _periodicEffects.Prepare(
                effectId,
                status.CreateEffectContext(tickAt)))
            .ToArray();

        StatusTransition transition = expireWithThisTick
            ? StatusTransition.Expire(StatusRemoveReason.Expired)
            : StatusTransition.Schedule(nextTickAt);

        // 이 plan 하나에 Shield/HP 같은 resource 변경, 다음 tick 또는
        // Status 제거, DamageCommitted/StatusTicked/StatusExpired outbox가 모두 들어간다.
        CommitPlan plan = _statusPlanner.BuildTickPlan(
            status,
            effects,
            transition);

        CommitReceipt receipt = _runtimeCommitter.Commit(plan); // due tick당 정확히 한 번
        if (receipt.Status != CommitStatus.Committed)
            return; // state와 outbox 모두 그대로다.

        if (expireWithThisTick)
            return;

        status = _statusSnapshots.Read(status.InstanceId);
        processedTicks++;
    }

    bool catchUpLimited = IsTickDue(status, now);
    if (catchUpLimited)
        _trace.RecordCatchUpLimit(status.InstanceId, processedTicks);

    if (catchUpLimited && status.ExpiresAt > now)
        return; // active: defer overdue ticks to the next advance

    if (status.ExpiresAt <= now)
        CommitRemovalPlan(
            status,
            catchUpLimited
                ? StatusRemoveReason.CatchUpLimited
                : StatusRemoveReason.Expired);
}

private static bool IsTickDue(StatusSnapshot status, long now) =>
    status.IsActive &&
    status.NextTickAt <= now &&
    status.NextTickAt <= status.ExpiresAt;

private void CommitRemovalPlan(
    StatusSnapshot status,
    StatusRemoveReason reason)
{
    CommitPlan plan = _statusPlanner.BuildRemovalPlan(status, reason);
    _runtimeCommitter.Commit(plan); // instance·Modifier·예약·outbox를 한 번에 확정
}

실행 C# 정화 선택 정책

ReferenceStatusCleansePolicy는 저장소가 제공한 candidate snapshot에서 제거할 instance ID만 선택한다. 후보와 질의 태그는 모두 TagSet이므로 대소문자 보정이나 부분 문자열 비교가 없고, 잘못된 태그는 정책 실행 전에 계약 경계에서 거부된다.

단계실행 규칙빈 집합 의미
1 · any 필터RequiredAnyTags 중 하나 이상을 가진 후보만 유지조건 없음
2 · all 필터RequiredAllTags를 모두 가진 후보만 유지조건 없음
3 · exclude 필터ExcludedTags 중 하나라도 가진 후보는 제외제외 없음
4 · 전체 정렬적격 후보 전체를 priority 내림차순, 동률이면 canonical InstanceId 오름차순으로 정렬항상 같은 total order
5 · 상한정렬이 끝난 뒤 앞에서 MaxRemovals개 선택MaxRemovals는 1 이상

필터가 모두 비어 있으면 “모든 상태”를 명시적으로 뜻한다. 적격 후보가 있으면 Selected와 전체 적격 수·선택 ID를 반환하고, 없으면 예외나 가짜 성공 대신 NoMatch와 빈 목록을 반환한다. 입력 후보 순서가 달라도 결과가 같으며, Take(MaxRemovals)를 저장소 열거 순서에 먼저 적용하는 편향을 막는다.

선택과 제거는 다른 단계다.

이 정책은 어떤 instance를 제거할지만 계산한다. 실제 정화는 선택한 ID의 version을 다시 확인하고 StatusInstance, 해당 SourceRef의 Modifier, tick 예약, 구독, outbox를 같은 commit으로 정리해야 한다. protected 같은 태그를 excluded filter로 빼는 것은 실행되지만, 1회 보호 charge 소비는 이 참조의 실행 범위가 아니다.

이해도 확인

Status가 소유하는 동일성·수명·중첩·선택·정리와 Stat·Effect·Combat에 위임할 실행을 구분해 본다.

  1. 01 · 책임 경계
    Burn tick이 HP를 직접 감소시키고 Rage가 attack_power를 직접 30% 증가시켜도 될까?
    정답과 해설

    안 된다. 목표 product 구조에서 Burn은 periodic DamageEffect를 Combat으로 넘긴다. 현재 JavaScript fixture는 Status가 HP를 직접 쓰지 않고 공용 resolveDamageAgainstTarget와 commit plan을 호출하며, C#에는 아직 tick executor가 없다. Rage는 추적 가능한 StatModifier로 연결해야 한다. Status가 소유하는 것은 수명·중첩·예약과 정리다.

  2. 02 · 경계 상황
    Burn의 다음 tick과 만료가 같은 simulation tick에 도달하면 현재 계약은 어떤 순서로 처리할까?
    정답과 해설

    그 tick이 MaxCatchUpTicks 안에서 처리되면 마지막 tick의 피해와 상태 만료, outbox를 같은 원자적 commit으로 확정한다. 이미 만료된 backlog가 상한을 넘으면 남은 tick을 CatchUpLimited로 버리므로 advance 호출 cadence가 다르면 총 tick 수도 달라질 수 있다.

  3. 03 · 시스템 연결
    Fireball의 Burn을 정화할 때 SourceRef와 StatusInstanceId는 각각 어디에 사용될까?
    정답과 해설

    SkillExecution SourceRef는 어떤 스킬 실행이 상태를 부여했는지 추적한다. 상태가 제공하는 Modifier는 SourceRef.Status(definitionId, statusInstanceId)를 사용하며, 정화 시 StatusInstanceId로 Modifier·예약·구독을 함께 정리한다.

  4. 04 · 재적용 진리표
    기존 4 stack Poison에 3 stack을 AddStacksCapped(max 5)로 재적용하면 결과에 무엇을 남겨야 할까?
    정답과 해설

    결과 stack은 5, 실제 증가량은 1, StackCapReached는 true다. 기존 StatusInstanceId와 기존 SourceRef를 보존하고 선택한 duration 정책만 별도로 적용한다. 요청량 3을 그대로 “추가됨”으로 기록하면 후속 이벤트와 UI가 거짓이 된다.

  5. 05 · 결정적 정화
    정화가 최대 1개만 제거한다면 저장소에서 처음 발견한 적격 상태를 바로 선택해도 될까?
    정답과 해설

    안 된다. 모든 후보에 any/all/exclude 필터를 적용한 뒤 적격 집합 전체를 priority 내림차순, InstanceId 오름차순으로 정렬하고 마지막에 MaxRemovals를 적용해야 저장소 열거 순서와 무관한 replay 결과를 얻는다.

  6. 06 · 수치 경계
    base duration 3 ticks에 50% 저항을 적용하면 몇 tick이며, 100%를 면역 대신 써도 될까?
    정답과 해설

    decimal 중간값 1.5를 AwayFromZero로 한 번 반올림해 2 ticks다. 이 참조의 저항 범위는 최대 90%이고 최소 1 tick을 보장한다. 완전 면역은 duration 0이 아니라 별도 Immune 결과로 표현해야 적용 거부와 즉시 만료를 혼동하지 않는다.

설계 점검

  • StatusDefinition과 StatusInstance가 분리되어 있다.
  • StackPolicy, DurationPolicy, TickPolicy가 상태별로 명시되어 있고 재적용 진리표가 실행 C# 정책과 일치한다.
  • 지속시간 저항은 BPS 범위, decimal 중간값, AwayFromZero 단일 반올림, 최소 1 tick, overflow 실패를 함께 고정한다.
  • Status가 제공하는 Modifier는 SourceRef.Status(StatusDefinitionId, StatusInstanceId)를 가지며 tick 예약도 같은 StatusInstanceId로 정리할 수 있다.
  • 상태 제거 시 Modifier, tick 예약, 이벤트 구독이 함께 정리된다.
  • DoT/HoT/aura는 직접 값을 수정하지 않고 periodic Effect로 실행된다.
  • 각 due tick은 피해·다음 tick 또는 만료·outbox를 포함한 하나의 plan으로 정확히 한 번 commit된다.
  • 면역과 정화는 canonical TagSet과 명시적 Immune/NoMatch 결과를 사용하며, 정화 상한은 전체 정렬 뒤 적용한다.
  • Burn과 Rage 예제가 Stat, Effect, Combat과 연결된다.
