Status System

Status System은 목표형 StatusDefinition·StatusInstance 설계, C# compact 요청·결과 계약, JavaScript Burn 실행 참조를 구분해 버프, 디버프, DoT, HoT, CC의 지속시간·중첩·tick·해제·면역을 설명한다. 상태는 값을 직접 바꾸지 않고 Modifier를 제공하거나 주기 Effect를 요청한다.

Status의 역할

Effect는 “Burn을 부여해라”라고 요청할 수 있다. 하지만 Burn이 이미 있을 때 중첩할지, 지속시간을 갱신할지, 주기 피해를 언제 계산할지, 제거될 때 어떤 Modifier를 정리할지는 Status System이 담당한다. 적용 조건도 hit, any_damage, hp_damage 중 하나로 명시해야 한다.

핵심 계약. Status는 스탯 값을 직접 수정하지 않는다. 상태를 누가 부여했는지는 SourceRef 로 남기고, 상태가 제공하는 StatModifier와 tick 예약은 SourceRef.Status(StatusDefinitionId, StatusInstanceId) 로 등록한다. 제거할 때는 SourceRef.StatusInstanceId로 안전하게 정리한다. 주기 피해가 필요하면 Status tick에서 Effect를 실행한다.

개념적 정의와 C# compact 요청·결과

아래 StatusDefinition은 production 데이터 모델을 설명하는 Conceptual extension이다. 현재 C# 실행 참조가 실제 제공하는 범위는 ApplyStatusRequest, compact StatusResult, IStatusService와 별도의 StatusCatchUpPolicy다. 대상별 StatusInstance 저장소와 service 구현은 포함하지 않는다.

public sealed class StatusDefinition
{
    public EntityId StatusId { get; }
    public int DurationTicks { get; }
    public int TickInterval { get; }
    public int MaxCatchUpTicks { get; }
    public int MaxStacks { get; }
    public IReadOnlyList<EntityId> PeriodicEffectIds { get; }

    public StatusDefinition(
        EntityId statusId,
        int durationTicks,
        int tickInterval,
        int maxCatchUpTicks,
        int maxStacks,
        EntityId[] periodicEffectIds)
    {
        if (durationTicks < 0)
            throw new ArgumentOutOfRangeException(nameof(durationTicks));
        if (maxStacks < 1)
            throw new ArgumentOutOfRangeException(nameof(maxStacks));
        if (maxCatchUpTicks < 1)
            throw new ArgumentOutOfRangeException(nameof(maxCatchUpTicks));
        if (periodicEffectIds == null)
            throw new ArgumentNullException(nameof(periodicEffectIds));
        if (periodicEffectIds.Length > 0 && tickInterval <= 0)
            throw new ArgumentOutOfRangeException(nameof(tickInterval));

        StatusId = statusId;
        DurationTicks = durationTicks;
        TickInterval = tickInterval;
        MaxCatchUpTicks = maxCatchUpTicks;
        MaxStacks = maxStacks;
        PeriodicEffectIds = Array.AsReadOnly((EntityId[])periodicEffectIds.Clone());
    }
}

public sealed class ApplyStatusRequest
{
    public EntityId StatusId { get; }
    public EntityId TargetId { get; }
    public SourceRef Source { get; }
    public int StackDelta { get; }

    public ApplyStatusRequest(
        EntityId statusId,
        EntityId targetId,
        SourceRef source,
        int stackDelta)
    {
        if (stackDelta == 0)
            throw new ArgumentOutOfRangeException(
                nameof(stackDelta),
                "A status application must change the stack count.");

        StatusId = statusId;
        TargetId = targetId;
        Source = source;
        StackDelta = stackDelta;
    }
}

public sealed class StatusResult
{
    private StatusResult(
        bool succeeded,
        EntityId? statusInstanceId,
        StatusRemoveReason? removeReason,
        StatusFailureReason? failureReason)
    {
        Succeeded = succeeded;
        StatusInstanceId = statusInstanceId;
        RemoveReason = removeReason;
        FailureReason = failureReason;
    }

    public bool Succeeded { get; }
    public EntityId? StatusInstanceId { get; }
    public StatusRemoveReason? RemoveReason { get; }
    public StatusFailureReason? FailureReason { get; }

    public static StatusResult Applied(EntityId instanceId) =>
        new(true, instanceId, null, null);

    public static StatusResult Removed(
        EntityId instanceId,
        StatusRemoveReason reason) =>
        new(true, instanceId, reason, null);

    public static StatusResult Failed(StatusFailureReason reason) =>
        new(false, null, null, reason);
}

public enum StatusFailureReason
{
    NotFound,
    Immune,
    StackPolicyRejected
}

public interface IStatusService
{
    StatusResult Apply(ApplyStatusRequest request);
    StatusResult Remove(EntityId instanceId, StatusRemoveReason reason);
    void AdvanceTo(long simulationTick);
}

StatusResult는 공개 생성자 대신 Applied·Removed·Failed factory만 제공한다. 성공 적용은 instance와 no removal/failure reason, 성공 제거는 instance와 removal reason, 실패는 failure reason과 no instance라는 유효 조합만 생성한다.

개념 확장 경계. 아래 StatusInstance 생명주기, 면역·중첩·refresh·Modifier 조율과 풍부한 StatusResult는 production 설계 모델이다. JavaScript fixture는 단일 Burn instance·tick·expire만 실행하며, C#은 요청·결과 계약과 catch-up 정책 계산만 검증한다.

상태 생명주기

  1. ApplyStatusEffect가 StatusSystem에 적용 요청을 보낸다.
  2. 면역, 태그, 대상 생존 여부를 검사한다.
  3. 기존 StatusInstance가 있으면 StackPolicy와 DurationPolicy를 적용한다.
  4. 새 상태라면 StatusInstance를 만들고 Modifier를 등록한다.
  5. TickPolicy가 있으면 다음 tick을 예약한다.
  6. 만료, 정화, 사망, 태그 제거 조건이 발생하면 상태를 제거한다.
  7. 제거 시 StatusInstance, StatModifier, 예약 tick, 이벤트 구독을 정리한다.

인스턴스 키와 동일 시각 순서

재적용 시 같은 상태로 볼 기준을 먼저 정한다. 기본 키는 (TargetId, StatusDefinitionId, StackingGroup, SourceScope) 이며 independent stack은 각자 StatusInstanceId를 가진다. expire와 tick이 같은 시각이면 어느 쪽을 먼저 처리할지도 clock domain마다 고정한다. 이 사이트의 fixed simulation tick 계약은 tickAt ≤ expireAt 인 마지막 tick 계산을 먼저 plan에 넣고, 같은 원자적 commit에서 피해·다음 상태·outbox와 함께 expire를 확정한다.

Tick 흐름

Burn 같은 DoT는 tick 시점마다 DamageEffect를 실행해 DamageRequest를 만들고 Combat의 저항·보호막 계산을 그대로 통과한다. 이후 RuntimeCommitter가 HP·Shield, 다음 tick 또는 만료 상태, outbox와 사망 사실을 하나의 plan으로 확정한다. 이때 공격 계수는 적용 시점에 고정하는 snapshot 방식과 매 tick 다시 계산하는 dynamic 방식 중 하나를 선택해야 한다.

방식 설명 장점 주의점
Snapshot 상태 적용 시점의 공격자 스탯과 계수를 저장한다. 재현성이 좋고 계산이 단순하다. 버프가 사라져도 DoT 피해가 유지될 수 있다.
Dynamic Recalculation tick마다 현재 스탯과 상태를 다시 읽는다. 현재 전투 상황을 잘 반영한다. 계산량과 디버그 복잡도가 증가한다.
Hybrid 기본 계수는 snapshot, 대상 저항은 dynamic으로 처리한다. 밸런스 의도를 세밀하게 반영한다. 정책 문서화가 필수다.
Snapshot 주의. Caster 객체 참조를 보관하지 말고 필요한 계수·태그·DefinitionVersion을 불변 스냅샷으로 저장한다. 대상 저항을 dynamic으로 읽는다면 그 정책도 trace에 남긴다.
실행 참조의 시간 단조성.

복원한 active Status의 appliedTick은 state clock보다 미래일 수 없다. 새 status.add는 commit 시각에 적용되고 첫 nextTickAt은 그보다 뒤여야 한다. status.patch는 기존 nextTickAt보다 큰 값으로만 진행한다. 과거 시각으로 되감아 overdue tick을 중복 실행하는 입력은 commit 전에 거부된다. 재적용 refresh·merge는 이 최소 실행 참조의 범위 밖이므로 같은 instance ID를 add해 덮어쓰지 않는다.

개념적 C# Tick 조율(비실행 의사코드)

아래 코드는 production 조율 순서를 설명하는 Conceptual extension이다. 현재 C# 실행 참조는 StatusCatchUpPolicy.Evaluate의 실행·지연·제거 결정을 검증하지만 StatusSnapshot·EffectPlan·StatusTransition·BuildTickPlan·StatusService는 구현하지 않는다. JavaScript fixture는 마지막 tick과 만료가 같을 때 tick을 먼저 계산하고 같은 원자적 commit에서 만료를 확정하며, 한 advance의 MaxCatchUpTicks와 CatchUpLimited 정책을 실행 검증한다.

Catch-up의 호출 주기 의존성. 이 정책은 한 번의 AdvanceTo마다 상한을 적용하고, 이미 만료된 backlog가 남으면 그 tick을 버리고 제거한다. tick 2·4·6, 만료 6, 상한 2인 상태는 AdvanceTo(6) 한 번이면 tick 6을 잃지만 AdvanceTo(4) 뒤 AdvanceTo(6)을 호출하면 세 번 모두 처리된다. 따라서 frame 지연과 무관한 tick 횟수 불변식은 없다. replay가 같은 결과를 요구하면 advance 호출 cadence를 보존하거나 별도의 backlog 정책을 정의해야 한다.
private void AdvanceInstance(StatusSnapshot status, long now)
{
    int processedTicks = 0;

    while (IsTickDue(status, now) &&
           processedTicks < status.MaxCatchUpTicks)
    {
        long tickAt = status.NextTickAt;
        long nextTickAt = checked(tickAt + status.TickInterval);
        bool expireWithThisTick =
            status.ExpiresAt <= now &&
            nextTickAt > status.ExpiresAt;

        EffectPlan[] effects = status.PeriodicEffectIds
            .Select(effectId => _periodicEffects.Prepare(
                effectId,
                status.CreateEffectContext(tickAt)))
            .ToArray();

        StatusTransition transition = expireWithThisTick
            ? StatusTransition.Expire(StatusRemoveReason.Expired)
            : StatusTransition.Schedule(nextTickAt);

        // 이 plan 하나에 Shield/HP 같은 resource 변경, 다음 tick 또는
        // Status 제거, DamageCommitted/StatusTicked/StatusExpired outbox가 모두 들어간다.
        CommitPlan plan = _statusPlanner.BuildTickPlan(
            status,
            effects,
            transition);

        CommitReceipt receipt = _runtimeCommitter.Commit(plan); // due tick당 정확히 한 번
        if (receipt.Status != CommitStatus.Committed)
            return; // state와 outbox 모두 그대로다.

        if (expireWithThisTick)
            return;

        status = _statusSnapshots.Read(status.InstanceId);
        processedTicks++;
    }

    bool catchUpLimited = IsTickDue(status, now);
    if (catchUpLimited)
        _trace.RecordCatchUpLimit(status.InstanceId, processedTicks);

    if (catchUpLimited && status.ExpiresAt > now)
        return; // active: defer overdue ticks to the next advance

    if (status.ExpiresAt <= now)
        CommitRemovalPlan(
            status,
            catchUpLimited
                ? StatusRemoveReason.CatchUpLimited
                : StatusRemoveReason.Expired);
}

private static bool IsTickDue(StatusSnapshot status, long now) =>
    status.IsActive &&
    status.NextTickAt <= now &&
    status.NextTickAt <= status.ExpiresAt;

private void CommitRemovalPlan(
    StatusSnapshot status,
    StatusRemoveReason reason)
{
    CommitPlan plan = _statusPlanner.BuildRemovalPlan(status, reason);
    _runtimeCommitter.Commit(plan); // instance·Modifier·예약·outbox를 한 번에 확정
}

안티패턴

안티패턴 문제 대안
Status가 attackPower 값을 직접 더함 중첩, 제거, 저장, 디버그가 어려워진다. StatModifier.Source에 SourceRef.Status(definitionId, statusInstanceId)를 넣고 StatusInstanceId로 제거한다.
Tick이 HP를 직접 깎음 Combat, Effect, Log, Event 흐름이 우회된다. PeriodicEffect에서 DamageEffect를 실행한다.
재적용 규칙이 코드에 흩어짐 상태별 중첩 버그가 반복된다. StackPolicy와 DurationPolicy를 데이터로 명시한다.
면역과 정화를 문자열 if문으로 처리 태그가 늘어날수록 분기가 폭발한다. TagSet과 DispelPolicy로 처리한다.

    Burn tick이 HP를 직접 감소시키고 Rage가 attack_power를 직접 30% 증가시켜도 될까?
    정답과 해설

    안 된다. Burn은 periodic DamageEffect를 실행해 Combat으로 넘기고 Rage는 추적 가능한 StatModifier를 등록한다. Status는 수명·중첩·예약과 정리를 소유한다.

  2. 02 · 경계 상황
    Burn의 다음 tick과 만료가 같은 simulation tick에 도달하면 현재 계약은 어떤 순서로 처리할까?
    정답과 해설

    그 tick이 MaxCatchUpTicks 안에서 처리되면 마지막 tick의 피해와 상태 만료, outbox를 같은 원자적 commit으로 확정한다. 이미 만료된 backlog가 상한을 넘으면 남은 tick을 CatchUpLimited로 버리므로 advance 호출 cadence가 다르면 총 tick 수도 달라질 수 있다.

  3. 03 · 시스템 연결
    Fireball의 Burn을 정화할 때 SourceRef와 StatusInstanceId는 각각 어디에 사용될까?
    정답과 해설

    SkillExecution SourceRef는 어떤 스킬 실행이 상태를 부여했는지 추적한다. 상태가 제공하는 Modifier는 SourceRef.Status(definitionId, statusInstanceId)를 사용하며, 정화 시 StatusInstanceId로 Modifier·예약·구독을 함께 정리한다.

설계 점검

  • StatusDefinition과 StatusInstance가 분리되어 있다.
  • StackPolicy, DurationPolicy, TickPolicy가 상태별로 명시되어 있다.
  • Status가 제공하는 Modifier는 SourceRef.Status(StatusDefinitionId, StatusInstanceId)를 가지며 tick 예약도 같은 StatusInstanceId로 정리할 수 있다.
  • 상태 제거 시 Modifier, tick 예약, 이벤트 구독이 함께 정리된다.
  • DoT/HoT/aura는 직접 값을 수정하지 않고 periodic Effect로 실행된다.
  • 각 due tick은 피해·다음 tick 또는 만료·outbox를 포함한 하나의 plan으로 정확히 한 번 commit된다.
  • 면역, 저항, 정화, 해제 정책이 TagSet 기반으로 설명된다.
  • Burn과 Rage 예제가 Stat, Effect, Combat과 연결된다.
