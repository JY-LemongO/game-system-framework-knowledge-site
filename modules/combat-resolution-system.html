Resolution Layer

Combat Resolution System

Combat Resolution System은 DamageRequest를 불변 계산 결과로 만드는 계층이다. compact CombatResolver와 별도로, ReferenceCombatPolicy가 명중·치명타 경계, percent→flat 방어 관통, 쌍곡선 방어식, signed 저항, 안정적인 barrier 흡수, 단일 반올림 경계를 실제 C#로 검증한다. formula registry와 게임별 상태 조회 adapter는 확장 지점이다.

예제 언어 · C# DamageRequest HitOutcome Critical Armor Resistance Shield
Learning goals

학습 목표

C# 입력을 단계별로 계산해 불변 결과로 만들고, 확률 경계·관통 순서·완화식·반올림·피해 보존식과 상태 commit 경계를 설명한다.

Combat의 역할

DamageEffect는 “대상에게 fire damage를 주어라”라는 요청을 만들 수 있지만 최종 피해량을 직접 계산하지 않는다. compact CombatResolver는 사전 판정된 Outcome·Critical을 받는다. 학습용 전체 수치 참조인 ReferenceCombatPolicy는 0..9,999 roll과 BPS chance로 Hit/Critical을 판정하고, 방어·관통·저항·barrier·HP·overkill까지 순수 함수로 계산한다. 실제 프로젝트에서는 Stat/Status snapshot을 이 입력으로 바꾸는 adapter를 별도로 둔다.

핵심 계약. Effect는 DamageRequest 를 만들고 Combat은 불변 DamageResult 를 계산한다. 이 시점의 DamageCalculated는 내부 계산 결과일 뿐 외부 반응을 시작하지 않는다. 현재 C# 실행 참조에서는 RuntimeCommitter 가 HP/Shield mutation과 outbox를 원자적으로 확정한다. 제품이 단독 피해용 DamageCommitService adapter를 분리할 수는 있지만 현재 저장소의 실제 타입은 아니다. Fireball처럼 비용·쿨다운과 함께 묶인 실행도 같은 RuntimeCommitter에 하나의 plan으로 전달한다. 성공한 commit 뒤에만 DamageCommitted 도메인 이벤트를 발행한다. AfterDamage는 UI와 로그 같은 관찰용 후처리 단계다.
책임 표의 범위. 아래 ‘담당한다’ 열은 제품 Combat 계층이 소유할 수 있는 전체 책임이다. 현재 ReferenceCombatPolicy가 직접 실행하는 범위는 chance/roll 명중·치명타, 관통·armor·저항, barrier·HP·overkill 계산이다. Blocked·Immune·Rejected는 구별된 입력 outcome으로 보존하지만 block 조건, 무적 상태, 회피 프레임 자체를 조회하지는 않는다.

핵심 객체

객체 책임
DamageRequest AttackerId, DefenderId, Source, DamageType, BaseValue, Tags, Seed를 담는 C# 입력 요청이다.
CombatContext compact 실행 참조에서는 ScalingStatValue, 사전 판정된 Outcome·Critical, 유효 ResistanceBps, 가용 Shield·HP를 담는다. 전체 수치 참조는 별도 CombatPipelineInput에서 chance/roll, armor/penetration, signed resistance, barrier snapshot을 명시한다.
DamagePipeline 제품 구조에서 검증·명중 판정, 공식, 치명타, 방어·저항, 보호막, clamp·반올림의 실행 순서를 고정하는 조율자다. 작은 실행 참조에서는 같은 순서를 ReferenceCombatPolicy.Resolve 한 함수에 모아 계산 규칙을 읽고 테스트하기 쉽게 했다.

DamageFormula 기본 피해, 스탯 계수, 공격자 보정으로 치명타 전 formula subtotal을 계산한다. 치명타, 방어, 저항, 보호막은 뒤 단계에서 처리한다.

C# 요청·결과 계약

Resolve는 입력과 문맥만 읽고 DamageResult를 반환한다. HP와 보호막 변경은 별도의 Commit 서비스가 한 번만 수행한다.

Executable · abridged. 아래 블록은 실행 참조의 핵심 필드와 불변식만 남긴 교육용 발췌다. 생략된 ID·enum 검증과 canonicalization을 포함한 규범 계약은 Contracts/Combat.cs, 계산 구현은 Systems/CombatResolver.cs를 기준으로 한다.
public sealed class DamageRequest
{
    public EntityId AttackerId { get; }
    public EntityId DefenderId { get; }
    public SourceRef Source { get; }
    public EntityId FormulaId { get; }
    public string DamageType { get; }
    public int BaseValue { get; }
    public int CoefficientBps { get; }
    public TagSet Tags { get; }
    public uint Seed { get; }

    public DamageRequest(
        EntityId attackerId,
        EntityId defenderId,
        SourceRef source,
        EntityId formulaId,
        string damageType,
        int baseValue,
        int coefficientBps,
        IEnumerable<string> tags,
        uint seed)
    {
        var canonicalDamageType = new Tag(damageType);
        if (baseValue < 0)
            throw new ArgumentOutOfRangeException(nameof(baseValue));
        if (coefficientBps is < 0 or > 100_000)
            throw new ArgumentOutOfRangeException(nameof(coefficientBps));

        var canonicalTags = new TagSet(
            tags ?? throw new ArgumentNullException(nameof(tags)));

        AttackerId = attackerId;
        DefenderId = defenderId;
        Source = source;
        FormulaId = formulaId;
        DamageType = canonicalDamageType.Value;
        BaseValue = baseValue;
        CoefficientBps = coefficientBps;
        Tags = canonicalTags;
        Seed = seed;
    }
}

public enum HitOutcome
{
    Hit,
    Miss,
    Blocked,
    Immune,
    Rejected
}

public sealed class CombatContext
{
    public decimal ScalingStatValue { get; }
    public HitOutcome Outcome { get; }
    public bool Critical { get; }
    public int CriticalMultiplierBps { get; }
    public int ResistanceBps { get; }
    public int AvailableShield { get; }
    public int AvailableTargetHp { get; }

    public CombatContext(
        decimal scalingStatValue,
        HitOutcome outcome,
        bool critical,
        int criticalMultiplierBps,
        int resistanceBps,
        int availableShield,
        int availableTargetHp)
    {
        if (scalingStatValue < 0m)
            throw new ArgumentOutOfRangeException(nameof(scalingStatValue));
        if (criticalMultiplierBps is < 10_000 or > 100_000)
            throw new ArgumentOutOfRangeException(nameof(criticalMultiplierBps));
        if (resistanceBps is < 0 or > 10_000)
            throw new ArgumentOutOfRangeException(nameof(resistanceBps));
        if (availableShield < 0)
            throw new ArgumentOutOfRangeException(nameof(availableShield));
        if (availableTargetHp < 0)
            throw new ArgumentOutOfRangeException(nameof(availableTargetHp));

        ScalingStatValue = scalingStatValue;
        Outcome = outcome;
        Critical = critical;
        CriticalMultiplierBps = criticalMultiplierBps;
        ResistanceBps = resistanceBps;
        AvailableShield = availableShield;
        AvailableTargetHp = availableTargetHp;
    }
}

public sealed class DamageResult
{
    public HitOutcome Outcome { get; }
    public bool Critical { get; }
    public int RawDamage { get; }
    public int ResolvedDamage { get; }
    public int ShieldAbsorbed { get; }
    public int FinalHpDamage { get; }
    public int Overkill { get; }

    public DamageResult(
        HitOutcome outcome,
        bool critical,
        int rawDamage,
        int resolvedDamage,
        int shieldAbsorbed,
        int finalHpDamage,
        int overkill)
    {
        if (rawDamage < 0 || resolvedDamage < 0 ||
            shieldAbsorbed < 0 || finalHpDamage < 0 || overkill < 0)
            throw new ArgumentOutOfRangeException(nameof(rawDamage));

        if ((long)shieldAbsorbed + finalHpDamage + overkill != resolvedDamage)
            throw new ArgumentException(
                "Resolved damage must equal shield absorption, final HP damage, and overkill.");

        if (outcome != HitOutcome.Hit &&
            (critical || rawDamage != 0 || resolvedDamage != 0 ||
             shieldAbsorbed != 0 || finalHpDamage != 0 || overkill != 0))
            throw new ArgumentException(
                "Every non-Hit outcome must carry zero damage and no critical flag.");

        Outcome = outcome;
        Critical = critical;
        RawDamage = rawDamage;
        ResolvedDamage = resolvedDamage;
        ShieldAbsorbed = shieldAbsorbed;
        FinalHpDamage = finalHpDamage;
        Overkill = overkill;
    }
}

public interface ICombatResolver
{
    DamageResult Resolve(DamageRequest request, CombatContext context);
}

RawDamage는 공식 subtotal에 치명타 배율까지 적용한 뒤, 방어·저항 같은 완화를 적용하기 전의 값이다. CombatContext의 AvailableTargetHp까지 읽어 피해를 분배하므로 ResolvedDamage = ShieldAbsorbed + FinalHpDamage + Overkill이 항상 성립한다. DamageResult에는 이벤트나 디버그 문자열을 넣지 않는다. 현재 실행 참조에서는 상위 RuntimeCommitter가 Shield·HP 변경과 outbox 기록을 한 transaction으로 확정하고, 성공한 CommitReceipt의 outbox에서만 DamageCommitted를 전달한다.

두 실행 경로의 수치 범위. compact CombatResolver는 armor 단계 없이 ResistanceBps = 0..10,000만 받으며, decimal formula·critical subtotal을 저항 계산까지 유지한 뒤 최종 ResolvedDamage에서 한 번 정수화한다. 정수 RawDamage는 관찰용 projection일 뿐 다음 계산의 입력으로 되먹이지 않는다. 전체 ReferenceCombatPolicy도 최종 피해 직전까지 decimal을 유지하지만, percent→flat armor 관통과 ResistanceBps = -5,000..9,000의 signed 저항 정책을 추가로 실행한다.

피해 계산 파이프라인

실행 규칙과 제품 분해의 경계. 아래 단계 순서와 수치 정책은 ReferenceCombatPolicy에서 실행된다. HitResolver·DefenseResolver처럼 각 단계를 DI 가능한 객체로 분리하는 구조는 규모가 커졌을 때의 제품 확장 예시이며, 그 클래스들이 현재 구현되어 있다고 해석하면 안 된다.
  1. DamageRequest를 수신하고 대상이 피해를 받을 수 있는지 확인한다.
  2. HitChanceBps는 raw Accuracy가 아니라 상위 adapter가 accuracy·evasion의 제품 공식을 계산하고 0..10,000으로 clamp한 resolved chance다. ForcedOutcome이 없을 때 chance가 c, 0..9,999 roll이 r이면 r < c가 Hit다. 따라서 chance 0은 절대 성공하지 않고 10,000은 항상 성공하며, roll이 chance와 같으면 실패한다. Blocked·Immune·Rejected는 Miss와 합치지 않고 독립 Outcome으로 보존한다.
  3. DamageFormula가 기본 피해와 공격자 보정을 계산해 formula subtotal을 만든다.
  4. CriticalResolver가 formula subtotal에 치명타 배율을 적용한다. 이 단계의 결과가 완화 전 RawDamage다.
  5. percent 방어 관통을 적용한 뒤 flat 관통을 빼고 0으로 clamp한다. 유효 방어력은 쌍곡선 완화식에, signed 저항은 BPS 배율에 넣는다.
  6. barrier는 Priority ASC → BarrierId Ordinal ASC 순서로 흡수한다. 입력 collection 순서에 의존하지 않는다.
  7. decimal 중간값을 유지하고 barrier에 넘기기 직전에 MidpointRounding.AwayFromZero로 한 번만 정수화한다. 가용 HP만큼만 FinalHpDamage로 clamp한 뒤 초과분을 Overkill로 분리한다.
  8. 실행 참조는 불변 CombatPipelineResult를 만든다. 제품 adapter는 이 결과로 CombatLog·DebugTrace를 만들 수 있고, JavaScript Runtime Lab은 실제 damage_calculated trace를 남긴다. 어느 경우에도 resolve 단계가 상태 변경 이벤트를 발행하지 않는다.

개념 확장 C# 파이프라인 조율

아래는 현재 실행 참조 코드가 아니라 전체 resolver를 추가할 때의 구조 예시다. 각 단계가 새 DamageDraft를 반환하면 단계별 테스트와 trace를 단순화할 수 있다.

public DamageResult Resolve(DamageRequest request, CombatContext context)
{
    DamageDraft draft = DamageDraft.From(request);

    draft = _hit.Apply(draft, context);
    if (draft.Outcome != HitOutcome.Hit)
        return draft.ToZeroDamageResult();

    draft = _formula.Apply(draft, context);
    draft = _critical.Apply(draft, context); // RawDamage 확정
    draft = _defense.Apply(draft, context);
    draft = _resistance.Apply(draft, context);
    draft = _shield.Apply(draft, context);

    return draft.ToResult();
}

실행 C# 수치 정책

ReferenceCombatPolicy.cs와 CombatPolicyVerification.cs는 아래 식과 정상·경계·실패 사례를 함께 제공한다. 이 방어식은 모든 게임에 통용되는 자연 법칙이 아니라 이 프로젝트가 채택한 versioned reference policy다. 상수나 순서를 바꾸면 formula version과 golden fixture도 함께 올려야 한다.

formula = base + scalingStat × coefficientBps / 10,000
critical = formula × criticalMultiplierBps / 10,000
effectiveArmor = max(0,
    armor × (10,000 - percentPenetrationBps) / 10,000 - flatPenetration)
armorMultiplier = armorConstant / (armorConstant + effectiveArmor)
resistanceMultiplier = (10,000 - resistanceBps) / 10,000
resolved = RoundAwayFromZero(critical × armorMultiplier × resistanceMultiplier)

Armorarmor ≥ 0, percent 관통 후 flat 관통단계 교환 시 결과가 달라지므로 순서를 계약으로 고정한다.
Resistance-5,000..9,000 BPS음수는 vulnerability, 상한 9,000은 피해가 0으로 사라지는 극단을 막는다.
BarrierPriority, Ordinal ID 안정 정렬동일 입력 permutation에서도 어느 barrier가 소모되는지 재현된다.
Minimum damage별도 최소 피해 없음완화 뒤 값이 0.5 미만이면 0이 될 수 있다. 최소 1 정책이 필요하면 반올림 뒤 명시적 floor를 추가하고 formula version을 올린다.
ConservationResolved = BarrierAbsorbed + HpDamage + Overkill누락·중복 차감을 자동 테스트로 검출한다.
Numeric lanedecimal 중간값, 마지막 정수 경계 한 번단계마다 반올림해 생기는 누적 오차를 피하고 midpoint 정책을 명시한다.

데이터 모델

공식과 런타임 결과를 분리하면 장르별 전투 공식을 교체하기 쉽다. 액션 RPG는 명중과 회피 프레임을 중시할 수 있고, 턴제 RPG는 속성 상성과 상태 태그를 더 강하게 반영할 수 있다.

Fireball 계산 예시

아래 예시는 Runtime Contract Lab과 같은 Fireball fixture를 사용한다. 명중과 치명타 결과를 고정해 계산 순서를 보여 주며, 모든 비율은 10,000 BPS와 signed midpoint away-from-zero 반올림 정책을 사용한다.

항목 값
base damage 24
spell_power 120
skill coefficient 1.2
critical multiplier 1.5
target fire_resistance 20%
shield 40

계산 순서 

decimal formulaDamage = 24m + 120m * 12_000 / 10_000m; // 168
decimal rawDamage = formulaDamage * 15_000 / 10_000m;   // 252
decimal afterResistance = rawDamage * 8_000 / 10_000m;  // 201.6
int resolvedDamage = decimal.ToInt32(decimal.Round(
    afterResistance, 0, MidpointRounding.AwayFromZero)); // 202
int shieldAbsorbed = Math.Min(40, resolvedDamage);
int postShieldDamage = resolvedDamage - shieldAbsorbed;
int finalHpDamage = Math.Min(500, postShieldDamage);
int overkill = postShieldDamage - finalHpDamage;

{
  "damageResult": {
    "outcome": "Hit",
    "critical": true,
    "rawDamage": 252,
    "resolvedDamage": 202,
    "shieldAbsorbed": 40,
    "finalHpDamage": 162,
    "overkill": 0
  },
  "commitReceipt": {
    "status": "Committed",
    "outbox": ["DamageCommitted"]
  }
}

위 JSON의 damageResult는 실제 C# compact 결과의 일곱 필드만 표시한다. DamageType은 요청에, ResistanceBps는 CombatContext와 trace에 남고 결과 carrier에 중복 저장하지 않는다.

설계 변형 포인트

공식 교체형 Combat

개념 확장에서 DamagePipeline 순서를 유지하되 DamageFormulaDefinition을 바꾸면 장르별 공식 차이를 흡수할 수 있다. 현재 실행 참조는 FormulaId를 조회하지 않는 고정 단일 계수식이다.

무적과 회피 프레임

무적은 Stat이 아니라 CombatContext 또는 Status tag로 판정하는 편이 추적하기 쉽다.

보호막 우선순위

Barrier, guard point, absorb, HP 피해 순서를 명시해야 로그와 UI 표시가 일관된다.

결정론적 랜덤

compact CombatResolver는 Seed를 소비하지 않고 Outcome·Critical을 fixture로 받는다. ReferenceCombatPolicy는 확률 함수 자체와 RNG를 분리해 명시적 0..9,999 roll을 입력받으므로 경계를 직접 테스트할 수 있다. 제품 RandomStream은 replay key로 roll을 만든 뒤 이 정책에 넘기며, 전체 리플레이에는 입력 스냅샷, Definition·공식·수치 버전, 대상 정렬과 반올림 정책도 함께 고정해야 한다.

안티패턴

안티패턴 문제 대안
DamageEffect 안에 모든 피해 공식 작성 Effect가 게임별 전투 공식에 강하게 결합된다. DamageRequest를 만들고 CombatResolver에 위임한다.
Combat이 장비 인벤토리를 직접 조회 장비 교체와 피해 공식이 서로 묶인다. 장비는 StatModifier나 TriggeredEffect를 공급하고 Combat은 StatSystem을 읽는다.
저항과 보호막 순서가 문서화되지 않음 같은 상황에서 서로 다른 결과가 나온다. DamagePipeline 순서를 고정하고 테스트한다.
계산 로그가 없음 밸런스 디버깅과 QA가 어려워진다. DamageResult와 연결된 DebugTrace에 FormulaSteps와 StatReads를 남긴다.
Comprehension check

이해도 확인

Combat이 계산할 판정과 실제 상태 변경의 commit 경계를 나누고, 후속 반응이 시작될 시점을 확인한다.

  1. 01 · 책임 경계
    Combat이 장착 장비를 직접 조회하거나 공격력 Modifier를 등록해도 될까?
    정답과 해설

    안 된다. 장비는 StatModifier나 TriggeredEffect를 공급하고, Combat은 Stat의 최종값과 읽기 전용 Context만 사용해 피해를 판정한다.

  2. 02 · 경계 상황
    공격이 block되면 반드시 Miss나 피해 0으로 처리해야 할까?
    정답과 해설

    아니다. 현재 실행 C#은 Blocked의 피해를 0으로 계산하지만 Outcome은 Blocked로 남긴다. 다른 장르 정책은 피해 감소, guard 자원 소모, 별도 성공 이벤트를 계산할 수 있으며, 어느 경우에도 Blocked를 Miss나 bool Hit 하나로 합치지 않는다.

  3. 03 · 시스템 연결
    DamageEffect 요청 뒤 흡혈이나 Burn 같은 후속 반응은 언제 시작할 수 있을까?
    정답과 해설

    Combat이 불변 DamageResult를 계산하고 상태 저장소가 Shield와 HP를 원자적으로 commit한 뒤여야 한다. commit 성공 후 DamageCommitted가 발행된 시점부터 ReactionQueue가 후속 반응을 실행한다.

  4. 04 · 수치 정책
    armor 100, percent 관통 20%, flat 관통 30일 때 유효 armor는 56일까 50일까?
    정답과 해설

    50이다. 채택 정책은 percent를 먼저 적용해 80을 만든 뒤 flat 30을 빼는 순서다. flat을 먼저 빼고 percent를 적용한 56은 다른 정책이며, 순서를 바꾸려면 formula version을 바꿔야 한다.

설계 점검

  • DamageRequest와 DamageResult가 분리되어 있다.
  • Hit, Formula, Critical, Defense, Resistance, Shield 단계가 독립 테스트 가능하다.
  • 방어/저항/보호막 적용 순서가 문서화되어 있다.
  • Combat은 스탯을 읽지만 Modifier를 직접 등록하지 않는다.
  • DamageResult는 HitOutcome, Critical, RawDamage, ResolvedDamage, ShieldAbsorbed, FinalHpDamage, Overkill을 담으며, RawDamage는 치명타 이후·완화 이전 값이고 피해 보존식을 만족한다. 거부 상세는 CombatLog, 계산 과정은 DebugTrace, commit 이벤트는 Runtime CommitReceipt와 outbox로 분리된다.
  • Fireball 같은 실제 예제에서 계산 근거를 재현할 수 있다.

근거와 이식 주의

  • C# midpoint 동작은 Microsoft의 Math.Round 문서에 정의된 MidpointRounding.AwayFromZero를 명시적으로 사용한다. 기본 overload의 반올림 mode를 암묵적으로 가정하지 않는다.
  • 동률 barrier ID는 문화권과 무관한 StringComparison.Ordinal 의미로 비교한다.
  • Unity adapter가 float, Burst, Job System용 표현을 사용하더라도 domain 입력과 golden 결과를 이 C# 참조에 대조한다. decimal을 지원하지 않는 실행 경로에서는 같은 BPS·반올림 fixture를 만족하는 fixed-point adapter가 필요하다.
