Resolution Layer

Combat Resolution System

Combat Resolution System은 C# DamageRequest를 명중 판정, 공식, 치명타, 방어, 저항, 보호막, clamp·반올림 정책에 통과시켜 불변 DamageResult로 만드는 계산 계층이다. 계산과 상태 변경을 분리해 피해가 왜 그 숫자가 됐는지 추적한다.

예제 언어 · C# DamageRequest HitOutcome Critical Armor Resistance Shield
Learning goals

학습 목표

C# DamageRequest를 단계별로 계산해 DamageResult로 만들고, 순수 계산과 상태 변경의 경계를 설명한다.

Prerequisites

선수 개념

Stat 조회, Effect handoff, SourceRef와 이벤트 추적 개념을 먼저 이해한다.

Combat의 역할

DamageEffect는 “대상에게 fire damage를 주어라”라는 요청을 만들 수 있지만, 최종 피해량을 직접 계산하지 않는다. Combat은 이 요청을 받아 대상이 맞았는지, 치명타가 터졌는지, 방어력과 저항이 얼마나 줄였는지, 보호막이 얼마를 흡수했는지를 계산한다.

핵심 계약. Effect는 DamageRequest 를 만들고 Combat은 불변 DamageResult 를 계산한다. 이 시점의 DamageCalculated는 내부 계산 결과일 뿐 외부 반응을 시작하지 않는다. 실제 HP/Shield 변경은 단 하나의 DamageCommitService 가 단독 피해를 원자적으로 적용한다. Fireball처럼 비용·쿨다운과 함께 묶인 실행은 상위 RuntimeCommitter가 같은 Damage mutation을 포함해 한 번에 적용한다. 두 경우 모두 성공 후에만 DamageCommitted 도메인 이벤트를 발행한다. AfterDamage는 UI와 로그 같은 관찰용 후처리 단계다.

핵심 객체

객체 책임
DamageRequest AttackerId, DefenderId, Source, DamageType, BaseValue, Tags, Seed를 담는 C# 입력 요청이다.
CombatContext 거리, 방향, 지형, 상태 태그, 난이도, RandomStream 등 전투 상황을 담는다.
DamagePipeline 검증·명중 판정, 공식, 치명타, 방어·저항, 보호막, clamp·반올림의 실행 순서를 고정하는 조율자다.
HitResolver 명중, 회피, 블록, 무적, 회피 프레임을 판단한다.
CriticalResolver 공식 subtotal에 적용할 치명타 여부와 배율을 판정해, 완화 전 피해인 RawDamage를 만든다.
DamageFormula 기본 피해, 스탯 계수, 공격자 보정으로 치명타 전 formula subtotal을 계산한다. 치명타, 방어, 저항, 보호막은 뒤 단계에서 처리한다.
DefenseResolver armor, defense, guard, flat reduction, percent reduction을 처리한다.
ResistanceResolver 속성 저항, 관통, 취약, 저항 상한을 처리한다.
ShieldResolver 보호막, barrier, absorb, damage redirect를 처리한다.
DamageResult HitOutcome, 치명타 이후·완화 이전의 RawDamage, 최종 HP 피해, 흡수량, overkill을 담는 불변 계산 결과다. 거부 상세는 CombatLog, 계산 근거는 DebugTrace, commit 이벤트는 AppliedDamage와 outbox에 둔다.
CombatLogBuilder DamageResult의 계산 근거를 사람이 읽을 수 있는 로그와 DebugTrace 단계로 변환한다.

C# 요청·결과 계약

Resolve는 입력과 문맥만 읽고 DamageResult를 반환한다. HP와 보호막 변경은 별도의 Commit 서비스가 한 번만 수행한다.

public sealed class DamageRequest
{
    public EntityId AttackerId { get; }
    public EntityId DefenderId { get; }
    public SourceRef Source { get; }
    public EntityId FormulaId { get; }
    public string DamageType { get; }
    public int BaseValue { get; }
    public int CoefficientBps { get; }
    public IReadOnlyList<string> Tags { get; }
    public uint Seed { get; }

    public DamageRequest(
        EntityId attackerId,
        EntityId defenderId,
        SourceRef source,
        EntityId formulaId,
        string damageType,
        int baseValue,
        int coefficientBps,
        string[] tags,
        uint seed)
    {
        if (tags == null)
            throw new ArgumentNullException(nameof(tags));

        AttackerId = attackerId;
        DefenderId = defenderId;
        Source = source;
        FormulaId = formulaId;
        DamageType = damageType;
        BaseValue = baseValue;
        CoefficientBps = coefficientBps;
        Tags = Array.AsReadOnly((string[])tags.Clone());
        Seed = seed;
    }
}

public enum HitOutcome
{
    Hit,
    Miss,
    Blocked,
    Immune,
    Rejected
}

public sealed class CombatContext
{
    public decimal ScalingStatValue { get; }
    public HitOutcome Outcome { get; }
    public bool Critical { get; }
    public int CriticalMultiplierBps { get; }
    public int ResistanceBps { get; }
    public int AvailableShield { get; }
    public int AvailableTargetHp { get; }

    public CombatContext(
        decimal scalingStatValue,
        HitOutcome outcome,
        bool critical,
        int criticalMultiplierBps,
        int resistanceBps,
        int availableShield,
        int availableTargetHp)
    {
        ScalingStatValue = scalingStatValue;
        Outcome = outcome;
        Critical = critical;
        CriticalMultiplierBps = criticalMultiplierBps;
        ResistanceBps = resistanceBps;
        AvailableShield = availableShield;
        AvailableTargetHp = availableTargetHp;
    }
}

public sealed class DamageResult
{
    public HitOutcome Outcome { get; }
    public bool Critical { get; }
    public int RawDamage { get; }
    public int ResolvedDamage { get; }
    public int ShieldAbsorbed { get; }
    public int FinalHpDamage { get; }
    public int Overkill { get; }

    public DamageResult(
        HitOutcome outcome,
        bool critical,
        int rawDamage,
        int resolvedDamage,
        int shieldAbsorbed,
        int finalHpDamage,
        int overkill)
    {
        if (shieldAbsorbed + finalHpDamage + overkill != resolvedDamage)
            throw new ArgumentException("Resolved damage must be conserved.");

        Outcome = outcome;
        Critical = critical;
        RawDamage = rawDamage;
        ResolvedDamage = resolvedDamage;
        ShieldAbsorbed = shieldAbsorbed;
        FinalHpDamage = finalHpDamage;
        Overkill = overkill;
    }
}

public interface ICombatResolver
{
    DamageResult Resolve(DamageRequest request, CombatContext context);
}

public interface IDamageCommitService
{
    AppliedDamage Commit(EntityId defenderId, DamageResult result);
}

RawDamage는 공식 subtotal에 치명타 배율까지 적용한 뒤, 방어·저항 같은 완화를 적용하기 전의 값이다. CombatContext의 AvailableTargetHp까지 읽어 피해를 분배하므로 ResolvedDamage = ShieldAbsorbed + FinalHpDamage + Overkill이 항상 성립한다. DamageResult에는 이벤트나 디버그 문자열을 넣지 않는다. IDamageCommitService 또는 상위 RuntimeCommitter가 Shield·HP 변경과 outbox 기록을 한 transaction으로 확정하고, 그 성공 결과인 AppliedDamage에서만 DamageCommitted를 발행한다.

피해 계산 파이프라인

  1. DamageRequest를 수신하고 대상이 피해를 받을 수 있는지 확인한다.
  2. HitResolver가 Hit, Miss, Blocked, Immune, Rejected 중 하나의 HitOutcome을 판정한다. 이 사이트의 기본 C# 정책에서는 Hit만 계산을 계속하고 나머지는 모든 피해 필드가 0인 결과로 끝난다.
  3. DamageFormula가 기본 피해와 공격자 보정을 계산해 formula subtotal을 만든다.
  4. CriticalResolver가 formula subtotal에 치명타 배율을 적용한다. 이 단계의 결과가 완화 전 RawDamage다.
  5. DefenseResolver와 ResistanceResolver가 방어, 저항, 관통, 취약을 처리한다.
  6. ShieldResolver가 보호막, 흡수, 피해 전환을 적용한다.
  7. 각 수치 경계에 문서화된 반올림 정책을 적용하고, 최종 HP 피해와 overkill을 가용 HP 범위로 clamp한다.
  8. DamageResult와 CombatLog를 만들고 DamageCalculated trace를 남긴다. 상태를 바꾸는 이벤트는 아직 발행하지 않는다.

C# 파이프라인 조율

각 단계는 같은 DamageDraft를 직접 공유하지 않고 새 값을 반환해 단계별 테스트와 trace를 단순하게 만든다.

public DamageResult Resolve(DamageRequest request, CombatContext context)
{
    DamageDraft draft = DamageDraft.From(request);

    draft = _hit.Apply(draft, context);
    if (draft.Outcome != HitOutcome.Hit)
        return draft.ToZeroDamageResult();

    draft = _formula.Apply(draft, context);
    draft = _critical.Apply(draft, context); // RawDamage 확정
    draft = _defense.Apply(draft, context);
    draft = _resistance.Apply(draft, context);
    draft = _shield.Apply(draft, context);

    return draft.ToResult();
}

판정·계산·적용 단계

단계 책임 대표 결과
Eligibility 대상이 존재하고 피해를 받을 수 있는지, 무적/면역인지 확인 Rejected / Immune
Resolution 명중·회피, formula subtotal, 치명타, block/guard, 방어/저항 계산 DamageResult (HitOutcome 포함)
Commit 계산된 Shield와 HP 변경을 한 번만 적용하고 death 사실을 확정 AppliedDamage
Aftermath 로그·UI 관찰 이벤트와 결정론적 ReactionQueue 실행 DamageCommitted, EntityDefeated, AfterDamage
Block은 곧 Miss가 아니다. 이 사이트의 실행 C#은 Blocked를 피해 0으로 계산하지만 Miss와는 다른 HitOutcome으로 보존한다. 장르에 따라 block은 피해 감소·guard resource 소모·별도 성공 이벤트일 수 있으므로, 정책을 바꾸더라도 bool Hit에 합치지 않는다.

데이터 모델

공식과 런타임 결과를 분리하면 장르별 전투 공식을 교체하기 쉽다. 액션 RPG는 명중과 회피 프레임을 중시할 수 있고, 턴제 RPG는 속성 상성과 상태 태그를 더 강하게 반영할 수 있다.

Fireball 계산 예시

아래 예시는 Runtime Contract Lab과 같은 Fireball fixture를 사용한다. 명중과 치명타 결과를 고정해 계산 순서를 보여 주며, 모든 비율은 10,000 BPS와 half-up 반올림 정책을 사용한다.

항목 값
base damage 24
spell_power 120
skill coefficient 1.2
critical multiplier 1.5
target fire_resistance 20%
shield 40

계산 순서 

int formulaDamage = 24 + MultiplyBpsHalfUp(120, 12_000); // 168
int rawDamage = MultiplyBpsHalfUp(formulaDamage, 15_000); // 252, 치명타 후·완화 전
int resolvedDamage = MultiplyBpsHalfUp(rawDamage, 8_000); // 202, 저항 완화 후
int shieldAbsorbed = Math.Min(40, resolvedDamage);
int postShieldDamage = resolvedDamage - shieldAbsorbed;
int finalHpDamage = Math.Min(500, postShieldDamage);
int overkill = postShieldDamage - finalHpDamage;

static int MultiplyBpsHalfUp(int value, int bps)
{
    return (int)(((long)value * bps + 5_000) / 10_000);
}

설계 변형 포인트

공식 교체형 Combat

DamagePipeline의 순서는 유지하되 DamageFormulaDefinition을 바꾸면 장르별 공식 차이를 흡수할 수 있다.

무적과 회피 프레임

무적은 Stat이 아니라 CombatContext 또는 Status tag로 판정하는 편이 추적하기 쉽다.

보호막 우선순위

Barrier, guard point, absorb, HP 피해 순서를 명시해야 로그와 UI 표시가 일관된다.

결정론적 랜덤

DamageRequest의 Seed와 RandomStream은 확률 분기를 재현하는 입력이다. 전체 리플레이에는 입력 스냅샷, Definition·공식·수치 버전, 대상 정렬과 반올림 정책도 함께 고정해야 한다.

안티패턴

안티패턴 문제 대안
DamageEffect 안에 모든 피해 공식 작성 Effect가 게임별 전투 공식에 강하게 결합된다. DamageRequest를 만들고 CombatResolver에 위임한다.
Combat이 장비 인벤토리를 직접 조회 장비 교체와 피해 공식이 서로 묶인다. 장비는 StatModifier나 TriggeredEffect를 공급하고 Combat은 StatSystem을 읽는다.
저항과 보호막 순서가 문서화되지 않음 같은 상황에서 서로 다른 결과가 나온다. DamagePipeline 순서를 고정하고 테스트한다.
계산 로그가 없음 밸런스 디버깅과 QA가 어려워진다. DamageResult와 연결된 DebugTrace에 FormulaSteps와 StatReads를 남긴다.
Comprehension check

이해도 확인

Combat이 계산할 판정과 실제 상태 변경의 commit 경계를 나누고, 후속 반응이 시작될 시점을 확인한다.

  1. 01 · 책임 경계
    Combat이 장착 장비를 직접 조회하거나 공격력 Modifier를 등록해도 될까?
    정답과 해설

    안 된다. 장비는 StatModifier나 TriggeredEffect를 공급하고, Combat은 Stat의 최종값과 읽기 전용 Context만 사용해 피해를 판정한다.

  2. 02 · 경계 상황
    공격이 block되면 반드시 Miss나 피해 0으로 처리해야 할까?
    정답과 해설

    아니다. 현재 실행 C#은 Blocked의 피해를 0으로 계산하지만 Outcome은 Blocked로 남긴다. 다른 장르 정책은 피해 감소, guard 자원 소모, 별도 성공 이벤트를 계산할 수 있으며, 어느 경우에도 Blocked를 Miss나 bool Hit 하나로 합치지 않는다.

  3. 03 · 시스템 연결
    DamageEffect 요청 뒤 흡혈이나 Burn 같은 후속 반응은 언제 시작할 수 있을까?
    정답과 해설

    Combat이 불변 DamageResult를 계산하고 상태 저장소가 Shield와 HP를 원자적으로 commit한 뒤여야 한다. commit 성공 후 DamageCommitted가 발행된 시점부터 ReactionQueue가 후속 반응을 실행한다.

설계 점검

  • DamageRequest와 DamageResult가 분리되어 있다.
  • Hit, Formula, Critical, Defense, Resistance, Shield 단계가 독립 테스트 가능하다.
  • 방어/저항/보호막 적용 순서가 문서화되어 있다.
  • Combat은 스탯을 읽지만 Modifier를 직접 등록하지 않는다.
  • DamageResult는 HitOutcome, Critical, RawDamage, ResolvedDamage, ShieldAbsorbed, FinalHpDamage, Overkill을 담으며, RawDamage는 치명타 이후·완화 이전 값이고 피해 보존식을 만족한다. 거부 상세는 CombatLog, 계산 과정은 DebugTrace, commit 이벤트는 AppliedDamage와 outbox로 분리된다.
  • Fireball 같은 실제 예제에서 계산 근거를 재현할 수 있다.
