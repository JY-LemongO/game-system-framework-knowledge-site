Skill / Action System

Skill System은 플레이어 입력 또는 AI 의도를 비용, 쿨다운, 타겟팅, 액션 타임라인, EffectBundle 실행으로 변환하는 행동 계층이다. Effect가 “결과”라면 Skill은 그 결과를 언제, 어떤 조건으로 실행할지 결정하는 “행동 패키지”다.

SkillDefinition Cost Cooldown Targeting Timeline EffectBundle
Learning goals

학습 목표

비용·쿨다운·타겟팅·타임라인을 조율하되 피해 계산과 상태 유지를 소유하지 않는 Skill 경계를 설계한다.

Prerequisites

선수 개념

EffectBundle과 EffectContext, Entity·Source 식별 규칙을 먼저 읽는다.

Skill System의 책임 경계

Skill은 HP를 직접 깎지 않는다. Skill은 입력과 행동 조건을 검증하고, 적절한 타이밍에 EffectBundle 실행을 요청한다. DamageEffect가 필요하면 Effect System이 CombatResolver에 DamageRequest를 전달한다.

담당한다 담당하지 않는다
스킬 정의와 Entity별 SkillRuntime 상태 관리 최종 피해 공식 계산
마나, 스태미나, 탄약 같은 비용 검증과 예약·확정 시점 스탯 최종값 계산 파이프라인
개별 쿨다운, 글로벌 쿨다운, 쿨다운 그룹, 차지 관리 상태 지속시간과 tick 관리
대상 지정 방식, 사거리, 시야, 무기 조건 검증 AI가 어떤 스킬을 선택할지 판단
시전, 선딜, 판정 프레임, 후딜, 캔슬 구간 진행 UI 버튼 렌더링과 입력 장치 표시
EffectBundle 실행 요청과 SkillResult/Event 발행 장비 옵션 생성과 보상 테이블 선택

Skill과 Effect의 차이

Skill은 “사용할 수 있는 행동”이고 Effect는 “적용되는 결과”다. 이 예제의 Fireball은 비용, 쿨다운, 단일 대상, release marker를 가지며 다른 스킬은 시전·차지·채널링 타임라인을 가질 수 있다. Fireball의 DamageEffect와 ApplyStatusEffect(effect.apply-burn)는 실제 대상에게 피해와 상태를 요청한다.

주의. 스킬마다 피해 계산 코드를 직접 넣기 시작하면 Effect, Combat, Status, Log 구조가 무너진다. 스킬은 EffectBundle을 실행하는 쪽에 머물러야 한다.

기본 흐름

Input / AI → SkillRequest → Validate → Timeline → EffectBundle → SkillResult

핵심 객체

객체 책임
SkillDefinition 스킬의 정적 데이터. 비용, 쿨다운, 타겟팅, 태그, 타임라인, EffectBundle 참조를 가진다.
SkillRuntime 특정 Entity가 가진 스킬의 현재 상태. 쿨다운, 차지, 시전 중 여부, 런타임 오버라이드를 가진다.
SkillExecution 한 번의 cast/channel을 나타내는 실행 인스턴스. executionId, phase, reserved cost, target snapshot, cancellation token을 가진다.
SkillBook Entity가 배운 스킬 목록과 레벨, 해금 상태, 장착 슬롯을 관리한다.
SkillRequest 스킬 사용 요청. caster, skillId, target, aimPosition, inputTime, seed를 포함한다.
SkillValidator 비용, 쿨다운, 상태 이상, 장비 조건, 사거리, 시야, 대상 유효성을 검증한다.
CostPolicy 비용을 언제 지불하고 언제 환불하는지 결정한다.
CooldownPolicy 쿨다운 시작 시점, 그룹 쿨다운, 차지 회복, 쿨다운 감소 적용 방식을 정의한다.
TargetingSpec 자기 자신, 단일 대상, 지점, 방향, 범위 미리보기 같은 입력 규칙을 정의한다.
ActionTimeline windup, cast, release marker, active frame, recovery, cancel window를 정의한다.
SkillExecutor 검증, 비용 예약, 타임라인 진행, EffectBundle 실행, 이벤트 발행을 조율한다.
SkillResult 성공/실패, 중단, 실행된 EffectResult 목록, 디버그 로그를 담는다.

실행 파이프라인

  1. PlayerInput 또는 AI가 SkillRequest를 만든다.
  2. SkillDefinition과 SkillRuntime을 조회한다.
  3. 스킬을 배웠는지, 쿨다운 중인지, 비용이 충분한지, 침묵/기절/무장해제 상태인지 검증한다.
  4. TargetingSpec으로 사거리, 시야, 대상 타입, 지점 유효성을 검사한다.
  5. 비용을 예약하거나 정책에 따라 선지불한다.
  6. ActionTimeline을 시작하고 windup, cast, release marker, recovery를 진행한다.
  7. release marker에 도달하면 EffectContext를 만들고 EffectBundle 실행을 요청한다.
  8. SkillResult를 만든다. 상태 commit이 성공하면 SkillCommitted를 발행하고, 검증 실패는 SkillResult와 trace로 반환한다. 중단이 실제 실행 상태를 바꿨다면 SkillInterrupted를 commit 이후 발행한다.

스킬 생명주기

축 · 상태 설명 전이 marker / commit event
Availability · Ready 사용 가능한 상태다. 입력 큐나 AI 선택을 받을 수 있다. AvailabilityChanged
Execution · Queued 입력은 들어왔지만 검증 또는 선행 행동 완료를 기다린다. SkillQueued trace
Execution · Casting 선딜 또는 시전 유지 구간이다. 캔슬과 중단 규칙이 적용된다. CastStarted
Execution · Hit Frame release marker에 도달해 EffectBundle을 실행한다. ReleaseMarker
Execution · Recovery 후딜 구간이다. 일부 게임에서는 이동/다음 행동 캔슬을 허용한다. RecoveryStarted
Availability · Cooldown 스킬을 다시 사용할 수 없는 상태다. charge 회복도 이 구간에서 진행된다. CooldownCommitted
Execution · Interrupted 기절, 침묵, 피격 등으로 중단된 상태다. 환불과 실패 쿨다운 정책을 적용한다. SkillInterrupted

SkillRuntime과 SkillExecution 분리

SkillRuntime은 배운 레벨, charge, cooldown처럼 오래 사는 availability 상태를 보관하고, SkillExecution은 cast 한 번의 임시 실행 상태를 보관한다. 둘을 합치면 동시 시전, 채널링, 취소, 서버 예측, 투사체가 남아 있는 상황을 안전하게 표현하기 어렵다. Ready/Cooldown과 Casting/Release/Recovery는 서로 다른 상태 축이며 구현에서는 별도 상태 머신으로 둔다.

Commit 직전 재검증. 요청 시점 검증만으로는 충분하지 않다. release marker에서 비용 예약, 대상 유효성, 제어 상태를 다시 확인하고 성공한 execution만 EffectBundle을 commit한다.

비용 정책

비용은 단순히 “사용 시 마나 감소”가 아니다. 시전형 스킬은 중단 가능성이 있으므로 지불 시점과 환불 정책을 명확히 해야 한다. Skill CostPolicy가 예약·확정 규칙을 소유하고, 실제 current resource 변경은 ResourceController가 atomic commit 안에서 수행한다. ModifyResourceEffect는 스킬 자체 비용이 아니라 회복·흡수 같은 게임 결과에 사용한다.

정책 설명 예시
on_request 입력 즉시 비용을 지불한다. 실패 여지가 적은 즉발 행동에 적합하다. 구르기, 즉발 회복
on_cast_start 시전 시작 시 비용을 지불한다. 가장 일반적인 방식이다. Fireball 마나 소모
on_release 판정 프레임에 도달했을 때 지불한다. 선딜 중 중단이 잦은 게임에 적합하다. 강한 차징 공격
reserve_then_commit 시작 시 비용을 예약하고 판정 시 확정한다. UI와 서버 검증에 유리하다. 네트워크 전투
refund_on_interrupt_before_hit hit frame 전 중단이면 일부 또는 전체 환불한다. 캐스팅 중 피격 중단
no_refund_after_release EffectBundle이 실행된 뒤에는 환불하지 않는다. 투사체가 이미 발사된 스킬

쿨다운과 차지

쿨다운은 스킬 하나에만 붙는 값이 아니라 액션 시스템의 리듬을 만드는 규칙이다. 개별 쿨다운, 쿨다운 그룹, 글로벌 쿨다운, 차지 회복을 구분해야 한다.

Individual Cooldown

각 스킬이 독립적으로 재사용 대기 시간을 가진다. 단순하고 대부분의 액션 RPG에 적합하다.

Cooldown Group

같은 그룹의 스킬이 쿨다운을 공유한다. 물약, 회피기, 동일 무기 기술 묶음에 적합하다.

Global Cooldown

하나의 스킬 사용 후 짧은 공통 대기 시간을 둔다. 입력 폭주와 애니메이션 충돌을 막는다.

Charges

여러 번 연속 사용 가능한 스킬에 쓰인다. 각 charge의 회복 타이머와 쿨다운 감소 적용 방식을 정해야 한다.

타겟팅 정책

타겟팅 타입 입력 검증 포인트
self 입력 없음 또는 자기 자신 침묵/기절 여부, 자기 대상 허용 여부
explicit_target 선택된 Entity 거리, 팀, 생존 여부, 시야, 대상 태그
ground_position 월드 좌표 사거리, 지형, 시야, 이동 가능/투사 가능 여부
direction 방향 벡터 입력 방향 유효성, 최소/최대 거리, 벽 충돌
area_preview 범위 미리보기 후 확정 범위 모양, 후보 대상, UI preview와 서버 판정 일치
smart_target 자동 대상 선택 우선순위, 조준 보정, 의도하지 않은 대상 방지

Action Timeline

타임라인은 애니메이션 길이와 같지 않다. 게임 규칙상 판정이 발생하는 marker, 입력을 받을 수 있는 cancel window, 중단 시 정책이 별도로 존재해야 한다. 

{
  "timeline": {
    "castTime": 0.7,
    "phases": [
      { "name": "windup", "from": 0.0, "to": 0.5, "canCancel": true },
      { "name": "release", "at": 0.7, "markers": ["execute_effect_bundle"] },
      { "name": "recovery", "from": 0.7, "to": 1.0, "canCancel": false }
    ]
  }
}

EffectBundle 실행 정책

하나의 스킬은 여러 Effect를 묶는다. 예를 들어 Fireball은 DamageEffect를 commit한 뒤 DamageCommitted reaction으로 ApplyStatusEffect(effect.apply-burn)를 요청한다. 가역 operation의 atomic 범위와 commit 이후 reaction 실패 정책을 명확히 해야 한다.

정책 설명 사용 예
ordered_all_or_nothing 앞 Effect가 실패하면 전체 실패로 취급한다. 자원 소모와 상태 부여가 반드시 함께 성공해야 하는 효과
ordered_partial 순서대로 실행하되 일부 실패를 EffectResult에 기록하고 다음 Effect를 계속 실행한다. 피해는 들어갔지만 Burn은 면역으로 실패하는 경우
parallel_best_effort 대상별·효과별 독립 실행을 허용한다. 광역 버프/디버프
stop_on_first_success 하나만 성공하면 나머지는 실행하지 않는다. 우선순위 기반 자동 대상 효과
rollback_on_failure 중간 실패 시 이미 적용한 효과를 되돌린다. 거래, 제작, 카드 비용 처리처럼 원자성이 중요한 경우

데이터 모델

Fireball 실행 예시

1. SkillRequest

플레이어가 살아 있는 단일 대상을 지정한다. 요청에는 actorId, targetId, skillDefinitionId, requestedTick, rootSeed가 들어간다.

2. Validator

마나 20 이상, 쿨다운 없음, 침묵/기절 아님, 단일 대상이 살아 있는지 확인한다.

3. Cost & Timeline

마나 20을 예약하고 release marker에서 비용과 EffectBundle을 한 transaction으로 commit한다.

4. EffectBundle

DamageEffect가 단일 대상 DamageRequest를 만들고, commit 성공 뒤 ApplyStatusEffect(effect.apply-burn)가 살아 있는 대상에게 Burn 상태를 요청한다.

5. Result

SkillResult는 성공 여부, 실행된 EffectResult, CombatLog, StatusResult, 이벤트 목록을 묶는다.

실패 케이스

실패 케이스 판정 위치 결과
마나 부족 SkillValidator.validateCost SkillResult.failedReason = OutOfResource, 비용/쿨다운 없음
쿨다운 중 SkillValidator.validateCooldown 요청 거부, UI는 남은 쿨다운 표시
침묵 상태 SkillValidator.validateControlState 스킬 사용 불가. 상태 태그로 분기
타겟 사거리 밖 SkillValidator.validateTarget 요청 거부 또는 자동 보정 실패
선딜 중 피격 중단 ActionTimeline interrupt rule 환불 정책 적용, 실패 쿨다운 선택 가능
판정 후 대상 이동 Effect TargetSelector 또는 Combat HitResolver 대상별 실패 결과를 EffectResult에 기록

안티패턴

스킬이 HP를 직접 수정

SkillExecutor에서 대상 HP를 바로 감소시키면 Effect, Combat, Status, Log가 우회된다. Skill은 EffectBundle만 실행한다.

스킬마다 타겟 검색 직접 구현

각 스킬 코드 안에 범위 검색을 넣으면 사거리, 시야, 팀 필터가 중복된다. TargetingSpec과 TargetSelector로 분리한다.

쿨다운을 UI에만 저장

UI는 표시만 해야 한다. 실제 쿨다운과 charge 상태는 SkillRuntime이 소유해야 저장, 서버 검증, AI 사용이 가능하다.

중단 정책 누락

시전 중 중단 시 비용과 쿨다운 처리가 정해져 있지 않으면 버그가 반복된다. 시점별 환불/쿨다운 정책을 명시한다.

Combat / Status로 이어지는 Handoff

Skill은 Damage를 계산하지 않고 Status를 유지하지 않는다. Skill은 ActionTimeline의 marker에서 EffectBundle을 실행할 뿐이다. EffectBundle 내부의 DamageEffect는 Combat Resolution으로, ApplyStatusEffect는 Status System으로 이어진다.

Fireball Skill → EffectBundle → DamageEffect → Combat → ApplyStatusEffect(effect.apply-burn) → Status
Resolution

DamageResult

명중, 치명타, 방어, 저항, 보호막 계산 결과를 CombatLog와 함께 반환한다.

Runtime State

StatusResult

Burn 같은 상태의 적용, 면역, 중첩, 지속시간 갱신, tick 예약 결과를 반환한다.

설계 점검

  • SkillDefinition이 비용, 쿨다운, 타겟팅, 타임라인, EffectBundle을 모두 데이터로 표현한다.
  • SkillRuntime이 Entity별 쿨다운, 차지, 시전 상태를 보관한다.
  • SkillValidator가 상태 이상, 비용, 사거리, 시야, 장비 조건을 검증한다.
  • ActionTimeline의 release marker에서만 EffectBundle이 실행된다.
  • 중단 시 비용 환불과 실패 쿨다운 정책이 정의되어 있다.
  • SkillResult가 EffectResult, 실패 이유, 이벤트, 디버그 로그를 포함한다.
  • Fireball 같은 예제가 Skill → Effect → Combat/Status로 끝까지 추적된다.
