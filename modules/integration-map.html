Architecture Map

Integration Map

Integration Map은 Core, Stat, Effect, Skill, Combat, Status의 현재 C# 실행 계약과 production에서 추가할 개념적 port를 구분해 의존 방향을 정리한다. 각 시스템을 독립적으로 테스트하면서도 Fireball 같은 실행 흐름이 끊기지 않게 만드는 것이 목표다.

Learning goals

학습 목표

여섯 시스템에서 실행 가능한 C# 인터페이스·handoff 타입과 개념적 product port를 구분해 한 흐름으로 읽고, 금지해야 할 역방향 의존성을 찾는다.

Architecture lens

하나의 화살표가 아니라, 세 가지 관점

학습 순서, 실제 실행 흐름, 코드 의존성은 서로 다르다. 관점을 전환해 같은 시스템을 다른 방식으로 읽는다.

목적 개념을 난이도 순서로 익히는 추천 경로다. 호출 방향을 뜻하지 않는다.
01 Core 공통 계약 → 02 Stat 숫자 엔진 → 03 Effect 결과 요청 → 04 Skill 행동 조율 → 05 Combat 피해 해석 → 06 Status 지속 상태
실행 입력은 Skill에서 시작해 Effect 요청으로 바뀌고 Combat과 Status로 분기한 뒤 Commit과 Event로 모인다.
Input / AI → Skill Execution → Effect Plan
↙ Damage Resolve
↘ Status Apply
Stat / Resource Commit → Event + DebugTrace
코드 구현체끼리 얽히지 않고 작은 포트를 호출한다. 화살표는 소유권이 아니라 계약 사용 방향이다.

학습 시스템 의존성 지도

아래 그림은 런타임 협업과 데이터 읽기 방향을 요약한다. 컴파일 의존성, 실행 호출, 이벤트 반응을 하나의 화살표 의미로 섞지 않으며, 구현에서는 각 연결을 작은 Port/API로 분리한다.

시스템별 계약

표 읽기. 현재 C# 실행 계약의 공통 값은 EntityId와 SourceRef다. TagSet·IEventBus·IGameClock·IRandomStream은 production adapter가 추가할 규범적 개념 port이며 현재 C# 프로젝트에는 선언되어 있지 않다.
흐름 전달 데이터 의미
Core → 모든 시스템 실행 C#: EntityId, SourceRef
개념 확장: TagSet, IEventBus, IGameClock, IRandomStream 		앞의 두 값은 현재 실행 참조가 공유하며, 뒤의 네 port는 production에서 같은 책임 경계를 유지하기 위한 목표 언어다.
Stat → Effect / Combat GetValue(EntityId ownerId, EntityId statId, StatContext context) Effect와 Combat은 최종 스탯 값을 읽지만 계산 파이프라인을 소유하지 않는다.
Skill → Effect EffectBundle , EffectContext Skill은 release marker에서 EffectBundle 실행을 요청한다.
Effect → Combat DamageRequest DamageEffect는 최종 피해 공식을 직접 계산하지 않는다.
Effect → Status ApplyStatusRequest 독립된 주 효과인 ApplyStatusEffect는 Status에 ApplyStatusRequest를 직접 전달할 수 있지만, 지속시간과 tick은 직접 관리하지 않는다.
Combat Commit → Domain Event DamageCommitted CombatResolver의 DamageResult는 계산 결과로만 유지한다. 현재 실행 참조의 RuntimeCommitter가 Shield·HP와 outbox를 commit한 뒤 DamageCommitted 사실을 전파한다. 전투별로 분리한 DamageCommitService는 가능한 product adapter 이름일 뿐 현재 타입은 아니다.
Status → Stat StatModifier.Source = SourceRef.Status(StatusDefinitionId, StatusInstanceId) 상태가 제공하는 수치 변경은 Modifier로 등록/제거한다.
Status → Effect PeriodicEffect DoT, HoT, aura tick은 Effect를 다시 실행한다.

C# 시스템 포트

상위 조율자는 구현 클래스를 직접 참조하지 않고, 각 시스템이 소유한 요청·결과 포트와 상태를 바꾸지 않는 planning 포트만 호출한다. IEffectExecutor는 독립 command 실행 경계이고, Skill처럼 비용·쿨다운과 주 Effect를 함께 commit할 때는 IEffectPlanner로 EffectBundlePlan을 먼저 준비한다.

public interface IStatQuery
{
    decimal GetValue(EntityId ownerId, EntityId statId, StatContext context);
}

public interface IEffectExecutor
{
    EffectOperationResult Execute(
        EffectOperation operation,
        EffectContext context);

    EffectBundleResult Execute(
        EffectBundle bundle,
        EffectContext context);
}

public interface IEffectPlanner
{
    EffectBundlePlan Prepare(
        EffectBundle bundle,
        EffectContext context);
}

public interface ISkillRequestValidator
{
    SkillDecision Validate(SkillRequest request);
}

public enum HitOutcome
{
    Hit,
    Miss,
    Blocked,
    Immune,
    Rejected
}

public interface ICombatResolver
{
    DamageResult Resolve(DamageRequest request, CombatContext context);
}

public interface IRuntimeCommitter
{
    CommitReceipt Commit(CommitPlan plan);
}

public interface IStatusService
{
    StatusResult Apply(ApplyStatusRequest request);
    StatusResult Remove(EntityId instanceId, StatusRemoveReason reason);
    void AdvanceTo(long simulationTick);
}

public interface IReactionQueue
{
    void Enqueue(ReactionCommand command);
    int Drain(ReactionBudget budget);
}

Combat 포트의 DamageResult는 HitOutcome Outcome을 사용한다. Hit, Miss, Blocked, Immune, Rejected를 구분하므로 Block을 bool Hit나 Miss에 합치지 않는다. 현재 실행 참조의 CombatContext는 사전 해석된 ScalingStatValue·Outcome·Critical·Resistance와 AvailableTargetHp를 포함하고, 결과는 ResolvedDamage = ShieldAbsorbed + FinalHpDamage + Overkill을 만족한다. Hit/Crit RNG·Defense·Formula registry는 개념 확장이다. Resolve는 상태를 바꾸거나 이벤트를 발행하지 않고 commit 성공 뒤에만 DamageCommitted를 만든다.

수직 슬라이스

SkillRequest → SkillValidator → ActionTimeline → EffectBundle → DamageRequest → DamageResult → CommitReceipt → DamageCommitted → ReactionQueue → StatusResult

Fireball 예제는 이 흐름을 한 번에 확인할 수 있는 대표 수직 슬라이스다. 일반적인 ApplyStatusEffect가 독립된 주 효과라면 Effect가 Status에 곧바로 ApplyStatusRequest를 전달할 수 있다. 그러나 Fireball의 주 EffectBundle에는 DamageEffect만 들어간다. Combat의 피해 결과를 RuntimeCommitter가 비용·쿨다운·Shield·HP·outbox와 함께 확정한 뒤, commit된 DamageCommitted를 ReactionCommand로 변환해 ReactionQueue가 ApplyStatusEffect(effect.apply-burn)를 실행한다.

금지 의존성

금지 구조 문제 대안
Skill이 최종 피해를 직접 계산 Effect / Combat / Log 구조가 무너진다. Skill은 EffectBundle 실행 타이밍만 만든다.
DamageEffect 안에 모든 전투 공식 작성 게임별 공식 교체와 테스트가 어려워진다. DamageRequest를 만들고 CombatResolver에 위임한다.
Status가 Stat 값을 직접 변경 중첩, 제거, 저장, 디버그가 어려워진다. StatModifier.Source에 SourceRef.Status(StatusDefinitionId, StatusInstanceId)를 넣고 StatusInstanceId로 제거한다.
Combat이 장비 인벤토리를 직접 조회 장비 시스템과 전투 공식이 강하게 결합된다. 장비는 Modifier를 공급하고 Combat은 StatSystem을 읽는다.
Status tick이 Skill 쿨다운을 직접 수정 시간, 상태, 스킬 책임이 뒤섞인다. 먼저 변경 사실을 commit하고, 그 committed event를 CausationId와 멱등 키가 있는 ReactionCommand로 변환해 ReactionQueue에 넣는다.

이벤트 흐름

Commit 소유자 범위. 현재 C# 실행 타입은 IRuntimeCommitter/RuntimeCommitter다. 아래 *CommitService 이름들은 product가 시스템별 adapter를 분리할 때 사용할 수 있는 개념적 대안이며 저장소의 실제 타입이 아니다. SkillCommitted·DamageCommitted는 C#과 JavaScript 실행 증거가 있고, 나머지 fact 예시는 해당 production 확장 지점의 이름이다.
이벤트 commit·outbox 소유자 구독 예시
SkillCommitted RuntimeCommitter (실행 C#) · SkillCommitService (개념 대안) UI 로그와 업적은 관찰하고, 장비 triggered effect는 ReactionCommand로 변환
EffectCompleted RuntimeCommitter (실행 C#) · EffectCommitService (개념 대안) DebugTrace와 튜토리얼은 관찰하고, 연쇄 효과는 ReactionQueue 명령으로 변환
DamageCommitted RuntimeCommitter (실행 C#) · DamageCommitService (개념 대안) 피해 숫자 UI와 카메라 흔들림은 관찰하고, 흡혈·반사는 ReactionQueue 명령으로 변환
StatusApplied RuntimeCommitter (실행 C#) · StatusCommitService (개념 대안) 상태 아이콘, 면역 로그, 업적
StatusTicked RuntimeCommitter (실행 C#) · StatusCommitService (개념 대안) DoT 표시와 사운드는 관찰하고, 추가 연쇄 효과는 ReactionCommand로 변환
ModifierChanged RuntimeCommitter (실행 C#) · StatCommitService (개념 대안) 스탯 패널 갱신, 캐시 무효화

각 실행기와 Controller는 event payload를 plan에 준비할 뿐 직접 발행하지 않는다. commit 소유자가 상태와 outbox를 함께 확정하고, transaction 성공 뒤 외부 adapter가 outbox를 dispatch한다.

ReactionQueue 계약

DamageCommitted뿐 아니라 EffectCompleted, StatusApplied, StatusTicked에 상태 변경 subscriber를 즉시 연결하면 실행 순서와 무한 연쇄가 불명확해진다. UI·로그·분석은 이벤트를 관찰하고, 흡혈·반사·연쇄 Effect처럼 상태를 바꾸는 반응은 commit된 이벤트를 새 명령으로 변환해 queue에 넣고 numeric priority 오름차순(작은 값 우선) → stableOrderKey ordinal 오름차순 → reactionId ordinal 오름차순 순서로 처리한다.

한 번의 Drain은 하나의 causation wave 전체를 처리한다. maxReactions와 maxBudget은 1 이상, maxDepth는 0 이상이어야 한다. dispatch 중 새로 enqueue된 반응도 같은 wave의 depth·총 reaction 수·총 budget에 포함된다. queue 생성 제한과 ReactionBudget 중 더 작은 상한을 넘거나 handler가 예외를 내면 오류를 호출자에게 전달하고 아직 dispatch하지 않은 항목은 폐기한다. 따라서 상한 때문에 남은 반응이 다음 Drain에서 뒤늦게 실행되는 일은 없다.

  • EventId / CorrelationId / CausationId를 분리한다.
  • 최대 reactionDepth와 반복 idempotencyKey를 둔다.
  • UI·분석 subscriber는 state-mutating queue와 분리한다.
  • Before/Calculated/Committed/After 이벤트 시점을 이름으로 구분한다.

이해도 확인

요청·계산·commit·reaction이 어느 시스템 경계에서 이어지는지 확인한다.

  1. 01 · 책임 경계
    Status tick의 주기 피해가 필요할 때 Status가 HP를 직접 줄여도 될까?
    정답과 해설

    안 된다. Status는 PeriodicEffect를 실행하고 Effect가 DamageRequest를 만들어 Combat에 전달한다. Combat은 DamageResult를 계산하고 commit 계층이 HP를 변경한다.

  2. 02 · Commit 경계
    CombatResolver가 DamageResult를 만든 즉시 DamageCommitted를 발행해도 될까?
    정답과 해설

    안 된다. CombatResolver는 순수 계산만 한다. 현재 실행 참조에서는 RuntimeCommitter가 Shield·HP와 outbox를 성공적으로 commit한 뒤에만 DamageCommitted를 발행한다. DamageCommitService는 이를 전투별로 분리할 때의 개념적 adapter 이름이다.

  3. 03 · 후속 반응
    EffectCompleted를 받은 연쇄 Effect와 튜토리얼 UI는 같은 방식으로 실행해야 할까?
    정답과 해설

    아니다. 튜토리얼 UI와 로그는 상태를 바꾸지 않는 관찰 subscriber다. 연쇄 Effect는 CausationId·우선순위·멱등 키를 가진 ReactionCommand로 변환해 ReactionQueue에서 순서대로 실행한다.

설계 점검

  • 각 시스템이 소유하는 데이터와 읽기 전용으로 참조하는 데이터가 구분되어 있다.
  • 독립된 주 DamageEffect와 ApplyStatusEffect는 각각 Combat과 Status로 연결된다. Fireball의 Burn처럼 commit 결과에 조건부로 이어지는 ApplyStatusEffect는 DamageCommitted에서 만든 ReactionCommand로 연결된다.
  • Status가 제공하는 수치 변화는 StatModifier로 표현된다.
  • 순수 계산 결과는 DebugTrace로 추적되고, 상태를 바꾼 commit 사실만 EventBus와 outbox로 발행된다.
  • Fireball 같은 수직 슬라이스를 통해 전체 흐름을 재현할 수 있다.
