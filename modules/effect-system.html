Effect System

Effect System은 스킬, 아이템, 상태, 보상이 실제 게임 결과를 만들 때 사용하는 공통 실행 계층이다. Stat System이 숫자를 제공한다면, Effect System은 그 숫자를 피해, 회복, 상태 부여, 자원 변화 같은 결과로 바꾼다.

EffectDefinition TargetSelector EffectExecutor EffectResult
Learning goals

학습 목표

대상 선택과 결과 요청을 Effect로 표현하고, Stat·Combat·Status에 책임을 넘기는 경계를 구분한다.

Prerequisites

선수 개념

Stat 조회와 SourceRef 추적, 불변 요청 객체가 필요한 이유를 먼저 이해한다.

Effect System의 책임 경계

Effect는 “무슨 결과를 적용할 것인가”를 담당한다. 하지만 키 입력, 애니메이션, 명중 공식, 장기 성장까지 모두 처리하면 안 된다.

담당한다 담당하지 않는다
Effect 정의와 실행 플레이어 입력 처리
대상 선택 규칙 실행 공격 애니메이션 타이밍 전체
값 공식 계산과 Stat 조회 게임별 상세 피해 공식 전체
EffectResult와 로그 생성 장비 드랍 경제
이벤트 발행 AI 의사결정

왜 Effect가 중간 계층인가?

스킬, 아이템, 상태, 보상은 겉으로는 다르지만 결국 결과를 적용한다. Fireball은 피해와 화상을 적용하고, Healing Potion은 회복을 적용하며, Rage Buff는 Modifier를 적용한다. 이들을 각각 별도 코드로 만들면 시스템이 흩어진다.

Skill → EffectBundle → EffectExecutor → Target → Result → Event

핵심 객체 상세

EffectDefinition

효과의 정적 데이터다. effectId, type, targetSelector, valueExpression, conditions, tags, timing, ordering policy를 가진다.

EffectContext

실행 시점의 상황이다. caster, source, initial target, skill, hit result, random seed, execution moment, custom variables를 담는다.

TargetSelector

Effect가 적용될 대상을 찾는다. 단일 대상, 원형 범위, 전방 부채꼴, 체인, 자기 자신, 소환 위치 등을 처리한다.

ValueExpression

회복량, 자원 변화량처럼 Effect가 직접 소유하는 값을 계산한다. DamageEffect에서는 계수를 여기서 확정하지 않고 formulaId, baseValue, scaling 입력을 DamageRequest에 담아 Combat에 넘긴다.

EffectOperation

Damage, Heal, ApplyStatus, AddModifier, ModifyResource, Dispel, Spawn 등 실제 적용 동작이다.

EffectResult

대상별 결과, 실패 이유, 생성 이벤트, 디버그 로그를 포함한다. CombatLog와 테스트에 매우 중요하다.

Effect 실행 파이프라인

  1. Effect 요청을 받는다.
  2. EffectDefinition을 로드한다.
  3. EffectContext를 만든다.
  4. 전역 조건과 실행 조건을 검증한다.
  5. TargetSelector로 대상 후보를 수집한다.
  6. 대상별 조건을 다시 검증한다.
  7. 일반 Effect는 ValueExpression을 계산한다. DamageEffect는 계산 입력을 DamageRequest로 정규화하고 Stat 조회와 피해 공식은 Combat에 위임한다.
  8. EffectOperation을 실행한다.
  9. EffectResult와 로그를 만든다.
  10. EventBus에 이벤트를 발행한다.

Effect 타입 카탈로그

초기 MVP에서는 모든 효과를 다 만들려고 하지 말고, 아래 타입부터 시작하면 좋다.

EffectOperation 필요성 Stat 연동
DamageEffect 전투 시스템의 핵심 공격자 스케일링, 속성 보너스, damageType, source tag
HealEffect 회복, 흡혈, 재생 회복력, 회복량 증가/감소
ModifyResourceEffect 마나 회복, 스태미나 감소, 자원 흡수 current/max resource

SpawnEffect 소환, 장판, 투사체 소환체의 StatBlock
MoveEffect 넉백, 끌어당김, 이동 무게, 넉백 저항

ValueExpression과 DamageRequest 예시

Heal이나 ModifyResource처럼 Effect가 결과값을 소유하는 경우에는 ValueExpression을 계산한다. 피해는 같은 표현식을 Effect와 Combat에서 두 번 계산하지 않도록, DamageEffect가 아래 입력을 요청으로 만들고 CombatFormula가 Stat 조회와 계산을 전담한다. 

{
  "id": "effect.fireball-damage",
  "type": "damage",
  "damageType": "fire",
  "target": { "type": "area_circle", "radius": 4.0 },
  "damageRequest": {
    "formulaId": "combat.fire.v3",
    "baseValue": 24,
    "scaling": [
      { "stat": "spell_power", "coefficientBps": 12000 }
    ]
  },
  "conditions": ["target_is_alive", "has_line_of_sight"],
  "tags": ["spell", "fire", "area"]
}

EffectResult와 로그

Effect가 실행되면 단순히 “성공”만 반환하면 안 된다. 대상별 성공/실패, 계산값, 적용된 상태, 생성 이벤트, 디버그 메시지를 남겨야 한다. 

EffectResult effect.fireball-damage
  - sourceId: command.fireball.cast.0001
  - sourceRef: { kind: skill-execution, definitionId: skill.fireball, instanceId: command.fireball.cast.0001 }
  - caster: player_001
  - targets:
  - enemy_001: committed, rawDamage=252, resolvedDamage=202, hpDamage=162
  - enemy_002: immune, reason=damage.immunity.fire
  - crate_003: invalid, reason=not_living_target
- events:
  - DamageCommitted
  - EffectCompleted
- debug:
  - base 24 + spell_power 120 × 1.2 = 168
  - critical 168 × 1.5 = 252
  - resistance 20% → 202, shield 40 → hp 162

실패 처리와 트랜잭션

여러 Effect가 묶인 스킬에서 앞 Effect는 성공하고 뒤 Effect는 실패할 수 있다. 이때 전체를 롤백할 것인지, 성공한 것만 유지할 것인지 정책이 필요하다.

정책 설명 사용 예
BestEffort 가능한 대상에게만 적용 광역 피해
AllOrNothing 하나라도 실패하면 전체 취소 거래, 제작, 자원 교환
OrderedPartial 순서대로 실행하되 실패한 대상이나 Effect를 EffectResult에 기록하고 가능한 다음 단계를 계속한다. 피해는 성공했지만 Burn은 면역으로 실패하는 스킬
RollbackOnFailure 실패 시 이전 적용을 되돌림 고위험 시스템, 네트워크 동기화

실행 계획과 Commit 경계

월드 이동, 투사체 생성, 이벤트 발행처럼 되돌리기 어려운 작업에 일반적인 rollback을 약속하면 구현이 불안정해진다. Effect는 먼저 검증하고 EffectPlan 을 만든 뒤, 가역적인 명령만 한 번에 commit한다. 외부 부작용은 commit 성공 후 발행하며 idempotency key로 중복 적용을 막는다.

Validate → Plan → Commit → Publish

결정론과 랜덤 시드

전투 로그 재현, 리플레이, 시뮬레이션 테스트를 생각하면 EffectContext에 randomSeed를 포함하는 편이 좋다. 그래야 같은 입력으로 같은 결과를 재현할 수 있다.

데이터 모델

정의 데이터는 콘텐츠로 저장하고, 런타임 결과는 로그와 재현 정보로 남긴다. EffectResult를 세이브 데이터처럼 영구 저장할 필요는 보통 없지만, 디버깅과 리플레이에는 중요하다.

생명주기

구현 안티패턴

  • 스킬 코드 안에서 직접 HP를 깎고 상태를 부여하는 구조
  • TargetSelector 없이 스킬마다 대상 검색을 따로 구현하는 구조
  • EffectResult가 없어 실패 원인과 계산 내역을 추적할 수 없는 구조
  • CombatResolver와 EffectOperation을 한 클래스에 섞어 게임별 공식을 교체할 수 없는 구조
  • 랜덤 시드 없이 테스트 재현이 불가능한 구조

Combat / Status 연결

Effect System은 DamageEffect와 ApplyStatusEffect의 다음 책임자를 명확히 가진다. DamageEffect는 최종 피해 공식을 직접 계산하지 않고 DamageRequest 를 만들어 CombatResolver에 전달한다. ApplyStatusEffect는 지속시간과 tick을 직접 관리하지 않고 ApplyStatusRequest 를 만들어 StatusSystem에 전달한다.

Combat

DamageEffect → CombatResolver

명중, 치명타, 방어, 저항, 보호막 계산은 Combat Resolution System에서 DamageResult로 정리한다.

Status

ApplyStatusEffect → StatusSystem

버프, 디버프, DoT, CC의 지속시간, 중첩, tick, 해제 정책은 Status System이 관리한다.

Effect 타입 Effect의 책임 위임 대상
DamageEffect 대상과 SourceRef, damageType, formulaId, baseValue, scaling 입력을 담은 DamageRequest 생성 CombatResolver
ApplyStatusEffect statusId, target, sourceId, durationScale, stackDelta를 담은 적용 요청 생성 StatusSystem
ModifyStatEffect 명시적 일회성 수치 변경 또는 Modifier 등록 요청 생성 StatSystem 또는 ResourceController
SpawnEffect / MoveEffect 게임 월드에 대한 명시적 요청 생성 World/Entity System
Comprehension check

이해도 확인

Effect가 결과를 직접 소유하는 부분과 Combat·Status에 요청으로 넘겨야 하는 부분을 구분해 본다.

  1. 01 · 책임 경계
    DamageEffect가 주문력 계수, 치명타, 저항과 보호막까지 계산해 최종 피해를 만들어도 될까?
    정답과 해설

    안 된다. Effect는 formulaId, baseValue, scaling, 대상과 SourceRef를 DamageRequest에 담고 최종 피해 계산은 Combat에 위임한다.

  2. 02 · 경계 상황
    Fireball 피해는 성공했지만 대상이 Burn 면역일 때 OrderedPartial 정책에는 어떤 결과가 남아야 할까?
    정답과 해설

    이미 commit된 피해는 유지한다. Burn 실패는 대상별 실패 이유와 함께 EffectResult에 기록하고, 정책상 실행 가능한 다음 Effect는 계속 처리한다.

  3. 03 · 시스템 연결
    입력 시점에 유효했던 대상이 release 시점에 이동했다면 Skill과 Effect는 각각 어디까지 다시 판단해야 할까?
    정답과 해설

    Skill의 TargetingSpec은 사용자가 조준할 수 있는 입력을 검증한다. Effect의 TargetSelector는 release 또는 hit 시점의 실제 영향 대상을 다시 해석해 Combat이나 Status 요청으로 넘긴다.

설계 점검

  • EffectDefinition과 EffectRuntime 실행 상태가 분리되어 있다.
  • EffectContext가 source, caster, target, skill, seed를 포함한다.
  • TargetSelector가 공통 모듈로 분리되어 있다.
  • EffectOperation은 Damage, Heal, Resource, Status, Modifier부터 시작한다.
  • EffectResult가 대상별 성공/실패와 로그를 포함한다.
  • EffectService가 이벤트를 발행하지만 UI나 전투 UI를 직접 호출하지 않는다.
