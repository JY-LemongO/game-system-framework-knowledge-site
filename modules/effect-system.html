Effect System

Effect System은 스킬, 아이템, 상태, 보상이 실제 게임 결과를 만들 때 사용하는 공통 실행 계층이다. Stat System이 숫자를 제공한다면, Effect System은 그 숫자를 피해, 회복, 상태 부여, 자원 변화 같은 결과로 바꾼다.

EffectDefinitionTargetSelectorEffectExecutorEffectResult

Effect System의 책임 경계

Effect는 “무슨 결과를 적용할 것인가”를 담당한다. 하지만 키 입력, 애니메이션, 명중 공식, 장기 성장까지 모두 처리하면 안 된다.

담당한다담당하지 않는다
Effect 정의와 실행플레이어 입력 처리
대상 선택 규칙 실행공격 애니메이션 타이밍 전체
값 공식 계산과 Stat 조회게임별 상세 피해 공식 전체
EffectResult와 로그 생성장비 드랍 경제
이벤트 발행AI 의사결정

왜 Effect가 중간 계층인가?

스킬, 아이템, 상태, 보상은 겉으로는 다르지만 결국 결과를 적용한다. Fireball은 피해와 화상을 적용하고, Healing Potion은 회복을 적용하며, Rage Buff는 Modifier를 적용한다. 이들을 각각 별도 코드로 만들면 시스템이 흩어진다.

Skill→EffectBundle→EffectExecutor→Target→Result→Event

핵심 객체 상세

EffectDefinition

효과의 정적 데이터다. effectId, type, targetSelector, valueExpression, conditions, tags, timing, ordering policy를 가진다.

EffectContext

실행 시점의 상황이다. caster, source, initial target, skill, hit result, random seed, execution moment, custom variables를 담는다.

TargetSelector

Effect가 적용될 대상을 찾는다. 단일 대상, 원형 범위, 전방 부채꼴, 체인, 자기 자신, 소환 위치 등을 처리한다.

ValueExpression

base + stat scaling + context variables를 계산한다. 직접 문자열 수식, 데이터 기반 공식, 코드 Formula 객체 중 하나로 구현할 수 있다.

EffectOperation

Damage, Heal, ApplyStatus, AddModifier, ModifyResource, Dispel, Spawn 등 실제 적용 동작이다.

EffectResult

대상별 결과, 실패 이유, 생성 이벤트, 디버그 로그를 포함한다. CombatLog와 테스트에 매우 중요하다.

Effect 실행 파이프라인

  1. Effect 요청을 받는다.
  2. EffectDefinition을 로드한다.
  3. EffectContext를 만든다.
  4. 전역 조건과 실행 조건을 검증한다.
  5. TargetSelector로 대상 후보를 수집한다.
  6. 대상별 조건을 다시 검증한다.
  7. ValueExpression을 계산하며 필요한 Stat을 조회한다.
  8. EffectOperation을 실행한다.
  9. EffectResult와 로그를 만든다.
  10. EventBus에 이벤트를 발행한다.

ValueExpression 예시

Effect의 값은 고정값만으로 끝나지 않는다. 대부분은 base value, 시전자의 스탯, 대상의 스탯, 스킬 태그, 난이도 배율, 랜덤 시드를 조합한다.

{
  "id": "fireball_damage",
  "type": "damage",
  "element": "fire",
  "target": { "type": "area_circle", "radius": 4.0 },
  "value": {
    "base": 40,
    "scaling": [
      { "stat": "spell_power", "ratio": 1.2 },
      { "stat": "fire_damage_bonus", "ratio": 1.0, "mode": "percent" }
    ]
  },
  "conditions": ["target_is_alive", "has_line_of_sight"],
  "tags": ["spell", "fire", "area"]
}

EffectResult와 로그

Effect가 실행되면 단순히 “성공”만 반환하면 안 된다. 대상별 성공/실패, 계산값, 적용된 상태, 생성 이벤트, 디버그 메시지를 남겨야 한다.

EffectResult fireball_damage
- source: skill:fireball
- caster: player_001
- targets:
  - enemy_001: hit, damage=128, fireResistance=0.2
  - enemy_002: immune, reason=fire_immune
  - crate_003: invalid, reason=not_living_target
- events:
  - OnDamageDealt
  - OnEffectApplied
- debug:
  - spell_power 70
  - base 40 + scaling 84 = 124
  - final after bonus/resistance = 128

실패 처리와 트랜잭션

여러 Effect가 묶인 스킬에서 앞 Effect는 성공하고 뒤 Effect는 실패할 수 있다. 이때 전체를 롤백할 것인지, 성공한 것만 유지할 것인지 정책이 필요하다.

정책설명사용 예
BestEffort가능한 대상에게만 적용광역 피해
AllOrNothing하나라도 실패하면 전체 취소거래, 제작, 자원 교환
OrderedPartial순서대로 실행하되 실패한 대상이나 Effect를 EffectResult에 기록하고 가능한 다음 단계를 계속한다.피해는 성공했지만 Burn은 면역으로 실패하는 스킬
RollbackOnFailure실패 시 이전 적용을 되돌림고위험 시스템, 네트워크 동기화

결정론과 랜덤 시드

전투 로그 재현, 리플레이, 시뮬레이션 테스트를 생각하면 EffectContext에 randomSeed를 포함하는 편이 좋다. 그래야 같은 입력으로 같은 결과를 재현할 수 있다.

데이터 모델

정의 데이터는 콘텐츠로 저장하고, 런타임 결과는 로그와 재현 정보로 남긴다. EffectResult를 세이브 데이터처럼 영구 저장할 필요는 보통 없지만, 디버깅과 리플레이에는 중요하다.

생명주기

구현 안티패턴

  • 스킬 코드 안에서 직접 HP를 깎고 상태를 부여하는 구조
  • TargetSelector 없이 스킬마다 대상 검색을 따로 구현하는 구조
  • EffectResult가 없어 실패 원인과 계산 내역을 추적할 수 없는 구조
  • CombatResolver와 EffectOperation을 한 클래스에 섞어 게임별 공식을 교체할 수 없는 구조
  • 랜덤 시드 없이 테스트 재현이 불가능한 구조

완료 체크리스트

  • EffectDefinition과 EffectRuntime 실행 상태가 분리되어 있다.
  • EffectContext가 source, caster, target, skill, seed를 포함한다.
  • TargetSelector가 공통 모듈로 분리되어 있다.
  • EffectOperation은 Damage, Heal, Resource, Status, Modifier부터 시작한다.
  • EffectResult가 대상별 성공/실패와 로그를 포함한다.
  • EffectService가 이벤트를 발행하지만 UI나 전투 UI를 직접 호출하지 않는다.

Combat / Status 연결

Phase 2부터 Effect System은 DamageEffect와 ApplyStatusEffect의 다음 책임자를 명확히 가진다. DamageEffect는 최종 피해 공식을 직접 계산하지 않고 DamageRequest를 만들어 CombatResolver에 전달한다. ApplyStatusEffect는 지속시간과 tick을 직접 관리하지 않고 ApplyStatusRequest를 만들어 StatusSystem에 전달한다.

Combat

DamageEffect → CombatResolver

명중, 치명타, 방어, 저항, 보호막 계산은 Combat Resolution System에서 DamageResult로 정리한다.

Status

ApplyStatusEffect → StatusSystem

버프, 디버프, DoT, CC의 지속시간, 중첩, tick, 해제 정책은 Status System이 관리한다.
