Effect System

Effect System은 스킬, 아이템, 상태, 보상이 실제 게임 결과를 요청하는 공통 실행 경계다. 현재 C# 참조는 target-independent Damage/Status spec, 결정적 target resolution, 명시적 operation outcome과 Effect 전용 CommitPlan fragment 조합을 실행한다. 피해·상태 계산은 각 소유 시스템에 위임하고 Skill·reaction fragment 통합은 확장 경계로 남긴다.

Learning goals

학습 목표

대상 선택과 결과 요청을 C# Effect 타입으로 표현하고, Stat·Combat·Status에 책임을 넘기는 경계를 구분한다.

Prerequisites

선수 개념

Stat 조회와 SourceRef 추적, 불변 요청 객체가 필요한 이유를 먼저 이해한다.

Effect System의 책임 경계

Effect는 “무슨 결과를 적용할 것인가”를 담당한다. 하지만 키 입력, 애니메이션, 명중 공식, 장기 성장까지 모두 처리하면 안 된다.

왜 Effect가 중간 계층인가?

스킬, 아이템, 상태, 보상은 겉으로는 다르지만 결국 결과를 적용한다. Fireball은 피해와 화상을 적용하고, Healing Potion은 회복을 적용하며, Rage Buff는 Modifier를 적용한다. 이들을 각각 별도 코드로 만들면 시스템이 흩어진다.

Skill → Target resolve → Resolved bundle → Commit fragment → CommitPlan → Commit / facts

핵심 객체 상세

증거 범위. EffectDefinition·범용 ValueExpression·Area/Arc/Chain selector는 콘텐츠 로딩 계층의 개념 확장이다. 실행 기준은 Contracts/Effects.cs의 guarded request/bundle과 ReferenceEffectPipeline.cs의 target-independent Damage/ApplyStatus spec, 세 target mode, outcome, Effect fragment composer다.

EffectDefinition

효과의 개념적 정적 데이터다. production 콘텐츠 정의에는 EffectId, Type, TargetSelector, ValueExpression, Conditions, Tags, Timing, OrderingPolicy 같은 필드를 둘 수 있지만 현재 C# 실행 참조에 이 타입은 없다.

EffectContext

실행 시점의 상황이다. 최소 계약은 CasterId, InitialTargetId, Source, RandomSeed를 담고, Effect 종류에 필요한 값은 별도 문맥 타입으로 확장한다.

TargetSelector

Effect가 적용될 대상을 찾는다. 단일 대상, 원형 범위, 전방 부채꼴, 체인, 자기 자신, 소환 위치 등을 처리한다.

ValueExpression

회복량, 자원 변화량처럼 Effect가 직접 소유하는 값을 계산한다. DamageEffect에서는 계수를 여기서 확정하지 않고 FormulaId, BaseValue, CoefficientBps를 DamageRequest에 담아 Combat에 넘긴다.

EffectOperation

Damage, Heal, ApplyStatus, AddModifier, ModifyResource, Dispel, Spawn 등 실제 적용 동작이다.

EffectResult

실행 결과의 공통 기반 타입이다. 실행 참조에서는 단일 operation의 EffectOperationResult와 묶음 commit의 EffectBundleResult로 구체화하며, 상세 계산 근거는 별도 trace에 남긴다.

C# 핵심 계약

Executable · abridged. 아래 블록은 기존 target-bound request 타입의 공개 모양을 축약한 것이다. Valid* helper, Bundle의 null·enum·중복 ID guard 전체는 Contracts/Effects.cs가 기준이다. target-independent spec과 resolver는 다음 절의 별도 실행 reference가 담당한다.

// Executable contract · abridged; helper and collection guard bodies are omitted.
public sealed record EffectContext(
    EntityId CasterId,
    EntityId? InitialTargetId,
    SourceRef Source,
    uint RandomSeed)
{
    private EntityId _casterId =
        ValidEntityId(CasterId, nameof(CasterId));
    private EntityId? _initialTargetId =
        ValidOptionalEntityId(InitialTargetId, nameof(InitialTargetId));
    private SourceRef _source = ValidSource(Source, nameof(Source));

    public EntityId CasterId
    {
        get => _casterId;
        init => _casterId = ValidEntityId(value, nameof(CasterId));
    }

    public EntityId? InitialTargetId
    {
        get => _initialTargetId;
        init => _initialTargetId =
            ValidOptionalEntityId(value, nameof(InitialTargetId));
    }

    public SourceRef Source
    {
        get => _source;
        init => _source = ValidSource(value, nameof(Source));
    }

    // Valid* helper bodies are omitted; they reject default IDs and sources.
}

public abstract record EffectOperation(EntityId OperationId)
{
    private EntityId _operationId =
        ValidOperationId(OperationId, nameof(OperationId));

    public EntityId OperationId
    {
        get => _operationId;
        init => _operationId =
            ValidOperationId(value, nameof(OperationId));
    }
}

public sealed record DamageEffectOperation : EffectOperation
{
    private DamageRequest _request;

    public DamageEffectOperation(EntityId operationId, DamageRequest request)
        : base(operationId)
    {
        _request = request ?? throw new ArgumentNullException(nameof(request));
    }

    public DamageRequest Request
    {
        get => _request;
        init => _request = value ?? throw new ArgumentNullException(nameof(value));
    }
}

public sealed record ApplyStatusEffectOperation : EffectOperation
{
    private ApplyStatusRequest _request;

    public ApplyStatusEffectOperation(EntityId operationId, ApplyStatusRequest request)
        : base(operationId)
    {
        _request = request ?? throw new ArgumentNullException(nameof(request));
    }

    public ApplyStatusRequest Request
    {
        get => _request;
        init => _request = value ?? throw new ArgumentNullException(nameof(value));
    }
}

public abstract record EffectResult(bool Succeeded);

public sealed record EffectOperationResult(
    bool Succeeded,
    EntityId OperationId) : EffectResult(Succeeded)
{
    private EntityId _operationId =
        ValidOperationId(OperationId, nameof(OperationId));

    public EntityId OperationId
    {
        get => _operationId;
        init => _operationId =
            ValidOperationId(value, nameof(OperationId));
    }
}

public sealed record EffectBundleResult : EffectResult
{
    public EffectBundleResult(
        bool committed,
        EntityId bundleId,
        int appliedEffectCount,
        int queuedReactionCount) : base(committed)
    {
        EntityId.ThrowIfInvalid(bundleId, nameof(bundleId));

        if (appliedEffectCount < 0)
            throw new ArgumentOutOfRangeException(nameof(appliedEffectCount));
        if (queuedReactionCount < 0)
            throw new ArgumentOutOfRangeException(nameof(queuedReactionCount));
        if (!committed &&
            (appliedEffectCount != 0 || queuedReactionCount != 0))
            throw new ArgumentException(
                "An uncommitted bundle cannot report applied work.");

        Committed = committed;
        BundleId = bundleId;
        AppliedEffectCount = appliedEffectCount;
        QueuedReactionCount = queuedReactionCount;
    }

    public bool Committed { get; }
    public EntityId BundleId { get; }
    public int AppliedEffectCount { get; }
    public int QueuedReactionCount { get; }
}

public interface IEffectExecutor
{
    EffectOperationResult Execute(
        EffectOperation operation,
        EffectContext context);

    EffectBundleResult Execute(
        EffectBundle bundle,
        EffectContext context);
}

public interface IEffectPlanner
{
    EffectBundlePlan Prepare(
        EffectBundle bundle,
        EffectContext context);
}
Executable 생성 경계. Damage/ApplyStatus operation은 null request를 생성자와 record copy 양쪽에서 거부한다. EffectBundle은 정의되지 않은 EffectExecutionPolicy, 중복 OperationId, ReactionRule.RuleId, ReactionId, IdempotencyKey를 거부하며 EffectBundlePlan도 같은 identity 규칙을 적용한다.

기존 EffectOperation의 DamageRequest·ApplyStatusRequest에는 대상이 이미 있으므로 EffectBundle과 EffectBundlePlan은 target-bound 경계다. resolver 입력으로 재사용하면 이중 resolution이 된다. 새 실행 참조는 대상 없는 DamageEffectSpec/ApplyStatusEffectSpec을 ReferenceEffectSpecification에 담고, resolver가 한 번만 실제 request로 bind한다. 기존 planner port와 새 reference pipeline은 같은 타입인 척 섞지 않는다.

실행 가능한 Effect 파이프라인과 확장 경계

Executable · bounded reference.

ReferenceEffectOperationSpec → ReferenceEffectTargetResolver → target-bound ResolvedEffectOperation → EffectOperationOutcome → CommitPlanFragment → DeterministicEffectCommitPlanComposer가 실제 C#으로 연결된다. resolver가 지원하는 payload는 Damage와 ApplyStatus, target mode는 Self·ExplicitTarget·CandidateSnapshot뿐이다. Combat/Status가 resolved request를 계산해 fragment를 만드는 일반 executor와 Skill 비용·cooldown fragment, reaction fragment API는 아직 없다. 경계 선택과 outbox 근거는 ADR-006에 기록했다.

// Executable API shape · abridged
var specification = new ReferenceEffectSpecification(
    specificationId,
    new ReferenceEffectOperationSpec[]
    {
        new DamageEffectSpec(
            operationId,
            formulaId,
            damageType: "fire",
            baseValue: 24,
            coefficientBps: 12_000,
            tags: new[] { "spell", "fire" })
    });

EffectTargetResolutionPlan resolution =
    ReferenceEffectTargetResolver.Resolve(
        specification,
        context,
        new[]
        {
            new EffectTargetRule(
                operationId,
                EffectTargetMode.ExplicitTarget)
        });

DamageEffectOperation request =
    (DamageEffectOperation)resolution.Operations.Single().BoundOperation;
실행 증거. AdvancedFoundationVerification.cs는 정상 Self/Explicit/candidate bind, 같은 metric의 EntityId tie-break, 빈 snapshot과 missing explicit target의 NotApplicable, duplicate candidate와 rule mismatch 실패를 검증한다.
  1. Intent. target-independent Effect spec과 실행 context를 검증한다.
  2. Resolve targets once. adapter가 eligibility와 정수 metric 값을 확정하면 EffectTargetCandidate 생성자가 metric 범위를, CanonicalTargetSnapshot 생성자가 null·중복과 canonical total order를 보장한다. resolver는 이 동결 입력을 받아 mode·필수 snapshot을 확인하고 MaxTargets만 적용하며 팀·거리·시야를 다시 판정하지 않는다.
  3. Resolved operations. 대상별 DamageRequest·ApplyStatusRequest를 만들며 이후 단계에서는 대상을 다시 고르지 않는다.
  4. Pure resolve. Combat 등 소유 resolver가 상태를 바꾸지 않고 결과를 계산한다. 현재 C# Combat은 미리 해석된 CombatContext.ScalingStatValue로 compact 공식을 실행한다.
  5. CommitPlanFragment. 현재 reference는 version precondition, mutation, outbox event를 준비한다. reaction command는 아직 fragment에 포함하지 않는다.
  6. Compose. 현재 composer는 Effect fragment끼리만 합친다. Skill 비용·쿨다운까지 일반화한 outer composer는 아직 없다.
  7. Atomic commit. stale snapshot이면 전체 plan을 거부하고 state와 outbox를 모두 유지한다.
  8. Commit then react. commit receipt의 fact만 외부 dispatcher가 전달하고, 상태 변경 반응은 ReactionQueue의 새 command가 된다.
현재 commit 증거의 저장 범위.

C# RuntimeCommitter와 JavaScript StateStore의 state+outbox 원자성은 in-memory reference다. 프로세스 장애와 다중 writer를 견디는 durable transaction, outbox row claim/lease, publisher retry, consumer idempotency는 제품 adapter의 책임이며 현재 실행 참조가 제공한다고 해석하면 안 된다.

Target Selection 설계

Executable target contract. Unity Physics·서버 spatial index adapter는 eligibility를 판정하고 정수 metric 값을 제공한다. EffectTargetCandidate 생성자는 metric 범위를 검증하고, CanonicalTargetSnapshot 생성자는 null·duplicate EntityId를 거부한 뒤 SelectionPriority DESC → DistanceSquared ASC → EntityId Ordinal ASC로 정렬한다. ReferenceEffectTargetResolver는 이 canonical 순서를 유지하며 MaxTargets를 적용한다. 따라서 Physics 반환 순서, culture, 부동소수 거리 tie가 domain 결과를 바꾸지 않는다. UnityEngine.Vector3나 Collider는 domain 계약에 들어오지 않는다.
Target 타입 의미 예시
Self 시전자 자신 회복, 보호막, 버프
ExplicitTarget 스킬이 지정한 대상 단일 타겟 주문
CandidateSnapshot adapter가 필터·metric 계산을 끝낸 후보 스냅샷 최대 N개를 priority·정수 거리·ID 순으로 선택
AreaCircle 지점 기준 원형 범위 Fireball, 폭발
ForwardArc 시전자 전방 부채꼴 근접 베기

현재 범위와 두 타겟팅의 차이. Skill의 TargetingSpec 은 사용자가 무엇을 조준할 수 있는지 검증하고, Effect의 TargetSelector 를 위한 공간 adapter는 팀·거리·시야·충돌 같은 eligibility를 먼저 판정한다. 현재 Effect domain 값 생성 경계가 candidate metric, snapshot null·중복·결정적 정렬을 보장하고, resolver는 Self를 caster에 bind하고 ExplicitTarget의 ID 존재를 확인하며 CandidateSnapshot mode에서 필수 snapshot과 MaxTargets만 처리한다. 입력 가능한 대상인지는 다시 검증하지 않는다. caller가 정한 release/hit 시점의 immutable snapshot에서 대상을 한 번만 bind하며, AreaCircle·ForwardArc·Chain·OwnerOrSource는 위 표의 production 설계 예시이고 현재 resolver mode가 아니다.

ValueExpression과 DamageRequest 예시

Heal이나 ModifyResource처럼 Effect가 결과값을 소유하는 경우에는 ValueExpression을 계산한다. 피해는 같은 표현식을 Effect와 Combat에서 두 번 계산하지 않도록 DamageEffect가 아래 입력을 요청으로 만든다. 현재 실행 참조는 한 개의 CoefficientBps와 미리 해석된 ScalingStatValue만 지원한다. 여러 scaling 항과 Combat 내부 Stat 조회는 개념 확장이다. 

{
  "id": "effect.fireball-damage",
  "type": "damage",
  "damageType": "fire",
  "target": { "type": "explicit_target" },
  "damageRequest": {
    "formulaId": "combat.fire.v3",
    "baseValue": 24,
    "coefficientBps": 12000
  },
  "combatContext": { "scalingStatValue": 120 },
  "conditions": ["target_is_alive"],
  "tags": ["spell", "fire", "single-target"]
}

C# DamageRequest handoff

JSON 정의를 읽은 DamageEffect adapter는 실제 C# 요청 타입을 만들고, 계산은 ICombatResolver에 위임한다. DamageRequest 생성자는 빈 damageType, 음수 base/coefficient, null tags와 빈 tag를 거부하고 IEnumerable<string>을 읽기 전용 복사본으로 보존한다.

DamageRequest request = new(
    attackerId,
    defenderId,
    SourceRef.SkillExecution(skillDefinitionId, commandId),
    formulaId,
    damageType: "fire",
    baseValue: 24,
    coefficientBps: 12_000,
    tags: new[] { "spell", "fire", "single-target" },
    seed: 61_710u);

// CombatContext는 상위 계층이 미리 해석한 snapshot 값이다.
DamageResult result = combatResolver.Resolve(request, combatContext);

전체 생성자 불변식과 필드 타입은 실행 참조의 Combat.cs가 기준이며, Combat 페이지의 C# 요청·결과 계약에서 함께 설명한다.

EffectResult와 실행 trace

Conceptual observability example. 아래 EffectTrace, EffectCompleted, EffectRuntimeTrace는 현재 C# source에 선언된 실행 타입·event가 아니라 제품 관찰 모델 예시다. 실행 C#에는 guarded EffectResult/EffectBundleResult, composition 결과와 실제 commit fact가 있고, JavaScript Fireball에는 DamageCalculated trace와 committed damage/status event가 있다. 제품 trace schema를 만들 때 둘을 연결해야 한다.

공개 결과 타입은 호출자가 분기할 최소 정보만 반환한다. 단일 operation은 성공 여부와 OperationId, 묶음은 commit 여부와 적용·reaction 개수를 제공한다. 대상별 계산값, 생성 이벤트, 실패 근거와 디버그 메시지는 EffectTrace와 각 소유 시스템 결과에 별도로 남겨 결과 계약이 비대해지지 않게 한다.

EffectBundleResult 생성자는 두 count의 음수를 거부하고, Committed == false이면 적용 effect와 queued reaction 수가 모두 0이도록 강제한다. 실패했는데 일부 적용을 보고하는 모순된 carrier를 만들 수 없다.

  - caster: entity.caster

실패 처리와 트랜잭션

실패 처리는 하나의 열거형으로 고를 문제가 아니다. 대상별 실행, 주 transaction 범위, commit 이후 reaction, 외부 부작용 보상은 서로 다른 축이며 필요한 정책을 조합한다.

정책 축 선택지 의미와 사용 예
대상·operation 실행 BestEffortPerTarget 대상별 독립 command를 만들어 가능한 대상만 적용한다. 광역 버프·디버프처럼 한 대상의 실패가 다른 대상의 성공을 지우면 안 될 때 사용한다.
대상·operation 실행 AllOrNothing 주 plan의 operation 하나라도 사전 검증이나 precondition에 실패하면 아무것도 commit하지 않는다.
transaction 경계 SameCommit / SeparateCommand 비용·쿨다운·주 피해처럼 함께 확정할 변경은 한 plan에 넣고, 독립적으로 실패할 수 있는 후속 상태 적용은 별도 command로 분리한다.
commit 이후 reaction CommitThenReact 주 transaction의 committed fact만 ReactionQueue command로 변환한다. 면역 정책을 추가한 확장에서는 Burn 면역 실패가 이미 확정된 피해를 되돌리지 않는다.
외부 부작용 보상 CompensatingAction 이미 외부로 전달된 작업은 과거 상태를 rollback한다고 가장하지 않고 결제 취소·지급 회수 같은 명시적 보상 command를 실행한다.
정책 증거 범위. DeterministicEffectCommitPlanComposer는 Effect fragment의 동일 precondition만 dedupe하고, 서로 다른 expected version, duplicate operation-target/fragment/event, 같은 resource의 복수 mutation을 commit 전에 거부한다. NotApplicable만 있으면 NoChanges, 하나라도 Rejected면 partial plan 없이 Rejected다. AllOrNothing·BestEffortPerTarget 선택, Skill cost/cooldown fragment, reaction fragment는 아직 production 확장이다. Fireball reference의 Skill+Damage 원자성은 여전히 수동 CommitPlan 구성 증거다.

실행 계획과 Commit 경계

월드 이동, 투사체 생성, 이벤트 발행처럼 되돌리기 어려운 작업에 일반적인 rollback을 약속하면 구현이 불안정해진다. Effect는 먼저 검증하고 EffectPlan 을 만든 뒤, 가역적인 상태 변경과 outbox event를 한 번에 commit한다. 외부 adapter는 commit된 outbox만 dispatch하며 idempotency key로 중복 적용을 막는다.

C# Effect fragment 조합

Combat·Status 같은 소유 resolver가 target-bound request를 계산한 뒤 precondition·mutation·event를 fragment로 투영한다. 아래 API는 실행 가능하지만 preconditions·mutations·outboxEvents를 만드는 소유 resolver는 호출자가 제공해야 한다.

// Executable API shape · abridged
var fragment = new CommitPlanFragment(
    fragmentId,
    resolved.OperationId,
    resolved.TargetId,
    preconditions,
    mutations,
    outboxEvents);
var outcome = EffectOperationOutcome.Applied(resolved, fragment);

EffectCommitComposition composition =
    DeterministicEffectCommitPlanComposer.Compose(
        commandId,
        new[] { outcome });

if (composition.Status == EffectCompositionStatus.Ready)
    runtimeCommitter.Commit(composition.Plan!);
정상·경계·실패 증거. verifier는 입력 outcome 순서를 뒤집어도 같은 mutation/event 순서가 나오는 정상 경로와 실제 InMemoryRuntimeCommitter commit, NotApplicable의 NoChanges 경계를 검증한다. conflicting precondition, mutation collision, duplicate fragment/event/outcome, operation-wide NotApplicable과 applied target의 모순은 모두 실패로 검증한다.

결정론과 랜덤 시드

EffectContext의 RandomSeed는 확률 분기를 재현하기 위한 입력 중 하나다. 같은 결과를 보장하려면 Core replay envelope처럼 입력 스냅샷, Definition·Formula·Numeric 버전, 대상 정렬, ClockDomain, 반올림 정책과 이벤트 순서도 함께 고정해야 한다. Seed만 같아서는 충분하지 않다.

데이터 모델

제품에서는 정의 데이터를 콘텐츠로 저장하고, 런타임의 EffectResult·outbox·trace를 재현 정보로 연결한다. 현재 C# reference는 이 영속 데이터·trace 저장소를 구현하지 않는다. EffectResult 자체를 세이브 데이터처럼 영구 저장할 필요는 보통 없지만, command와 SourceRef로 trace를 다시 찾을 수 있어야 디버깅과 리플레이가 가능하다.

Combat / Status 연결

Effect System은 DamageEffect와 ApplyStatusEffect의 다음 책임자를 명확히 가진다. DamageEffect는 최종 피해 공식을 직접 계산하지 않고 DamageRequest 를 만들어 CombatResolver에 전달한다. ApplyStatusEffect는 지속시간과 tick을 직접 관리하지 않고 ApplyStatusRequest 를 만들어 StatusSystem에 전달한다.

Combat

DamageEffect → CombatResolver

명중, 치명타, 방어, 저항, 보호막 계산은 Combat Resolution System에서 DamageResult로 정리한다.

Status

ApplyStatusEffect → StatusSystem

버프, 디버프, DoT, CC의 지속시간, 중첩, tick, 해제 정책은 Status System이 관리한다.

Comprehension check

이해도 확인

Effect가 결과를 직접 소유하는 부분과 Combat·Status에 요청으로 넘겨야 하는 부분을 구분해 본다.

  1. 01 · 책임 경계
    DamageEffect가 주문력 계수, 치명타, 저항과 보호막까지 계산해 최종 피해를 만들어도 될까?
    정답과 해설

    안 된다. Effect는 FormulaId, BaseValue, CoefficientBps, 대상과 SourceRef를 DamageRequest에 담고 최종 피해 계산은 Combat에 위임한다.

  2. 02 · 경계 상황
    개념 확장에 Burn 면역을 추가한 뒤 후속 판정이 실패하면 어떤 결과가 남아야 할까?
    정답과 해설

    주 transaction에서 이미 commit된 피해와 EffectBundleResult는 유지하고, 면역 실패는 별도 StatusResult와 trace에 남긴다. 현재 실행 fixture에는 면역 입력과 판정이 없다.

  3. 03 · 시스템 연결
    입력 시점에 유효했던 대상이 release 시점에 이동했다면 Skill과 Effect는 각각 어디까지 다시 판단해야 할까?
    정답과 해설

    Skill은 입력 가능한 대상을 검증한다. caller는 정책이 정한 release/hit 시점에 adapter snapshot을 만들고 Effect resolver는 그 snapshot을 한 번만 bind한다. 이후 commit precondition이 stale이면 전체 plan을 거부하며 같은 command에서 대상을 자동 재선택하지 않는다.

설계 점검

  • EffectDefinition과 EffectRuntime 실행 상태가 분리되어 있다.
  • EffectContext가 CasterId, InitialTargetId, Source, RandomSeed를 포함한다.
  • Self·ExplicitTarget·CandidateSnapshot resolver와 Area·Arc·Chain product 확장 범위가 구분되어 있다.
  • 현재 target-independent spec은 Damage·ApplyStatus만 실행하며 Heal·Resource·Modifier·Dispel·Spawn은 카탈로그 확장임을 표시한다.
  • EffectOperationResult와 EffectBundleResult는 호출 분기에 필요한 최소 결과를 담고, 상세 실패·계산 근거는 SourceRef로 trace와 연결한다.
  • Effect fragment composer가 precondition·mutation·outbox를 조합하는 범위와, 아직 없는 Skill·reaction fragment 및 post-commit dispatcher 범위가 구분되어 있다.
