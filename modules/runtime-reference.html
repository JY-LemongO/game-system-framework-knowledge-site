증거 수준과 실행 참조 한계

Normative contract는 요청·결과 경계, 정렬 순서, 원자성, replay 불변식처럼 구현이 지켜야 할 규칙이다. Executable reference는 이 저장소의 C# 참조와 JavaScript 커널이 실제 빌드·테스트로 검증하는 동작이다. Conceptual extension은 계약을 적용할 수 있는 설계 확장이지만 현재 실행 참조가 보증하지 않으며, Out of scope는 이 학습용 참조가 의도적으로 구현하지 않는 영역이다.

C# 참조는 net9.0 검증 프로젝트이며 특정 Unity 버전·scripting backend·API 호환성을 보증하지 않는다. Unity에 적용할 때는 Unity의 .NET API profile과 C# compiler 지원 범위를 확인하고, 엔진 resource·animation·clock·serialization adapter를 별도로 맞춰야 한다. Burst 경로는 지원 타입 제한 때문에 decimal domain 참조와 같은 결과를 내는 fixed-point adapter가 필요할 수 있다.

Reaction

상태 변경 반응은 queue로 직렬화한다

numeric priority 오름차순(작은 값 우선), stableOrderKey ordinal 오름차순, reactionId ordinal 오름차순으로 정렬한다. 한 번의 Drain이 하나의 causation wave를 열고, dispatch 중 추가된 자식의 depth는 호출자가 제출한 숫자가 아니라 queue가 parent.depth + 1로 파생한다. 개수·누적 budget·인과 depth·idempotency 상한을 넘으면 handler가 지역 예외를 삼켜도 wave 전체가 실패한다. 후속 command의 CausationId는 직접 촉발한 직전 event로 전진한다.

event → bounded causation wave → new command

예약 시각과 실제 처리 시각을 함께 보존한다

정상 Burn은 +2, +4, +6에 tick하고 +6 tick과 expire를 같은 commit으로 닫는다. 지연 복구에서는 payload의 tickAt/scheduledExpireTick은 원래 예약 시각을, envelope의 occurredTick과 endedTick은 max(schedule, current simulation tick)인 실제 commit 시각을 기록한다. instance별 catch-up budget을 넘겨도 아직 active인 status의 overdue tick은 다음 AdvanceTo로 미룬다. 이미 expire 시각까지 지난 status의 남은 backlog만 catch-up-limited로 명시 폐기하고 status를 제거한다.

JavaScript 공개 경계.

StateStore는 입력을 한 번만 canonical snapshot으로 만든 뒤 runtime state·StatusInstance·기존 outbox를 검증하고, 내부 state·clock·처리 command·outbox를 private하게 보관한다. commit은 CommandEnvelope와 CommitPlan의 v2 정확 필드 집합, 판별형 operation, 유한·비순환 JSON payload를 mutation 전에 강제한다. __proto__도 일반 데이터 키로 보존하며 sparse array·accessor처럼 JSON에 없는 shape은 거부한다. 모든 resource 덧셈은 매 단계 safe integer인지 확인하고, Status add/remove는 instance의 부재/존재를 각각 요구한다.

Fireball resolve는 command의 actor·tick·data version·target·skill을 정규화된 input에 RNG 전에 정확히 결속한다. formula·critical 중간값은 BigInt 기약 유리수로 유지하고, JSON에는 canonical numerator/denominator 문자열을 기록한다. 표시용 raw 정수를 같은 primary 완화 입력으로 재사용하지 않으며, raw 투영과 exact 저항 결과를 각각 half-away-from-zero 경계에서 변환한다. 이 fixture의 Burn은 별도 downstream formula로 committed raw 정수를 명시적으로 snapshot한다.

Commit은 임의 문자열 event를 확장점으로 받아들이지 않고 공개 schema의 닫힌 taxonomy만 허용하며, 각 built-in/probe fact를 command, pre-state, operation, post-state에 대조한다. 특히 피해 event는 exact raw의 canonical 표현·투영, 실제 저항 stat, Shield 우선 분배, 보존식, SourceRef를 검증하고 Status·사망 event는 해당 add/patch/remove·치명 전이와 provenance를 검증한다. enqueueReactions는 event를 reaction으로 투영하는 순수 builder이고 그 자체는 권위 경계가 아니다. 실제 applyStatusReaction은 causation ID가 같은 store outbox의 유일한 DamageCommitted인지 확인하고, 그 event에서 ID·정렬·budget과 actor/source/target/Burn payload 전체를 다시 파생해 정확히 일치할 때만 commit한다. event가 정한 최초 causal depth=1은 낮출 수 없고, 더 큰 depth는 ReactionQueue가 현재 부모에서 파생해 상한을 적용한다. 또한 handler는 queue의 동기 dispatch 중 동일 reaction object에만 부여되는 내부 권한을 요구하므로 직접 호출로 wave budget을 우회할 수 없다. 따라서 raw input이나 위조 reaction은 후속 상태를 만들 수 없고, 정상적인 nested reaction도 부모 depth를 보존한다.

Commit과 내부 clock은 재진입할 수 없고 Status의 appliedTick·nextTickAt은 되감을 수 없다. trace에는 동결 snapshot만 전달하며 관찰 callback은 어느 store나 ReactionQueue에도 새 mutation을 주입할 수 없다. 실패하면 state·tick·outbox·멱등 기록은 모두 그대로이고, 읽기 getter는 동결된 복사본만 반환한다.

제품 경계: 이 원자성은 한 JavaScript 프로세스의 in-memory working copy에 대한 교육용 보장이다. 서버 재시작·다중 writer·DB transaction·durable outbox 저장·claim/lease·publisher retry를 구현하지 않는다. 생성자의 기존 state/outbox도 신뢰된 restore snapshot이라는 전제이며, 외부 wire나 저장소에서 읽은 event의 인증·스키마 migration·aggregate 소속 검증은 제품 admission adapter가 먼저 수행해야 한다. 제품에서는 state 변경과 outbox insert를 같은 내구성 transaction에 넣고, dispatcher의 at-least-once 전달과 consumer idempotency를 별도로 설계해야 한다.

Reaction wave 실패와 재시도.

maxReactions와 maxBudget은 1 이상, maxDepth는 0 이상이어야 한다. Drain 인자와 queue 생성자의 제한 중 더 작은 값이 현재 wave의 상한이다. 시작 시 이미 상한을 넘었거나 dispatch 중 새 reaction이 상한을 넘으면 예외를 전달하고 아직 dispatch하지 않은 항목을 폐기한다. 이미 handler가 commit한 사실은 queue가 되돌릴 수 없다.

전체 wave를 새 business key로 다시 실행하지 않는다. A가 commit되고 B/C가 미실행인 상황에서 새 idempotency key를 주면 A가 중복 적용될 수 있다. 제품 구현은 reaction별 Committed / OutcomeUnknown / Undispatched / PermanentFailure disposition을 내구 저장하고, B/C만 같은 reaction ID·business idempotency key·command ID로 명시적으로 재시도해야 한다. 현재 메모리 reference는 durable disposition/retry API를 제공하지 않으므로 실패 trace는 진단 증거일 뿐 안전한 복구 기능이 아니다.

        현재 호환 lane.

        runtimeVersion=4.0.1-reference, contractSchemaVersion=2, replayFormatVersion=2다. v2는 피해 fact의 필수 exact fraction과 replay outcome/trace projection shape를 고정하며, v1 wire object는 묵시 변환하지 않고 거절한다.

        numericPolicyVersion=integer-bps-half-away-from-zero-v1과 formulaVersion=combat.fire.v3는 유지한다. 이 두 규범은 처음부터 exact intermediate와 primary 완화 전 raw 미되먹임을 요구했고, Runtime 4.0.0의 단계 반올림은 정책 변경이 아니라 구현 결함이었다. 4.0.1 runtime과 v2 wire/replay lane이 교정된 결과를 구 결과와 분리한다.

        조건부 envelope 필드.

        현재 단일 대상 Fireball replay header에는 대상 집합 정렬이 없으므로 targetOrderPolicyVersion을 넣지 않는다. 아래 다중 대상 캡스톤처럼 선택 순서가 결과를 바꿀 수 있는 실행에서는 이 필드를 추가해 정책 의미를 고정한다.

        증거 경계 · 실행 가능한 rubric, 비실행 설계 워크시트.

        아래 도구는 제출 JSON의 token 조합을 검사하고 선택한 token으로 예시 evidence를 합성한다. 학습자가 작성한 resolver나 CommitPlan builder를 실행하지 않으며, 구현 능력·중급 진입·production readiness를 인증하지 않는다. 용도는 책임과 정책 누락을 찾는 형성평가다.

        Chain Lightning 계약 워크시트

        입력 순서가 섞인 대상을 처리하는 Chain Lightning + Shock의 책임·결정성·실패 의미를 하나의 JSON 계약으로 작성한다. starter의 null과 빈 배열을 채운 뒤 token rubric과 합성 scenario feedback으로 누락된 정책을 찾는다. 실제 전이 학습 증거는 별도의 C# native candidate runner가 learner 구현을 정상·경계·실패 입력으로 실행할 때만 성립한다.

        형성평가
        01 · Resolve

        순서와 RNG를 입력 순회에서 분리

        중복 target 요청은 전체를 거부한다. 전체 후보를 distanceBucket asc, 동률이면 EntityId ordinal asc로 먼저 정렬한 뒤 최대 3개를 고른다. 각 Hit 판정은 correlationId + decisionKind + targetId keyed RNG를 사용한다.

        02 · Commit

        주 효과는 하나의 원자적 plan

        Mana 비용·쿨다운·모든 주 피해와 outbox를 함께 확정한다. caster와 모든 target version 중 하나라도 stale이면 아무 mutation과 event도 남기지 않는다.

        03 · Reaction

        직접 피해 event에서 Shock 생성

        각 DamageCommitted가 자기 Shock reaction의 직접 원인이다. budget 실패 전 이미 확정된 primary와 reaction commit은 유지하고 아직 dispatch하지 않은 항목만 폐기한다. 재시도는 reaction별 disposition을 확인해 미커밋 항목만 같은 business idempotency key와 command ID로 수행한다. 새 key로 전체 wave를 재실행하는 것은 이미 commit된 반응을 중복 적용하므로 금지한다.

        04 · Status

        제품 규칙과 시간·출처를 명시

        보호막이 전부 흡수해도 Hit이고 대상이 생존했다면 Shock을 적용한다. 적용 command와 StatusInstance.applicationCausationId는 직접 DamageCommitted를 가리키고, StatusApplied event는 그 적용 command를 가리킨다. 첫 tick command는 StatusApplied, 다음 tick command는 직전 StatusTicked를 원인으로 삼는다. simulation_tick +4에는 피해 resource delta·StatusTicked·StatusExpired·status remove를 한 status-tick plan에서 원자적으로 확정한다.

        제출 토큰을 찾는 방법

        제출 JSON Schema에 각 필드에서 사용할 수 있는 정확한 후보 token과 scenario slug가 있다. 후보에는 오답도 포함되므로 위 요구와 기존 계약을 근거로 선택한다. assessor source는 판정과 probe가 어떻게 계산되는지 검토할 수 있지만 완성 제출물을 생성하는 공개 API는 제공하지 않는다.

        판단 근거는 ADR-001 · 결정론, ADR-002 · resolve/commit, ADR-003 · reaction에서 확인한다.

        target:*는 “선택된 모든 대상마다 명시적 version precondition이 있어야 한다”는 assessor 표기다. 실제 CommitPlan에 wildcard ID를 넣으라는 뜻이 아니며, production plan에서는 선택된 각 EntityId를 열거한다.

        이해도 확인

        실행 결과를 계산하는 단계와 실제 상태를 확정하는 단계, 그리고 같은 실행을 재현하기 위해 고정해야 할 입력을 구분해 본다.

        1. 01 · 책임 경계
          Resolve가 DamageResult를 계산하면서 target HP도 바로 변경하면 왜 replay와 commit 경계가 무너질까?
          정답과 해설

          Resolve는 같은 snapshot과 command에서 같은 plan을 만드는 순수 단계여야 한다. 여기서 상태를 바꾸면 재실행할 때 입력이 달라지고, commit의 version 검사와 전체 성공·실패 보장을 적용할 수 없다.

        2. 02 · 실패 안전성
          Fireball을 resolve한 뒤 다른 command가 같은 대상 entity의 cooldown을 변경했다면 stale plan의 commit은 무엇을 남겨야 할까?
          정답과 해설

          변경한 필드가 피해와 직접 관련 없어 보여도 entity expectedVersion이 달라졌으므로 전체 commit을 거부해야 한다. stale plan의 HP·Shield mutation과 outbox event는 하나도 남지 않아야 하며, 최신 snapshot으로 다시 resolve할 수 있다.

        3. 03 · 재현 계약
          rootSeed가 같아도 formulaVersion이나 stableOrderKey 규칙이 달라지면 같은 결과를 보장할 수 있을까?
          정답과 해설

          보장할 수 없다. 결정론은 seed뿐 아니라 runtime·RNG algorithm·RNG key schema·clock domain·수치·definition·formula 버전과 정렬 규칙을 포함한 replay envelope 전체가 같을 때 성립한다.
