Learning goals

학습 목표

C# 구현이 보장할 동일 입력 replay, 원자적 commit, 제한된 reaction과 context cache 결과를 직접 관찰한다. 저장 문서의 단계별 migration은 필수 흐름을 익힌 뒤 선택해서 확인한다.

Prerequisites

선수 개념

Fireball 사례를 먼저 읽고 다음 경계를 설명할 수 있어야 한다.

  • C#의 불변 요청·결과, 인터페이스, 읽기 전용 snapshot
  • 순수 resolve와 원자적 commit, rollback, resource invariant
  • commandId 멱등성, optimistic version 사전 조건, state와 outbox의 동시 확정
  • 결정론적 RNG, stableOrderKey, correlation·causation, version envelope

낯선 용어는 용어집의 Runtime 계약 항목을 먼저 확인한다.

Resolve

읽고 계산하되 바꾸지 않는다

고정 snapshot, version envelope, keyed RNG만으로 immutable outcome과 operation plan을 만든다. target이나 caster 객체는 변경하지 않는다.

snapshot + command + policies → plan
Commit

전부 성공하거나 아무것도 바꾸지 않는다

idempotency와 entity version을 먼저 확인하고 working copy에서 모든 operation과 resource invariant를 검증한 뒤 state를 한 번 교체한다.

plan + preconditions → state + outbox
Reaction

상태 변경 반응은 queue로 직렬화한다

numeric priority 오름차순(작은 값 우선), stableOrderKey ordinal 오름차순, reactionId ordinal 오름차순으로 정렬한다. 한 번의 Drain이 하나의 causation wave를 열며, dispatch 중 추가된 reaction도 같은 depth·개수·budget 합계와 idempotency 상한 안에서 그 wave가 닫히기 전에 실행한다.

event → bounded causation wave → new command
Status time

마지막 tick 후 같은 시각에 만료한다

Burn은 +2, +4, +6에 tick하고 +6 tick을 commit한 뒤 expire한다. 프레임 지연 catch-up에는 instance별 budget이 있다.

tick ≤ expire → tick, then expire

Reaction wave 실패 정책.

Drain 인자와 queue 생성자의 제한 중 더 작은 값이 현재 wave의 상한이다. 시작 시 이미 상한을 넘었거나 dispatch 중 새 reaction이 상한을 넘으면 InvalidOperationException을 호출자에게 전달하고 아직 dispatch하지 않은 항목을 모두 폐기한다. dispatch handler 자체가 예외를 내도 같은 방식으로 중단하므로 남은 항목이 다음 Drain에서 조용히 실행되지 않는다. 이미 handler가 별도 commit한 사실은 queue가 되돌릴 수 없으므로 handler는 멱등 command와 원자적 commit 경계를 사용해야 한다.

      Hardening Probes

      happy path만 보여주지 않고 실제로 가장 위험한 세 실패를 같은 kernel에서 재현한다. 실패 전후 state hash가 같아야 probe를 통과한다.

      01 · Idempotency

      Duplicate command

      이미 처리한 commandId를 다시 commit한다.

      예상 결과 DUPLICATE_COMMAND로 거부되고 state와 outbox가 늘어나지 않는다.

      왜 유지해야 하나 같은 요청의 재전송이 비용·피해·이벤트를 두 번 적용하면 안 된다.

      확인할 점 commandId가 한 번만 처리되는 멱등 키인지 확인한다.

      실행 전 · 기대: 중복 요청 거부, 상태 해시 동일
      02 · Concurrency

      Version conflict

      resolve 뒤 shield version을 외부 command로 바꾼다.

      예상 결과 VERSION_CONFLICT로 stale plan을 거부하고 추가 mutation을 남기지 않는다.

      왜 유지해야 하나 최신 shield를 오래된 snapshot 결과로 덮으면 동시 변경이 유실된다.

      확인할 점 expectedVersion 사전 조건이 commit 직전에 다시 검사되는지 확인한다.

      실행 전 · 기대: stale plan 거부, 상태 해시 동일
      03 · Transaction

      Atomic rollback

      첫 operation 뒤 invalid resource operation을 넣는다.

      예상 결과 RESOURCE_OVERFLOW로 전체 commit이 실패하고 첫 operation도 취소된다.

      왜 유지해야 하나 비용만 빠지거나 피해만 남는 부분 성공은 transaction invariant를 깨뜨린다.

      확인할 점 working copy를 검증한 뒤 한 번에 교체하는지 확인한다.

      실행 전 · 기대: working copy 폐기, 상태 해시 동일

        Comprehension check

        이해도 확인

        실행 결과를 계산하는 단계와 실제 상태를 확정하는 단계, 그리고 같은 실행을 재현하기 위해 고정해야 할 입력을 구분해 본다.

        1. 01 · 책임 경계
          Resolve가 DamageResult를 계산하면서 target HP도 바로 변경하면 왜 replay와 commit 경계가 무너질까?
          정답과 해설

          Resolve는 같은 snapshot과 command에서 같은 plan을 만드는 순수 단계여야 한다. 여기서 상태를 바꾸면 재실행할 때 입력이 달라지고, commit의 version 검사와 전체 성공·실패 보장을 적용할 수 없다.

        2. 02 · 실패 안전성
          Fireball을 resolve한 뒤 다른 command가 대상 shield를 변경했다면 stale plan의 commit은 무엇을 남겨야 할까?
          정답과 해설

          expectedVersion 불일치로 전체 commit을 거부해야 한다. stale plan이 만든 HP·Shield mutation과 outbox event는 하나도 남지 않아야 하며, 최신 snapshot으로 다시 resolve할 수 있다.

        3. 03 · 재현 계약
          rootSeed가 같아도 formulaVersion이나 stableOrderKey 규칙이 달라지면 같은 결과를 보장할 수 있을까?
          정답과 해설

          보장할 수 없다. 결정론은 seed뿐 아니라 runtime·RNG·수치·definition·formula 버전과 정렬 규칙을 포함한 replay envelope 전체가 같을 때 성립한다.
