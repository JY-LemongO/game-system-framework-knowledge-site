runtime kernel 안의 규칙은 UI나 저장소를 직접 알지 않는다. 외부 시스템은 command, snapshot, committed result, observation event 포트로만 연결한다.

Resolve

읽고 계산하되 바꾸지 않는다

고정 snapshot, version envelope, keyed RNG만으로 immutable outcome과 operation plan을 만든다. target이나 caster 객체는 변경하지 않는다.

snapshot + command + policies → plan
Commit

전부 성공하거나 아무것도 바꾸지 않는다

idempotency와 entity version을 먼저 확인하고 working copy에서 모든 operation과 resource invariant를 검증한 뒤 state를 한 번 교체한다.

plan + preconditions → state + outbox
Reaction

상태 변경 반응은 queue로 직렬화한다

priority, stableOrderKey, reactionId로 정렬하고 depth·개수·budget·idempotency 상한을 강제한다.

event → bounded reaction → new command
Status time

마지막 tick 후 같은 시각에 만료한다

Burn은 +2, +4, +6에 tick하고 +6 tick을 commit한 뒤 expire한다. 프레임 지연 catch-up에는 instance별 budget이 있다.

tick ≤ expire → tick, then expire
Resolve to reaction실패 시 mutation과 event가 모두 남지 않는 흐름

        다음 생산 구현

        P3-E1 · Runtime Port Interface

        in-memory StateStore를 대체할 transaction port, durable outbox port, clock port, definition registry port를 interface로 분리한다.

        Acceptance: 동일 golden fixture를 memory adapter와 production adapter가 함께 통과

        P3-E2 · Authoritative Server Slice

        client request의 command identity, validation, server tick, retry, reconciliation 결과를 계약에 추가한다.

        Acceptance: duplicate packet과 out-of-order packet이 상태를 두 번 바꾸지 않음

        P3-E3 · Multi-target / AoE

        target 후보를 stable ID로 정렬하고 대상별 RNG key와 전체 commit 원자성 정책을 결정한다.

        Acceptance: 입력 순서가 달라도 target별 outcome과 replay hash가 동일

        P3-F · Equipment MVP

        ItemInstance sourceRef, Affix→StatModifier, TriggeredEffect→ReactionQueue, equip/unequip idempotency를 첫 외부 콘텐츠로 붙인다.

        Acceptance: 해제 시 해당 item source의 변경만 제거
