Fireball Case Study

Fireball은 Skill, Effect, Combat, Status, Stat, Core Runtime이 한 번에 연결되는 대표 예제다. 이 페이지는 “스킬을 눌렀다”가 어떻게 마나 소모, 시전 타임라인, 피해 계산, Burn 상태, tick 피해, 이벤트 로그로 이어지는지 끝까지 추적한다.

SkillRequestEffectBundleDamageRequestDamageResultBurnDebugTrace

Fireball 전체 흐름

Input→Skill→EffectBundle→Combat→Status→EventLog

기존 예제가 Stat과 Effect의 연결을 설명했다면, 이제 Fireball은 전체 프레임워크의 수직 슬라이스다. Skill은 행동 사용 가능 여부와 실행 타이밍을 담당하고, Effect는 DamageEffect와 ApplyStatusEffect(effect_apply_burn)를 실행하며, Combat은 피해를 계산하고, Status는 Burn을 유지한다.

1. SkillRequest

플레이어가 지면 위치를 지정하면 SkillRequest가 생성된다. 요청에는 caster, skillId, aimPosition, inputTime, seed가 포함된다. 이때 아직 마나가 차감되거나 피해가 적용되지 않는다.

{
  "caster": "player_001",
  "skillId": "skill_fireball",
  "aimPosition": { "x": 14.5, "y": 0.0, "z": 7.0 },
  "inputTime": 1042.315,
  "seed": 930144
}

2. SkillValidator / Cost / Timeline

검사 항목판정실패 시 결과
배운 스킬인가SkillBook에서 skill_fireball 보유 확인SkillResult.failedReason = NotLearned
마나가 충분한가mana current ≥ 25OutOfResource
쿨다운 중인가cooldownRemaining = 0OnCooldown
상태 이상인가silence, stun, disarm 태그 없음BlockedByStatus
사거리와 시야range ≤ 18, lineOfSight trueInvalidTarget

검증이 통과되면 CostPolicy가 시전 시작 시 마나 25를 지불하고, ActionTimeline이 0.7초 뒤 release marker를 발생시킨다.

3. EffectBundle

release marker에 도달하면 SkillExecutor는 EffectContext를 만들고 EffectBundle을 실행한다. Fireball은 피해와 화상 부여를 순서대로 실행하지만, Burn이 면역으로 실패해도 피해 결과는 유지되도록 ordered_partial 정책을 사용한다.

{
  "effectBundle": {
    "policy": "ordered_partial",
    "effects": [
      "effect_fireball_damage",
      "effect_apply_burn"
    ]
  }
}

4. Combat Resolution

예시 값은 다음과 같다. spell_power와 fire_damage_bonus는 StatSystem에서 읽고, 대상의 fire_resistance와 보호막은 CombatPipeline에서 적용한다.

항목값출처
base damage40EffectDefinition
spell_power70attacker StatSystem
coefficient1.2Skill/Effect scaling
fire_damage_bonus+20%attacker StatSystem 또는 Status/Equipment Modifier
fire_resistance15%defender StatSystem
shield20defender Resource/Status
base = 40 + 70 × 1.2 = 124
124 × 1.2 fire bonus = 148.8
148.8 × 0.85 fire resistance = 126.48
126.48 - 20 shield = 106.48
finalHpDamage = floor(106.48) = 106

5. Status Application

Burn은 5초 지속, 최대 3중첩, 1초마다 periodic DamageEffect를 실행하는 DoT다. 재적용 시 stackCount가 증가하고 지속시간은 refresh된다. 적용 원천은 appliedBySourceId=skill cast sourceId로 추적하고, Status가 제공하는 Modifier나 tick 예약은 statusInstanceId를 sourceId로 사용해 정리한다.

{
  "status": "status_burn",
  "duration": 5.0,
  "stackCount": 1,
  "maxStacks": 3,
  "tickInterval": 1.0,
  "periodicEffect": "effect_burn_tick_damage",
  "appliedBySourceId": "skill:fireball:cast_771",
  "statusInstanceId": "statusInstance:burn:enemy_001:771",
  "modifierSourceId": "statusInstance:burn:enemy_001:771"
}

6. Burn Tick

Burn tick 피해는 snapshot 또는 dynamic 정책 중 하나를 선택한다. 이 문서의 기본 예시는 적용 당시의 공격자 계수를 보존하는 snapshot 방식이다. 대상의 현재 저항을 다시 읽을지 여부는 프로젝트별로 정책화한다.

7. Event / DebugTrace

Fireball의 모든 실행 결과는 sourceId로 묶여야 한다. 이렇게 해야 QA, 밸런스 조정, 리플레이, 서버 검증에서 같은 결과를 추적할 수 있다.

sourceId: skill:fireball:cast_771
OnSkillUsed
  caster=player_001, manaSpent=25, cooldown=6.0
OnEffectExecuted
  effect_fireball_damage targets=[enemy_001]
OnDamageResolved
  hit=true, crit=false, finalHpDamage=106, shieldAbsorbed=20
OnStatusApplied
  status_burn stack=1 duration=5.0 nextTick=+1.0
OnStatusTick
  status_burn tick=1 periodicEffect=effect_burn_tick_damage

책임 분리
