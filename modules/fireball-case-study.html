Fireball Case Study

Fireball은 Skill, Effect, Combat, Status, Stat, Core Runtime 경계를 연결하는 대표 예제다. C#은 대상/cooldown/mana 승인, snapshot 기반 피해 plan, 원자 commit, truthful outbox, bounded reaction과 dispatch 시 live-target 재검증을 실행한다. JavaScript fixture는 여기에 Burn instance 생성·tick·만료와 replay까지 이어서 추적한다.

예제 언어 · C# 실행 fixture · JavaScript SkillRequest EffectBundle DamageRequest DamageResult Burn DebugTrace
Learning goals

학습 목표

C# 계약과 JavaScript 실행 fixture의 증거 범위를 구분하면서 하나의 command가 Skill·Effect·Combat·Status 경계를 지나는 흐름과 실패 분기를 추적한다.

Prerequisites

선수 개념

각 시스템의 요청·결과 객체와 SourceRef, commit 이후 이벤트 개념을 먼저 이해한다.

Fireball 전체 흐름

Input → Skill → EffectBundle → Combat → Atomic Commit → DamageCommitted → ReactionQueue → Status → EventLog

Fireball은 전체 프레임워크의 수직 슬라이스다. Skill은 행동 사용 가능 여부와 실행 타이밍을 담당하고, 주 EffectBundle은 DamageEffect만 준비하며, Combat은 피해를 계산한다. RuntimeCommitter가 피해를 확정한 뒤 DamageCommitted를 ReactionCommand로 변환하고, ReactionQueue가 ApplyStatusEffect(effect.apply-burn)를 실행하면 Status가 Burn을 유지한다. 독립된 주 ApplyStatusEffect는 Status를 직접 요청할 수 있지만, 이 Fireball fixture의 Burn은 주 Damage와 병렬 실행하지 않는다. 아래 기본 피해 수치는 C# fixture와 JavaScript Runtime Lab이 공유하지만, C#은 Hit/Critical을 미리 정한 CombatContext로 받고 JavaScript는 keyed RNG로 판정한다.

개념적 C# 조율(비실행 의사코드)

목표형 Skill 정책에서는 release 시점에 비용·쿨다운·주 Damage의 변경 계획을 만들고 RuntimeCommitter가 mutation과 outbox를 한 transaction으로 확정한 뒤, commit된 DamageCommitted로 Burn ReactionCommand를 만든다. 이 경계라면 Burn 실패가 이미 commit된 피해를 되돌리지 않는다.

아래 코드는 현재 저장소에서 빌드되는 구현이 아니라 내부 조율 순서를 보여주는 의사코드다. SkillExecution·SkillReleasePlan·CommitPlanFactory·release timeline과 자원 예약은 Conceptual extension이다. 실행 가능한 실제 C# 경로는 FireballSkillPlanner.cs와 FireballReferenceScenario.cs이며, 검증 뒤 비용·쿨다운·Damage commit plan을 직접 만든다.

public FireballResult Execute(FireballCommand command)
{
    SkillExecution execution = _skills.Begin(command.SkillRequest);
    SkillReleasePlan release = _skills.PrepareRelease(execution);

    EffectContext context = command.CreateEffectContext(release);
    EffectBundlePlan effects = _effectPlanner.Prepare(
        release.EffectBundle, context);

    CommitPlan plan = _commitPlanFactory.Compose(
        release,
        effects);

    CommitReceipt receipt = _runtimeCommitter.Commit(plan);

    foreach (CommittedOutboxEvent fact in receipt.OutboxEvents)
        foreach (ReactionCommand reaction in
                 _reactionFactory.Create(release.EffectBundle, fact))
            _reactionQueue.Enqueue(reaction);

    return FireballResult.From(receipt);
}

1. SkillRequest

플레이어가 대상을 지정하면 불변 요청이 생성되지만 C# 계약과 JavaScript envelope의 필드는 다르다. C# SkillRequest는 CommandId, CasterId, SkillId, TargetId, RequestedTick, RootSeed 여섯 필드를 가진다. JavaScript는 command envelope에 command·actor·correlation·data version과 payload를 두고, RootSeed와 definition/formula/RNG version은 별도 replay header에 둔다. 어느 경로에서도 이 단계에는 아직 마나가 차감되거나 피해가 적용되지 않는다.

JavaScript command와 resolver input은 같은 권위여야 한다.

resolveFireball은 RNG를 읽기 전에 command의 actorId·requestedTick·dataVersion과 payload의 targetId·skillDefinitionId를 정규화된 input에 정확히 결속한다. payload의 누락·추가 필드까지 COMMAND_INPUT_MISMATCH로 거부하며, 이 실패에서는 RNG 호출·state·outbox가 모두 그대로다. command와 별도 input을 병렬 권위로 두고 서로 다른 값을 조용히 선택하면 replay와 보안 경계가 동시에 무너진다.

요청에서 cast identity를 파생한다.

C# FireballExecutionIdentity는 CommandId에서 SkillExecution Source와 bundle·event·reaction·멱등·정렬 ID를 만들고, RequestedTick + 60으로 cooldown ready tick을 계산한다. DamageRequest는 같은 요청에 기록된 RootSeed를 전달한다. 같은 CommandId를 다시 commit하면 DuplicateCommand이고, 정상적인 반복 시전은 새 CommandId와 최신 tick·snapshot을 사용하므로 이전 cast의 event·reaction ID와 충돌하지 않는다. RootSeed는 replay 입력이지만 전역 유일 ID일 필요는 없다.

이 반복 시전 증거는 현재 C# verifier에 있다. JavaScript runFireballScenario는 고정 command/correlation ID로 새 StateStore를 만드는 단일 fixture라서 같은 월드의 새 두 번째 cast를 공개 API로 검증하지 않으며, JS 쪽 확장 시 store와 새 command envelope를 주입하는 별도 orchestration이 필요하다.

2. SkillValidator / Cost / Timeline

검사 항목 판정 실패 시 결과
배운 스킬인가 SkillBook에서 skill.fireball 보유 확인 SkillResult.FailureReason = NotLearned
마나가 충분한가 Mana.Current ≥ 20 SkillResult.FailureReason = OutOfResource
쿨다운 중인가 cooldownReadyTick ≤ requestedTick SkillResult.FailureReason = Cooldown
상태 이상인가 silence, stun, disarm 태그 없음 SkillResult.FailureReason = ControlLocked
대상 생존 유효성 target 존재, target HP > 0 SkillResult.FailureReason = InvalidTarget
실행 범위와 실패 우선순위.

C# SkillAdmissionPolicy와 JavaScript Fireball은 모두 target 존재·생존 → cooldown ready → mana 충분 순서를 실행 검증한다. learned와 control lock은 아직 Conceptual이다. 쿨다운은 남은 시간을 별도 계산하지 않고 snapshot의 절대 readyTick을 비교하며, readyTick == requestedTick이면 사용 가능하다. 미래 ready tick은 Combat/RNG 전에 Cooldown으로 거절되고, cooldown과 mana가 동시에 실패해도 Cooldown 하나만 반환한다.

목표형 reserve_then_commit 정책에서는 검증 뒤 마나 20을 예약하고 release marker에서 live state를 다시 검사한 뒤 비용·쿨다운·주 Damage mutation·outbox를 하나의 commit plan으로 준비한다. 현재 실행 fixture는 예약과 timeline을 구현하지 않고 snapshot 검증 뒤 이 원자적 plan을 직접 만든다. resolve 뒤 cooldown·대상 version이 바뀌면 commit precondition이 plan 전체를 거절한다. 별도 Burn reaction의 실패는 이미 확정된 비용·쿨다운·피해를 되돌리지 않는다.

3. EffectBundle

목표형 release 조율에서는 SkillExecutor가 EffectContext를 만들고 Effect 계층에 EffectBundle plan 생성을 요청한다. 현재 실행 fixture는 이 timeline 단계를 축약하지만, 주 Damage mutation을 비용·쿨다운과 함께 commit하고 그 도메인 사실로 Burn 적용 reaction을 만드는 경계는 실행 검증한다. 면역 정책을 추가한 확장에서도 후속 Burn 실패가 이미 commit된 피해 결과를 되돌리지 않도록 commit_then_react 정책을 사용한다. 

{
  "effectBundle": {
    "policy": "commit_then_react",
    "effects": ["effect.fireball-damage"],
    "reactions": [
      {
        "on": "DamageCommitted",
        "when": "outcome == Hit && targetAlive",
        "effect": "effect.apply-burn"
      }
    ]
  }
}

4. Combat Resolution

예시 값은 다음과 같다. 현재 fixture에서는 호출자가 공격자 spell_power snapshot을 CombatContext.ScalingStatValue로 미리 해석하고, 대상의 fire_resistance 와 보호막도 미리 해석된 CombatContext 값으로 계산한다. CombatFormula가 직접 Stat을 조회하는 구조는 확장 설계다.

항목 값 출처
base damage 24 EffectDefinition
spell_power 120 attacker StatSystem
coefficient 1.2 CombatFormulaDefinition
critical multiplier 1.5 결정론적 CriticalResolver
fire_resistance 20% defender StatSystem
shield 40 defender Resource/Status 
decimal formulaDamage = 24m + (120m * 12_000 / 10_000m);
decimal exactRawDamage = formulaDamage * 15_000 / 10_000m;

// RawDamage=252 is a reporting projection. Mitigation keeps exactRawDamage.
int rawDamage = RoundDamage(exactRawDamage);
int resolvedDamage = RoundDamage(exactRawDamage * 8_000 / 10_000m);
int shieldAbsorbed = Math.Min(40, resolvedDamage);
int postShieldDamage = resolvedDamage - shieldAbsorbed;
int finalHpDamage = Math.Min(500, postShieldDamage);
int overkill = postShieldDamage - finalHpDamage; // fixture: 0

static int RoundDamage(decimal value) =>
    decimal.ToInt32(decimal.Round(
        value, 0, MidpointRounding.AwayFromZero));
반올림 위치가 공식의 일부다.

이 축약은 CombatResolver.cs와 같은 compact 정책이다. C#은 formula·critical 중간값을 decimal로, JavaScript Runtime 4.0.1은 BigInt 기반 기약 유리수로 유지한다. JavaScript damage_calculated trace의 rawDamageExact와 피해 outcome/event의 exactRawDamage는 JSON에서 정밀도를 잃지 않도록 canonical numerator/denominator 문자열을 쓴다.

RawDamage는 같은 primary 피해의 저항 계산에서는 exact scalar의 정수 투영이다. 저항 계산은 반올림 전 exact scalar를 사용하고, raw 표시값과 완화 결과를 각각 그 scalar에서 half-away-from-zero로 변환한다. 예를 들어 0 + 1 × 50% = 1/2에 치명타 150%를 적용하면 exact raw는 3/4, 표시 raw는 1이다. 저항 50%의 authoritative 결과는 3/8 → 0이지, 표시값을 되먹인 1 × 50% → 1이 아니다. C#·JavaScript 경계 테스트가 이 차이를 고정한다.

반면 이 Fireball fixture의 Burn은 별도의 downstream 정책으로, committed RawDamage 정수 투영에 ratio를 적용해 tick damage를 snapshot한다. exact primary scalar를 쓰는 정책으로 바꾸려면 Burn formula version과 golden fixture를 함께 올려야 한다. “primary 완화에 되먹이지 않는다”와 “후속 정책이 명시적으로 소비한다”는 서로 다른 경계다.

5. Status Application

현재 JavaScript fixture의 단일 Burn 인스턴스는 6 simulation tick 동안 유지되고 +2, +4, +6에 periodic damage resolution을 실행한다. 실제 경로는 StatusInstance의 예약 시각마다 공용 resolveDamageAgainstTarget를 직접 호출한다. 보호막이 최초 피해를 모두 흡수해도 Hit이고 대상이 살아 있으면 적용된다. 적용 원천은 skill execution SourceRef로 추적하고 tick 예약은 StatusInstanceId로 정리한다. 아래 maxStacks=1·refresh 정책은 목표 설계 예시이며, 현재 fixture는 재적용 병합이나 면역을 구현·검증하지 않는다.

6. Burn Tick

JavaScript Burn tick의 공격 계수는 적용 당시 raw damage에서 snapshot하고, 대상의 현재 저항과 보호막은 매 tick 다시 읽는다. 이 fixture에서는 raw tick 30이 화염 저항 20%를 거쳐 매번 HP 24를 감소시키며, 세 번의 tick 뒤 대상 HP는 266이다. 따라서 매 tick도 공용 피해 resolver의 DamageCalculated trace와 원자적 DamageCommitted 경로를 거치며 치명 피해라면 EntityDefeated를 남긴다. C# reference에는 아직 Status 저장소·tick executor가 없으므로 이 주기 실행은 JavaScript 증거다.

JavaScript fixture의 최소 tick 정책.

적중한 공격의 raw damage와 Burn ratio가 모두 양수라면 BPS half-away-from-zero 계산 결과가 0이어도 raw tick을 최소 1로 올린다. ratioBps = 0은 Burn 비활성으로 취급해 StatusApplied를 만들지 않는다. 이는 보편 법칙이 아니라 저피해 DoT가 소실되지 않게 선택한 이 fixture의 명시적 정책이며, 제품에서는 최소값·반올림·0 ratio 의미를 데이터 계약과 경계 테스트로 고정해야 한다.

7. Event / DebugTrace

JavaScript Fireball replay 전체는 command/correlation과 replay header의 DefinitionVersion으로 묶는다. 공통 JavaScript event envelope는 eventId·correlationId·causationId·tick을 운반하고, source는 event별 payload의 sourceRef/sourceId에 둔다. 반면 C# DomainEvent는 EventId·CommandId·Source만 가지며 CorrelationId나 CausationId를 가진다고 주장하지 않는다. C#의 직접 인과는 committed event ID를 가리키는 ReactionCommand.CausationId에서 시작한다. 두 경로 모두 Seed 하나만으로 재현된다고 가정하지 않고 입력/스탯 snapshot과 정렬·반올림 정책을 함께 고정해야 한다.

Event도 commit과 대조한다.

C# SkillCommitted는 mana/cooldown resource ID를 함께 들고 pre→post 값과 version 증가를 검증한다. JavaScript는 닫힌 event taxonomy만 허용하고, Skill의 target/definition, canonical exactRawDamage와 raw 투영, 실제 pre-state 저항, exact scalar 기반 완화, Shield 우선 흡수와 피해 보존식, resource operation, HP/Shield after, SourceRef, Status add/tick/remove, 사망의 치명 전이를 working state와 command에 대조한다. periodic 피해의 exact scalar는 StatusInstance의 정수 raw tick과 정확히 같은 n/1이어야 한다. 따라서 event와 mutation을 함께 그럴듯하게 바꾸거나 임의 event type을 추가해도 publish되지 않으며, 불일치 시 state·tick·outbox·command idempotency가 모두 그대로 남는다.

단, committer가 keyed RNG, Fireball formula, Burn definition·ratio에서 파생된 tick damage·duration·interval·catch-up 값을 command에서 다시 계산하는 것은 아니다. 이 upstream 사실의 권위는 command에 결속된 pure resolver와 runtime/formula/data version, trace, golden fixture에 있다. committer가 보장하는 범위는 exact raw의 canonical 표현·정수 투영부터 저항·보호막·HP 전이와 provenance, 그리고 Burn payload의 shape·data version·생존 적용 조건까지이며, 원시 gameplay policy 자체의 독립 재계산은 아니다.

Reaction의 입력 권위.

JavaScript 실행 시나리오는 성공한 commit receipt의 event만 enqueueReactions에 넘긴다. 이 함수는 raw scenario input을 다시 읽지 않고 DamageCommitted payload에 동결된 Burn definition·tick damage·duration·interval·catch-up budget·data version을 reaction으로 투영한다. 다만 builder 호출 관례만 신뢰하지 않는다. applyStatusReaction은 causation ID로 같은 StateStore outbox의 유일한 DamageCommitted를 찾은 뒤 reaction ID·멱등 키·정렬·budget과 actor/source/target/Burn payload 전체를 event에서 다시 파생해 canonical 비교한다. event가 정한 최초 causal depth=1보다 작게 낮출 수 없으며, 그보다 큰 depth는 ReactionQueue가 dispatch 중 parent.depth + 1로 파생하고 상한을 적용한다. handler는 queue가 현재 dispatch 중인 동일 reaction object에 부여한 내부 권한이 있을 때만 동기적으로 실행되므로 직접 호출로 budget을 건너뛸 수 없다. 원인 event가 없거나 event와 다른 값을 섞은 reaction은 mutation 전에 REACTION_SOURCE_NOT_COMMITTED 또는 REACTION_SOURCE_MISMATCH로 거부된다.

C# CreateReactionCommands는 event/result/source/target과 command 파생 ID를 대조하는 실행 가능한 projection helper이지만, public CommittedOutboxEvent wrapper가 특정 committer receipt의 소속을 capability로 증명하지는 않는다. 이 C# 참조에는 Status commit service도 없으므로 handoff까지만 실행 증거다. 제품 adapter는 receipt/event 생성을 내부 권한으로 닫거나 durable outbox의 aggregate·transaction 소속을 검증한 뒤 이 helper를 호출해야 한다.

권장 커밋 순서. Prepare/Resolve와 DamageCalculated trace → 비용·쿨다운·Shield/HP·outbox atomic commit → SkillCommitted → DamageCommitted → Burn ReactionQueue → StatusApplied → periodic DamageCommitted/StatusTicked → 관찰 이벤트 순서로 한 번의 trace를 만든다.
여기서 “atomic”은 내구성 저장소를 뜻하지 않는다.

JavaScript StateStore는 한 프로세스의 working copy에서 state와 outbox를 함께 교체하는 교육용 구현이다. 실제 서버에서는 동일 DB transaction으로 aggregate state와 outbox row를 저장하고, 재시작 가능한 publisher·claim/lease·at-least-once 중복에 대한 consumer idempotency까지 추가해야 한다. C# RuntimeCommitter도 같은 의미의 in-memory reference이며 DB transaction을 구현하지 않는다.

과거 event는 현재 상태가 아니다.

DamageCommitted.TargetHpAfter > 0은 Burn reaction 후보를 만드는 조건이다. 실제 적용 전에 target을 다시 읽으며 이미 죽었다면 NotApplicable / TARGET_NOT_ALIVE로 끝내고 Status와 outbox를 남기지 않는다. 살아 있는 snapshot으로 만든 status plan도 target version precondition이 어긋나면 전체 commit을 거절한다.

JavaScript fixture에서 CorrelationId는 이 전체 cast를 유지하고, 각 후속 command의 CausationId는 자신을 직접 촉발한 직전 event를 가리킨다. 따라서 첫 Burn tick은 StatusApplied, 다음 tick은 직전 StatusTicked를 trigger로 저장한다. 한 상태가 대상을 쓰러뜨려 다른 상태가 정리되면 그 정리 command의 직접 원인은 방금 발생한 EntityDefeated다. C# 참조에서는 primary DamageCommitted.EventId가 Burn ReactionCommand.CausationId가 되는 경계까지 검증한다.

Comprehension check

이해도 확인

Fireball 한 번의 prepare·commit·reaction 경계와 시스템별 책임을 확인한다.

  1. 01 · 원자적 Commit
    Damage를 resolve한 뒤 대상 version이 바뀌어 commit precondition이 실패하면 마나와 쿨다운은 남아야 할까?
    정답과 해설

    남지 않아야 한다. 비용·쿨다운·주 Damage mutation·outbox가 하나의 CommitPlan이므로 JavaScript의 VERSION_CONFLICT와 C#의 CommitStatus.PreconditionFailed 모두 전체 plan을 미적용으로 끝낸다.

  2. 02 · 경계 상황
    HitOutcome은 Hit이지만 보호막이 피해를 전부 흡수하면 outcomePredicate=hit Burn은 적용될까?
    정답과 해설

    현재 fixture에서는 대상이 살아 있으면 적용된다. outcomePredicate=hit은 FinalHpDamage가 아니라 HitOutcome을 기준으로 하므로 full shield와 구분된다. 현재 fixture에는 Burn 면역 검사가 없다.

  3. 03 · 후속 반응
    개념 확장에 Burn 면역을 추가했을 때 DamageCommitted 뒤 면역 판정이 실패하면 이미 적용된 피해도 롤백해야 할까?
    정답과 해설

    아니다. commit_then_react에서 주 Damage는 먼저 commit되고 Burn은 별도 ApplyStatusEffect다. 확장된 면역 실패는 StatusResult에 남기되 비용·쿨다운·피해는 유지한다. 현재 실행 fixture는 면역 입력과 판정을 제공하지 않는다.
