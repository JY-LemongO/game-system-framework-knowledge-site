권장 구현 순서

P3-A. Contract types

ExecutionEnvelope, SourceRef, ResultEnvelope, DefinitionVersion, clock domain을 값 객체로 고정한다.

P3-B. Commit pipeline

DamageRequest → DamageResult → DamageCommit → AfterDamage를 구현하고 중복 commit 테스트를 추가한다.

P3-C. Reaction queue

lifesteal, reflect, TriggeredEffect를 queue command로 바꾸고 depth/idempotency 제한을 검증한다.

P3-D. Cache & tick semantics

조건부 Stat query와 Burn tick/expire 동률, snapshot/dynamic 정책을 테스트 가능한 규칙으로 만든다.

P3-E. Fireball regression slice

동일 snapshot과 version으로 결과가 재현되고 trace가 전 단계에서 연결되는지 확인한다.

P3-F. Equipment MVP

Gate 통과 후 ItemInstance sourceId, 장착/해제 Modifier, TriggeredEffect를 첫 외부 콘텐츠 공급자로 추가한다.

수용 기준

  • 같은 input snapshot, definitionVersion, seed, target order에서 DamageResult와 StatusResult가 동일하다.
  • Damage commit을 같은 idempotencyKey로 두 번 요청해도 HP/Shield는 한 번만 변경된다.
  • 반사 ↔ 흡혈 연쇄가 depth limit에서 종료되고 모든 causationId가 trace에 남는다.
  • 조건부 Modifier가 다른 target/context 사이에서 잘못 캐시되지 않는다.
  • Burn의 마지막 tick과 만료가 같은 시각일 때 정의한 순서대로 항상 처리된다.
  • 장비 해제 시 해당 ItemInstance source의 Modifier와 Reaction만 제거된다.
  • 키보드만으로 검색, 전체 문서 메뉴, 다이어그램 확대, 모든 주요 링크에 접근할 수 있다.

이번 Gate에서 보류 가능한 항목

완전한 네트워크 prediction/rollback, 분산 서버, 대규모 ECS 최적화, 에디터 툴체인은 프로젝트 요구가 확정된 뒤 설계해도 된다. 다만 execution identity와 idempotency는 나중에 덧붙이기 어렵기 때문에 지금 포함한다.

최종 권고. Equipment 문서를 먼저 늘리기보다 P3-A~E를 작게 구현하고 Fireball 회귀 테스트를 통과한 뒤 Equipment MVP로 이동한다.
