게임 시스템 설계 지식 베이스

Core Runtime에서 공통 계약을 잡고, Stat으로 숫자를 계산하고, Effect로 결과 요청을 만들고, Skill / Action으로 행동을 실행하며, Combat과 Status로 피해 계산과 지속 상태까지 완성하는 학습형 게임 시스템 프레임워크 문서다.

Core ContractStatEffectSkillCombatStatusUML
Foundation

Core Runtime

Entity, Tag, SourceId, EventBus, Context, Time, RandomSeed 같은 공통 계약을 먼저 정리한다.

Core

Stat System

스탯 정의, Modifier, 계산 순서, Resource, Debug Breakdown을 다룬다.

Bridge

Effect System

TargetSelector와 EffectOperation으로 DamageRequest와 ApplyStatusRequest를 만든다.

Action

Skill / Action System

비용, 쿨다운, 타겟팅, 시전 타임라인, EffectBundle 실행 구조를 다룬다.

Resolution

Combat Resolution

DamageEffect가 만든 DamageRequest를 명중, 치명타, 방어, 저항, 보호막 계산으로 해석한다.

Runtime State

Status System

ApplyStatusEffect 이후 지속시간, 중첩, tick, Modifier 연결, 해제 정책을 관리한다.

이번 2차 확장 범위

새로 추가된 문서

Combat Resolution System과 Status System을 본문 문서로 추가했다. 이제 DamageEffect와 ApplyStatusEffect가 실제 피해 계산과 지속 상태 관리로 이어지는 흐름을 읽을 수 있다.

함께 보강된 문서

Effect, Stat, Skill, Integration Map, Fireball Case Study, Glossary, Diagram Gallery, Implementation Roadmap을 Combat/Status 기준으로 갱신했다.

전체 시스템 지도

핵심 방향은 책임을 분리하면서도 데이터 흐름을 끊지 않는 것이다. Skill은 EffectBundle을 실행하고, Effect는 DamageRequest 또는 ApplyStatusRequest를 만들며, Combat과 Status가 각각 계산과 지속 상태를 맡는다.

Core Runtime→Stat→Effect→Skill→Combat→Status→Equipment / Progression

추천 학습 루트

1. Core Runtime

EntityId, SourceId, Tag, EventBus, Context, Time, RandomSeed 같은 공통 계약을 먼저 이해한다.

2. Stat System

숫자 엔진, Modifier, 계산 파이프라인, Resource 정책을 익힌다.

3. Effect System

스탯 값을 DamageRequest, ApplyStatusRequest 같은 결과 요청으로 바꾸는 구조를 익힌다.

4. Skill / Action System

비용, 쿨다운, 타겟팅, 타임라인을 통과해 EffectBundle이 실행되는 구조를 읽는다.

5. Combat Resolution

DamageEffect가 명중, 치명타, 방어, 저항, 보호막 계산을 거쳐 DamageResult가 되는 과정을 확인한다.

6. Status System

Burn, Rage, Freeze 같은 상태가 지속시간, 중첩, tick, Modifier로 관리되는 방식을 이해한다.

7. Fireball Case Study

하나의 스킬이 Skill, Effect, Combat, Status, EventLog를 통과하는 수직 슬라이스를 확인한다.
